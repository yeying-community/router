# 套餐说明

这篇文档专门解释 Router 里的“套餐”到底是什么、按次套餐怎么工作、并发字段为什么有两层，以及当前实现下应该怎么配置。

这篇文档回答的是产品语义和运行边界，不替代接口字段定义，也不替代结算细节：

- 接口字段见 [接口文档.md](./接口文档.md)
- 准入、预占、结算见 [扣费逻辑.md](./扣费逻辑.md)
- 整体计费分层见 [计费架构.md](./计费架构.md)

## 1. 套餐是什么

在 Router 里，套餐是用户侧商品，不是上游渠道原始资源的直接映射。

套餐的职责是定义：

1. 用户可以买什么
2. 这个商品能在哪些分组里使用
3. 它按什么口径扣减
4. 遇到额度不足或并发冲突时如何处理

套餐有两类：

### 1.1 `yyc_quota`

传统额度套餐。

特点：

- 按 `YYC` 扣减
- 常见字段是 `daily_quota_limit` 和 `package_emergency_quota_limit`
- 更适合“按金额预算”治理

### 1.2 `request_quota`

请求次数套餐，也可以叫按次套餐。

特点：

- 按请求次数扣减，不按 token / 图片张数 / 视频时长二次折算
- 常见字段是 `period_type`、`period_limit`
- 更适合“每月 N 次”这一类稳定型商品

例如：

```text
period_type = monthly
period_limit = 25000
```

表示：

```text
每月 25000 次请求
```

## 2. 套餐和分组的关系

套餐自己不直接维护“可用供应商 / 可用模型 / 可用渠道”。

套餐通过 `group_id` 绑定到一个分组，再由分组定义可用范围。

所以应该这样理解：

- 套餐定义的是“商业权益”
- 分组定义的是“技术可用范围”

这也是为什么按次套餐的购买页和管理页只需要选择 `group_id`，而不是在套餐里再单独选模型。

需要特别区分两层视角：

### 2.1 系统实现视角

当前运行时仍然使用 `group_id` 来决定：

- 这个套餐落在哪个技术范围里
- 这个请求可以命中哪些供应商、模型和渠道
- 请求时应该去哪里找“请求分组的生效套餐”

也就是说，`group` 现在仍然是 Router 内部的运行时边界。

### 2.2 用户产品视角

从产品视角，用户不应该感知 `group`。

用户更合理的认知应该只有两类资产：

- 生效套餐
- 账户余额

用户应该看到的是：

- 我买了哪些套餐
- 这些套餐分别包含什么权益
- 这些套餐什么时候到期
- 用完后是否自动转余额
- 我当前还有多少余额

用户不应该被要求理解：

- 自己在哪个分组
- 某个分组能用哪些渠道
- 套餐为什么绑定了某个 `group_id`

因此可以把当前状态理解成：

```text
group 是后台运行边界，不是前台产品概念
```

后续产品收敛时，前台和接口表达应尽量围绕“套餐 + 余额”，而不是围绕“分组”。

## 3. 按次套餐的核心字段

### 3.1 `period_type`

表示次数额度的周期类型。

当前常见值：

- `daily`
- `monthly`
- `package_total`

含义：

- `daily`：每天重置
- `monthly`：每月重置
- `package_total`：整个套餐周期内共用一个总次数池，不按日/月重置

### 3.2 `period_limit`

表示单个周期内最多可用多少次请求。

例子：

- `monthly + 25000` = 每月 25000 次
- `daily + 1000` = 每天 1000 次
- `package_total + 50000` = 套餐总共 50000 次

### 3.3 `allow_balance_fallback`

表示按次套餐不可用时，是否允许回落余额模式。

- `true`：次数不足或并发不可用后，尝试从余额扣费继续执行
- `false`：直接拒绝

这个字段决定的是准入策略，不影响套餐本身怎么计数。

## 4. 为什么会有两种并发限制

按次套餐里有两个并发字段：

- `max_concurrency_per_user`
- `max_concurrency_per_package`

它们看起来相似，但设计上解决的是两个不同层级的问题。

### 4.1 `max_concurrency_per_user`

用户维度的并发限制。

它限制的是：

```text
某一个用户自己的这份按次套餐订阅，同时最多有多少个请求正在占用套餐次数额度、但还没有结算完成
```

它主要解决：

- 单个用户脚本刷接口
- 某个用户的 bug 瞬间打出大量并发
- 单个用户挤占分组或渠道的公共资源

可以把它理解成：

```text
单用户流量闸门
```

### 4.2 `max_concurrency_per_package`

套餐实例维度的并发限制。

它限制的是：

```text
这份套餐订阅整体，最多同时允许多少个请求占用这个套餐的并发池
```

它主要解决：

- 套餐本身作为一个资源池的保护
- 未来共享套餐 / 团队套餐的总并发上限
- 把并发能力本身做成套餐规格的一部分

可以把它理解成：

```text
资源池总闸门
```

## 5. 为什么当前看起来像重复

因为当前实现里，套餐订阅是按用户开通的。

也就是说，当前多数场景下：

- 一个订阅只属于一个用户
- 没有多人共享同一订阅实例

于是这两个限制经常落到同一个对象上：

- “这个用户最多并发 5”
- “这份套餐实例最多并发 5”

所以它们在当前形态下看起来很接近，甚至像重复字段。

这不是设计错误，而是当前业务形态还没有把“用户维度”和“套餐实例维度”真正拉开。

## 6. 为什么仍然要保留两层

即使当前一人一订阅，保留两层限制仍然有价值。

### 6.1 为未来共享套餐预留能力

如果未来出现这些形态：

- 团队套餐
- workspace 共享套餐
- 一个套餐供多个子账号共用

就会立刻需要两个不同问题的答案：

1. 每个人自己最多并发多少？
2. 整个套餐实例最多并发多少？

这时两层限制就不再重叠，而是必要条件。

### 6.2 区分用户治理和商品治理

即使还是个人套餐，也可能希望：

- 基础版套餐最多 3 并发
- 专业版套餐最多 10 并发
- 企业版套餐最多 30 并发

此时 `max_concurrency_per_package` 就不只是防滥用字段，而是套餐规格的一部分。

### 6.3 降低未来兼容性改造成本

如果现在只做一个并发字段，将来再拆成两层，就要面对：

- 数据迁移
- API 兼容
- UI 重构
- 历史默认值推断

现在先保留两层，成本更低。

## 7. 当前实现下怎么理解最合适

在当前版本里，可以这样理解：

- `max_concurrency_per_user`：这个用户最多同时跑多少个按次套餐请求
- `max_concurrency_per_package`：这份套餐实例最多同时跑多少个按次套餐请求

由于当前是一人一订阅，建议默认这么配：

```text
max_concurrency_per_user = max_concurrency_per_package
```

例如：

- `3 / 3`
- `5 / 5`

这样最好理解，也最不容易让运营误解。

## 8. 当前实现与目标产品模型

这里要明确区分“现在系统怎么跑”和“之后产品应该怎么卖”。

### 8.1 当前实现

当前实现更偏运行时模型：

- 用户余额可以独立存在，并且可以多次充值
- 套餐订阅记录带有 `group_id`
- 运行时按 `user_id + group_id` 查找生效套餐
- 一个用户理论上可以在不同分组下同时存在多份生效订阅
- 具体到某一次请求，只会按固定优先级命中一个实际扣费来源

也就是说，当前并不是全局“三选一”：

```text
不是 充值 / YYC 套餐 / 按次套餐 只能存在一个
```

而是：

```text
余额和套餐可以并存
一次请求只会命中一种实际扣费来源
```

### 8.2 目标产品模型

当前产品模型收敛为下面这个模型：

- 用户只感知套餐和余额
- 用户可以同时拥有多个不同套餐槽位的生效套餐
- 同一套餐槽位内只保留一个生效套餐，同槽位新购/升级/降级会立即替换当前生效套餐
- 用户可以多次充值余额
- 套餐不足时，是否允许自动走余额，由套餐策略决定

套餐槽位由后台的 `group_id + package_type + quota_metric` 决定，用户侧不展示“全局唯一套餐”概念。管理员新增套餐时也不手动填写槽位；选择分组后，该分组的模型范围就是套餐适用范围。

### 8.3 推荐的产品规则

建议后续对外规则收敛为：

1. 用户只感知“生效套餐”和“账户余额”
2. 用户可以同时拥有多个不同槽位的套餐
3. 用户可以反复充值余额
4. 同槽位套餐支持续费，续费立即延长当前套餐到期时间
5. 同槽位套餐支持升级，默认立即生效
6. 同槽位套餐支持降级，默认立即生效
7. 不同槽位套餐购买视为新增套餐，不自动替换已有套餐
8. 余额始终独立存在，可作为兜底支付手段

### 8.4 为什么推荐这样收敛

原因主要有四个：

1. 用户认知更简单，只需要理解“我当前是什么套餐”
2. 购买、续费、升级、降级、转化的规则更容易讲清楚
3. 前台页面不需要暴露分组和运行时边界
4. 运行时不需要在多个套餐之间做复杂竞争和解释

因此，这份文档后面的“并发、周期、回落”说明，应该优先理解为：

```text
它们是套餐权益规则，不是让用户理解 group 的理由
```

## 9. 什么时候把两个值配成不同

只有在下面场景里，才建议刻意拉开两个值。

### 9.1 共享套餐

例如：

```text
max_concurrency_per_user = 3
max_concurrency_per_package = 20
```

含义：

- 单个人最多同时跑 3 个
- 整个团队套餐总共最多同时跑 20 个

### 9.2 套餐档位区分

例如：

- 基础版：`2 / 2`
- 专业版：`5 / 5`
- 企业版：`10 / 10`

这时候并发能力本身就是套餐差异化权益。

## 10. 按次套餐的准入和扣减

按次套餐的运行口径是：

1. 按 `user_id + group_id` 匹配生效订阅
2. 按当前请求类型确定本次要预占多少次
3. 检查周期剩余次数
4. 检查并发限制
5. 预占对应数量的并发槽位 / 次数额度
6. 请求结束后结算对应数量，并释放预占

重要的是：

- 它按“套餐定义的请求计数口径”计数
- 文本类当前默认是“1 次请求 = 1 次”
- 图片类当前按图片张数计次

当前可以这样理解：

```text
文本请求：
一次成功命中的按次套餐请求 = 结算 1 次

图片请求：
按 n 张图片结算
n=1 -> 结算 1 次
n=4 -> 结算 4 次
```

这意味着：

- 文本、音频、视频等当前仍可理解为“按请求次数”商品
- 图片模型更准确地说是“按图片输出数量计入套餐次数”

因此对用户解释时，不应简单写成“所有模型都是一次请求扣一次”，否则会和图片模型的真实口径冲突。

## 11. 回落余额是什么意思

如果按次套餐遇到下面情况：

- 周期次数已用完
- 单用户并发满了
- 套餐并发满了

系统会看 `allow_balance_fallback`：

- `true`：尝试转余额模式
- `false`：直接拒绝

这让按次套餐可以有两种产品策略：

### 严格型

```text
allow_balance_fallback = false
```

适合：

- 用户必须明确感知套餐是否耗尽
- 不希望余额被自动扣掉

### 平滑型

```text
allow_balance_fallback = true
```

适合：

- 不希望请求中断
- 愿意把次数套餐当作优先层，余额当兜底层

## 12. 当前配置建议

如果你现在还没有共享套餐实例需求，推荐这样配置：

### 12.1 普通个人按次套餐

```text
package_type = request_quota
quota_metric = request_count
period_type = monthly
period_limit = 25000
max_concurrency_per_user = 3~5
max_concurrency_per_package = 与单用户并发相同
allow_balance_fallback = false
```

如果套餐会覆盖图片模型，还应补一条对外说明：

```text
图片生成按张计次，例如 n=4 记 4 次
```

### 12.2 希望不中断的个人按次套餐

```text
allow_balance_fallback = true
```

表示：

- 先吃按次套餐
- 不够了再扣余额

### 12.3 暂时不要做的配置

如果当前没有共享套餐需求，不建议一开始就把两个并发值配得差很多，例如：

```text
max_concurrency_per_user = 2
max_concurrency_per_package = 50
```

这种配置在当前一人一订阅模型下通常没有额外收益，只会增加理解成本。

## 13. 一句话总结

按次套餐的本质是：

```text
在指定分组范围内，按“请求次数”而不是按 YYC 金额结算的周期性商品
```

两层并发限制的本质是：

```text
单用户并发 = 防单个用户打爆系统
套餐并发 = 防这份套餐实例把资源池打穿
```

在当前一人一订阅实现里，建议默认把两者配成相同值；只有当共享套餐或套餐分级能力真的出现时，再把它们拉开。
