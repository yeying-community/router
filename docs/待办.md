# 待办

本文档仅保留“未完成且已确认要做”的事项。  
已完成事项不在此重复记录，避免与实现状态偏离。

当前按优先级分为 `P0 / P1 / P2`：

- `P0`：直接影响线上稳定性、扣费正确性、支付闭环。
- `P1`：影响运营排障效率、金额审计能力、身份一致性。
- `P2`：影响产品完整性与代码整洁度，但短期不阻断主链路。

## P0

1. 路由策略治理
- 在管理台和文档中清晰区分：
- 请求内切换：`RetryTimes`（`0` 关闭，`>0` 尝试剩余候选）。
- 后续避障：自动禁用、账务刷新、人工关闭。
- 上游限流、余额不足、鉴权失败、网关异常在表象上容易混淆。
- 需按 [渠道错误处理策略](./渠道错误处理策略.md) 落地窗口熔断、自动恢复和可观测性。
- `channel_model_endpoints` 已参与真实选路，但后台可视化仍不足。
- 需展示端点支持状态、暂停原因、暂停时间、恢复状态。
- `unsupported_channel_endpoint` 目前支持端点能力自动暂停。
- 运行态自动暂停通知、能力级自动测试恢复、人工测试通过后兜底恢复、恢复通知、低成功率熔断自动恢复、熔断状态持久化、半开探测已落地。
- 低成功率熔断冷却到期后先进入半开探测，成功后完全恢复，失败则重新熔断等待；半开渠道以最低优先级参与真实选路。
- 渠道列表已展示熔断状态，并可查看原因、触发成功率、熔断时间、预计恢复时间、最近恢复时间。
- 渠道详情页已补熔断事件列表，后端已持久化 `open / half_open / recovered / canceled` 事件历史。
- 后续如需要更强运营追踪，可在当前事件表基础上扩展全局熔断事件检索页。

### P0 建议分三轮落地

1. 第一轮：先稳住路由主链路
- 目标：让“为什么失败、为什么切换、为什么自动暂停、为什么恢复”变得可解释。
- 输入：现有 `RetryTimes`、端点自动暂停逻辑、渠道错误分类、当前日志字段。
- 输出：
- 路由失败分类表：明确哪些错误可切换、可暂停模型、可暂停端点、可熔断整条渠道。
- 熔断闭环：已具备连续失败阈值、熔断时长、自动恢复、状态持久化、半开探测和列表可视化。
- 管理台最小可视化：展示 `channel + model + endpoint` 当前状态、最近失败原因、最近暂停时间，并在渠道详情展示熔断事件历史。
- 验收：
- 同一渠道连续失败时，不再只靠单请求内切换顶住。
- 运营能在后台直接看到“当前为什么不可用”，不需要只翻日志。

2. 第二轮：定稿支付、充值、兑换闭环
- 目标：把“下单、支付、补偿、入账、发码、审计”串成单一确定流程。
- 输入：现有外部支付 API 模式、回调、查单补偿、充值订单、兑换码、余额批次。
- 输出：
- 统一术语：前端文案、接口名、业务动作统一。
- 发码责任边界：明确由 Router 发码还是外部系统发码。
- 幂等与补偿规则：支付成功重复回调、发码失败、入账失败、查单补偿等场景统一处理。
- 多支付方回调协议已先进入统一归一化层，错误码映射沿用稳定本地状态机。
- 验收：
- 任一订单都能解释“当前处于哪个阶段、是否已入账、是否需要人工处理”。
- 支付成功但链路中断时，能通过查单或后台“补发放”补偿入口收敛到最终一致状态。

## P1

1. 汇率治理
- 当前仅接入 `frankfurter` 单一来源。
- 需补主备来源、失败重试和降级告警。
- 当前仅保留最新汇率参数。
- 需补按日或按同步批次快照，用于审计和回放。
- 当前仅有最近成功/失败时间与错误信息。
- 需补连续失败计数、同步耗时、最近 N 次执行记录、告警阈值。
- 当前是全局单一 provider。
- 需评估按币种或按租户选择来源与更新策略。

2. 余额账本治理
- 当前管理员用户详情页“余额批次”仅固定第一页（`page=1,page_size=20`）。
- 需补页码切换与总量展示。
- 当前来源 ID 仅文本展示。
- 需按 `source_type` 下钻：`topup_order`、`redemption`、`legacy_migration`。

3. JWT/UCAN 默认记账 token 策略显式化
- 当前钱包 JWT/UCAN 调用 `/v1/*` 时，默认命中该用户“最早创建且可用”的 token。
- 需补文档可视化、可配置默认 token，以及日志/令牌页的命中明细。

4. 钱包地址身份约束收口
- 当前应用层已统一做小写归一，但数据库层仍需补大小写无关唯一约束。
- 需补 `wallet_address` 历史数据清洗与唯一索引迁移，避免因地址大小写差异生成重复账号。
- 需提供标准化订正方案：重复账号识别、资产归并、钱包重绑后的审计记录。

## P2

1. 半成品渠道协议/适配器收口
- 当前管理端仍暴露 `Cohere`、`Cloudflare`、`Replicate` 等渠道协议类型，但部分能力仍是 `not implemented` 或仅支持部分模式。
- 需明确策略：要么补齐，要么标记为实验性并从默认可选项中收起。
- 避免“管理台可配置”与“运行时不可用”长期并存。

2. 历史兼容分支清理
- 仍有少量 `quota`、`legacy_migration`、旧返回字段的兼容读取逻辑留在前端与接口层。
- 当充值、兑换、日志、余额批次的新字段口径稳定后，需逐步删除兼容分支。
- 保持“文档描述、接口返回、页面展示、数据库语义”一致，避免后续误读。

3. Tokenizer 远期精度演进
- 当前已在 Router 内维护 OpenAI `model/prefix -> encoding` 规则，并覆盖 `gpt-5*`、`o*`、`gpt-realtime*`、`gpt-audio*`、`gpt-image*` 等现代模型族。
- 当前仍使用 `github.com/pkoukk/tiktoken-go` 的本地 encoding 能力，未引入服务化 sidecar。
- 后续若主流模型本地估算与最终 usage 偏差不可接受，再评估接入官方 `tiktoken` sidecar 或独立 tokenizer 服务。
