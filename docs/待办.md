# 待办

本文档仅保留“未完成且已确认要做”的事项。  
已完成事项不在此重复记录，避免与实现状态偏离。

当前按优先级分为 `P0 / P1 / P2`：

- `P0`：直接影响线上稳定性、扣费正确性、支付闭环。
- `P1`：影响运营排障效率、金额审计能力、身份一致性。
- `P2`：影响产品完整性与代码整洁度，但短期不阻断主链路。

## P0

1. 路由策略治理
- 在管理台和文档中清晰区分：
- 请求内切换：`RetryTimes`（`0` 关闭，`>0` 尝试剩余候选）。
- 后续避障：自动禁用、余额刷新、人工摘除。
- 上游限流、余额不足、鉴权失败、网关异常在表象上容易混淆。
- 需按 [渠道错误处理策略](./渠道错误处理策略.md) 落地窗口熔断、自动恢复和可观测性。
- `channel_model_endpoints` 已参与真实选路，但后台可视化仍不足。
- 需展示端点支持状态、摘除原因、摘除时间、恢复状态。
- `unsupported_channel_endpoint` 目前支持摘除。
- 需补恢复路径：人工恢复、模型同步恢复、定时探测恢复。
- 当前仍以单次请求内切换为主。
- 需补连续失败阈值、熔断时长、半开探测、恢复放量。

2. Tokenizer 方案
- 现使用 `github.com/pkoukk/tiktoken-go`，模型映射偏旧。
- 评估切换到更底层仅负责 encoding 的库。

- 至少覆盖 `gpt-5*`、`o1/o3/o4*`、`gpt-realtime*`、`gpt-audio*`、`gpt-image*`。
- 评估通过 sidecar/服务化接入官方 `tiktoken`，提升估算精度。
- 未知模型不再回退到 `gpt-3.5-turbo`。
- 优先同家族 encoding 回退，并降低可预期新模型家族的错误日志噪音。

3. 充值与兑换链路定稿

### P0 建议分三轮落地

1. 第一轮：先稳住路由主链路
- 目标：让“为什么失败、为什么切换、为什么摘除、为什么恢复”变得可解释。
- 输入：现有 `RetryTimes`、端点摘除逻辑、渠道错误分类、当前日志字段。
- 输出：
- 路由失败分类表：明确哪些错误可切换、可摘模型、可摘端点、可熔断整条渠道。
- 熔断最小闭环：连续失败阈值、熔断时长、半开探测、恢复放量。
- 管理台最小可视化：展示 `channel + model + endpoint` 当前状态、最近失败原因、最近摘除时间。
- 验收：
- 同一渠道连续失败时，不再只靠单请求内切换顶住。
- 运营能在后台直接看到“当前为什么不可用”，不需要只翻日志。

2. 第二轮：收口 token 估算与扣费前置判断
- 目标：降低新模型接入时的 token 估算误差和错误日志噪音。
- 输入：当前 `tiktoken-go` 依赖、模型到 encoding 的回退逻辑、现有消费日志。
- 输出：
- Router 内部维护的 `model/prefix -> encoding` 映射规则。
- 未知模型回退策略调整：不再回退到 `gpt-3.5-turbo`，优先回退到同家族 encoding。
- 至少覆盖 `gpt-5*`、`o*`、`gpt-realtime*`、`gpt-audio*`、`gpt-image*`。
- 验收：
- 新增同家族模型时，不需要临时打补丁也不会刷大量错误日志。
- 预估 token 与最终 usage 的偏差在主流模型上保持可接受范围。

3. 第三轮：定稿支付、充值、兑换闭环
- 目标：把“下单、支付、补偿、入账、发码、审计”串成单一确定流程。
- 输入：现有外部支付 API 模式、回调、查单补偿、充值订单、兑换码、余额批次。
- 输出：
- 统一术语：前端文案、接口名、业务动作统一。
- 发码责任边界：明确由 Router 发码还是外部系统发码。
- 幂等与补偿规则：支付成功重复回调、发码失败、入账失败、查单补偿等场景统一处理。
- 多支付方回调协议和错误码映射规范。
- 验收：
- 任一订单都能解释“当前处于哪个阶段、是否已入账、是否需要人工处理”。
- 支付成功但链路中断时，能通过查单/补偿自动收敛到最终一致状态。

## P1

1. 汇率治理
- 当前仅接入 `frankfurter` 单一来源。
- 需补主备来源、失败重试和降级告警。
- 当前仅保留最新汇率参数。
- 需补按日或按同步批次快照，用于审计和回放。
- 当前仅有最近成功/失败时间与错误信息。
- 需补连续失败计数、同步耗时、最近 N 次执行记录、告警阈值。
- 当前是全局单一 provider。
- 需评估按币种或按租户选择来源与更新策略。

2. 余额账本治理
- 当前管理员用户详情页“余额批次”仅固定第一页（`page=1,page_size=20`）。
- 需补页码切换与总量展示。
- 当前来源 ID 仅文本展示。
- 需按 `source_type` 下钻：`topup_order`、`redemption`、`legacy_migration`。

3. JWT/UCAN 默认记账 token 策略显式化
- 当前钱包 JWT/UCAN 调用 `/v1/*` 时，默认命中该用户“最早创建且可用”的 token。
- 需补文档可视化、可配置默认 token，以及日志/令牌页的命中明细。

4. 钱包地址身份约束收口
- 当前应用层已统一做小写归一，但数据库层仍需补大小写无关唯一约束。
- 需补 `wallet_address` 历史数据清洗与唯一索引迁移，避免因地址大小写差异生成重复账号。
- 需提供标准化订正方案：重复账号识别、资产归并、钱包重绑后的审计记录。
