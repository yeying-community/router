# 问题排查

## 2026-04-30 `stream` 与 `Accept` 冲突的当前结论

### 结论

- Router 当前只以请求体里的 `stream` 作为流式语义来源。
- OpenAI / Anthropic 上游请求头 `Accept` 由 Router 按 `stream` 统一生成，不再透传下游 `Accept`。
- 若下游出现 `stream` 与 `Accept` 冲突，当前仅记录告警日志，便于识别不规范客户端。

### 原因

- 历史代码曾透传下游 `Accept`，会出现“`stream=false` 但 `Accept: text/event-stream`”这类冲突。
- 某些上游会优先按 `Accept` 返回 SSE，导致 Router 进入非流式 JSON 解析时出现：
  - `invalid character 'd' looking for beginning of value`
- 其中 `d` 通常来自 SSE 的 `data:` 前缀。

### 现状

- 当前策略是“`stream` 决定语义，`Accept` 只是派生输出”。
- 文本端点是否可用，仍以渠道模型已启用端点为准，不再依赖桥接或头部猜测。

## 2026-04-15 模型测试说明列文本重复（OpenAIo4-miniOpenAI o4-mini）

### 现象

- 管理端渠道模型测试页面，`说明` 列出现重复文本。
- 典型示例：`OpenAIo4-miniOpenAI o4-mini`。

### 影响范围

- 仅影响模型测试结果展示文案（`/api/v1/admin/channel/test_models` 的解析结果）。
- 不影响真实 relay 调用的请求转发与计费。

### 根因分析

`responses` 流式测试响应（SSE）解析逻辑存在两处叠加问题：

1. `response.output_text.delta` 的增量文本被 `TrimSpace`，导致前导空格丢失。  
示例：`"OpenAI"` + `" o4-mini"` 被拼成 `"OpenAIo4-mini"`。

2. `response.completed` 的完整 `output_text` 又被追加到增量拼接结果后面，导致重复。  
示例：`OpenAIo4-mini` + `OpenAI o4-mini`。

### 修复方案

文件：`internal/admin/controller/channel/model_test_response.go`

1. 保留 delta 原始空格
- `extractTextFromStreamPayload` 中，对 `delta/text/output_text` 不再执行 `TrimSpace`。
- 仅 `message` 字段保持 `TrimSpace` 处理（错误文案/提示文案场景）。

2. 为 completed 事件增加去重聚合策略
- 在 `parseTextModelTestStreamResponse` 中，遇到 `response.output_text` 或 `response.completed` 事件时：
  - 若 completed 文本包含此前已累计的 delta 文本：以 completed 文本为准（单份）。
  - 若 completed 文本是已累计文本的子串：保持已累计文本不变。
  - 若 completed 文本是增量尾部（如仅标点）：继续追加。

### 验证结果

新增/更新单测（文件：`internal/admin/controller/channel/model_test_response_test.go`）：

- `TestParseTextModelTestResponse_ResponsesSSE`
  - 期望值由 `Helloworld!` 修正为 `Hello world!`（验证空格保留）。

- `TestParseTextModelTestResponse_ResponsesSSECompletedContainsFullText`
  - 验证 `delta + completed(完整文本)` 只输出单份：`OpenAI o4-mini`。

执行结果：

- `go test ./internal/admin/controller/channel` 通过。

### 结论

本次问题属于测试响应解析逻辑的文本聚合缺陷，已通过最小改动修复，不改变模型测试请求路径和上游协议行为。

## 2026-04-15 结论纠偏：上游是否支持非流式

### 背景

- 历史阶段曾出现结论：`/v1/responses` 上游“不支持非流式”。
- 2026-04-15 新增测试页“是否流式”动态勾选后，针对同一渠道同一模型（`gpt-5.4`）验证显示：流式与非流式均可测试通过。

### 具体改了什么（代码级）

> 核心修复点是“请求体结构对齐 responses 标准 + 解析逻辑按端点收敛”，不是 `is_stream` 开关本身。

1. 失败请求与成功请求的核心差异（导致结果不同）
- 失败样本（2026-04-14，非流式）：
  - `{"model":"gpt-5.4","input":[{"role":"user","content":"Output only your specific model name with no additional text."}]}`
  - 返回：`status=completed` 但 `output=[]`，最终被判 `response has no output text, output is empty`。
- 成功样本（2026-04-15，非流式）：
  - `{"input":[{"content":[{"text":"Output only your specific model name with no additional text.","type":"input_text"}],"role":"user"}],"model":"gpt-5.4"}`
  - 返回：`output:[{type:"message",content:[{type:"output_text",text:"OpenAI o4-mini"}]}]`，可正常解析。
- 关键差异：
  - 旧请求是“chat 风格 content 字符串”塞进 `responses` 的 `input`。
  - 新请求是“responses 标准 input_text 结构”。

2. 固化 `responses` 测试请求模板（对齐标准结构）
- 文件：`internal/admin/controller/channel/model_runtime.go`
- 函数：
  - `buildResponsesTextModelTestRequest`
  - `buildResponsesTextModelTestRequestBody`
- 关键变化：
  - 测试请求统一生成 `input_text` 结构，不再在多种 `input` 形态间漂移。
- 修复作用：
  - 直接消除“同一模型、同一端点、因请求体形态不同导致结果不一致”的主因。

3. 模型测试按端点解析，避免跨协议误判
- 文件：`internal/admin/controller/channel/model_runtime.go`
- 函数：`parseTextModelTestResponseByEndpoint`
- 关键变化：
  - `/v1/responses` 只按 `responses`/SSE 解析；不再混入 chat/messages 解析报错。
- 修复作用：
  - 错误信息聚焦在 `responses` 本身，便于定位“请求体不匹配”这类问题。

4. `responses` 解析能力补齐（避免“实际成功但解析失败”）
- 文件：`internal/admin/controller/channel/model_test_response.go`
- 函数：
  - `parseResponsesModelTestResponse`
  - `parseTextModelTestStreamResponse`
- 关键变化：
  - 支持读取 `output_text`。
  - 对 SSE 的 `response.output_text.delta`、`response.output_text.done`、`response.completed` 去重聚合。
  - 保留 delta 空格，修复 `OpenAIo4-miniOpenAI o4-mini` 重复/粘连。
- 修复作用：
  - 解析层不再把可用结果误判为失败。

5. `is_stream` 的定位（非根因）
- `is_stream` 只是为了在同一页面上做流式/非流式 A/B 对照验证，证明两种模式都可通过。
- 它不是导致“从失败到成功”的关键修复项。

6. 关键提交（便于追溯）
- `b6b8f55`：端点定向解析 + responses 解析收敛。
- `628635a`：测试页流式开关、`is_stream` 全链路透传与持久化。

### 复盘证据

1. 非流式成功（`is_stream=false`）
- 日志：`2026/04/15 - 11:29:05`，`status=supported`。
- 请求体（无 `stream` 字段）：
  - `{"input":[{"content":[{"text":"Output only your specific model name with no additional text.","type":"input_text"}],"role":"user"}],"model":"gpt-5.4"}`
- 响应体为标准非流式 JSON，包含：
  - `output: [{"type":"message","role":"assistant","content":[{"type":"output_text","text":"OpenAI o4-mini"}]}]`
- 参考：`logs/router.log` 对应 `task_id=6c3fc4afd74c433eb240610e5d7a009b`。

2. 流式成功（`is_stream=true`）
- 日志：`2026/04/15 - 11:30:23`，`status=supported`。
- 请求头 `Accept: text/event-stream`，请求体包含 `stream:true`，返回 SSE 事件流并正常解析。
- 参考：`logs/router.log` 对应 `task_id=3e02303ca1f14ba3bbb7d522c6f4c31b`。

3. 历史“非流式失败”样本并不能直接推出“协议不支持”
- 典型失败日志：`parse as responses failed: response has no output text, output is empty`（如 `2026/04/14 - 16:20:46`）。
- 同一时期还伴随大量上游 `503`（`所有供应商暂时不可用`），属于可用性干扰项，不能用于判定“接口能力不支持”。
- 早期失败请求模板存在差异：`input` 结构在部分样本中为 `[{role,content}]` 或纯字符串；后续稳定为 `responses` 推荐结构（`input_text`）后，非流式可正常返回可解析文本。

### 原因归纳

- 旧结论形成时，混合了三类问题：
1. 上游可用性波动（`503`）。
2. 模型测试请求模板在演进期不稳定（`input` 结构与 `responses` 实际期望存在差异）。
3. 当时解析策略偏严格，遇到 `output` 为空但请求已完成时直接判失败，放大了误判概率。

### 修正结论

- `2026-04-15` 后的有效结论应为：
  - 该上游 `responses` 端点支持流式与非流式。
  - 历史失败主要来自“可用性波动 + 测试模板/解析策略问题”，而非“非流式协议不被支持”。

### 后续约束

- 结论类文档必须基于“同渠道同模型同时间窗口”的流式/非流式对照样本，避免跨时段混合推断。
- 遇到 `503` 或上游资源池错误时，仅标记“可用性异常”，不得直接升级为“协议能力不支持”。
