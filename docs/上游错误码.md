# 上游错误码

本文档用于沉淀 Router 在运行过程中遇到的“上游相关错误码”及其排查结论。

这里的“错误码”包含两类来源：

1. 上游供应商直接返回的 `error.code` / `error.type`。
2. Router 在请求上游、解析上游响应、构造协议请求时，本地包装出的错误码。

本文重点回答四个问题：

1. 这个错误码是谁产生的。
2. 它在当前系统里的含义是什么。
3. 它通常由什么原因触发。
4. 出现后应该优先怎么排查。

## 文档信息

- 最后核对日期：2026-05-15
- 适用分支：`main`
- 当前代码锚点：
  - `internal/relay/adaptor/common.go`
  - `internal/relay/controller/image.go`
  - `internal/admin/controller/relay.go`

## 1. 核心结论

1. `do_request_failed` 是 Router 本地包装错误，不是上游标准错误码。
2. `upstream_transport_goaway` 是 Router 针对图片上游 HTTP/2 `GOAWAY` 新增的细分错误码，用于区分普通请求失败与连接层主动关闭。
3. `unsupported_image_billing` 是 Router 本地计费或计价校验错误，发生在真正请求上游之前。
4. 判断“是计费问题还是上游问题”的最直接方法，不是只看最终返回给客户端的文案，而是看日志里有没有：
   - `image_token_estimate`
   - `upstream_req`
   - 上游 `status` 或上游传输错误

## 2. 判定方法

可按下面顺序判断：

1. 如果日志里没有 `upstream_req`，通常说明错误发生在 Router 本地，例如计费、请求构造阶段。
2. 如果日志里已经出现 `upstream_req`，但后面没有上游成功状态，而是连接异常、EOF、超时、GOAWAY，则说明错误发生在上游传输层。
3. 如果上游已经返回了结构化错误体，则结合 `status`、`error.type`、`error.code`、`message` 一起判断。

## 3. 错误码明细

### 3.1 `upstream_transport_goaway`

- 来源：Router 本地包装
- 当前出现位置：
  - 图片请求调用上游后，收到 HTTP/2 `GOAWAY` 时
  - 当前代码由 `internal/relay/controller/image.go` 识别并细分
- 当前含义：
  - 上游服务端或其前置网关主动发送了 HTTP/2 `GOAWAY` 帧，并关闭了连接
  - 请求已发往上游，但在传输层失败
- 常见日志特征：
  - `http2: server sent GOAWAY and closed the connection`
  - `ErrCode=NO_ERROR`
  - `LastStreamID=...`
- 典型原因：
  - 上游网关或反代优雅重载
  - CDN / LB 主动回收 HTTP/2 长连接
  - 图片请求耗时较长，触发连接生命周期或空闲策略
  - 上游实例滚动发布、摘流或重启
- 排查优先级：
  1. 查上游网关在错误发生时间点是否 reload / deploy / pod 重建
  2. 查 HTTP/2、keepalive、idle timeout、read timeout 配置
  3. 查是否只有图片接口容易出现，文本接口正常
  4. 查是否由特定 CDN / 反代层导致
- 当前处理策略：
  - 归类为瞬时上游传输错误
  - 对图片端点不做自动重试
  - 对客户端最终文案统一收敛为“当前分组可用上游暂时不可用，请稍后再试”
- 典型样例：
  - `trace id: 2026051521272457138600005817608`
  - 上游 URL：`https://cdn.hanbbq.top/v1/images/generations`

### 3.2 `do_request_failed`

- 来源：Router 本地包装
- 当前出现位置：
  - `DoRequestHelper(...)` 调上游失败时的通用兜底
- 当前含义：
  - Router 已准备好上游请求，但真正发起 HTTP 请求时失败
  - 这是一个“传输阶段失败”的大类，不等于某一个具体根因
- 常见原因：
  - 连接失败
  - TLS 握手失败
  - 读写超时
  - DNS 解析失败
  - 对端主动断开
  - 某些尚未细分的连接层错误
- 排查优先级：
  1. 先看 `error` 原始消息，不要只看 `code`
  2. 看是否存在更细的特征，例如 `timeout`、`EOF`、`connection reset`、`GOAWAY`
  3. 确认是否已发出 `upstream_req`
  4. 结合上游 URL、端点、模型和请求耗时一起判断
- 当前处理策略：
  - 文本等非图片请求，若允许切换且还有候选，可继续走 Router 内部故障转移
  - 图片请求不自动重试
  - 候选耗尽后统一收敛为临时上游不可用
- 注意事项：
  - `do_request_failed` 只是“通用外壳”
  - 真正的根因要看 `error` 文本

### 3.3 `unsupported_image_billing`

- 来源：Router 本地包装
- 当前出现位置：
  - 图片计费、计价、估算逻辑校验阶段
- 当前含义：
  - 当前图片模型、价格单位或价格配置，不满足 Router 对图片计费的要求
  - 错误发生在请求上游之前
- 常见日志特征：
  - `image billing strategy is not supported for model ...`
  - `traditional image token billing text input price is not configured for model ...`
  - 整个请求只耗时几十毫秒
  - 没有 `upstream_req`
- 典型原因：
  - 图片模型价格单位与当前图片端点计费方式不兼容
  - 图片输入价未配置
  - 图片输出价未配置
  - 供应商默认定价和渠道覆写定价之间缺字段
- 排查优先级：
  1. 查模型定价配置
  2. 查图片模型的 `price_unit`
  3. 查是否是 `gpt-image-*` 这类按传统图片 token 估算的模型
  4. 查渠道是否做了价格覆写但字段不完整
- 当前处理策略：
  - 直接返回
  - 不发上游请求
  - 不应误判为上游不稳定

### 3.4 `invalid_api_key`

- 来源：通常由上游直接返回
- 当前含义：
  - 渠道使用的 API Key 无效、已删除、已禁用或已过期
- 常见日志特征：
  - 上游响应状态通常为 `401`
  - `type=invalid_api_key`
  - `code=invalid_api_key`
  - 常见消息：`API 密钥无效。提供的密钥不存在、已被删除、已被禁用或已过期。`
- 典型原因：
  - 渠道密钥填错
  - 上游控制台已删除或重置密钥
  - 密钥已被供应商禁用
  - 渠道切换后旧密钥未同步更新
- 排查优先级：
  1. 到上游平台核对密钥是否仍有效
  2. 核对 Router 渠道配置里的密钥值是否为最新
  3. 查同一渠道是否所有模型都失败
  4. 查是否伴随 `401`、`authentication_error`
- 当前处理策略：
  - 归为渠道永久性故障
  - 当前请求可切换其他候选
  - 渠道可被自动禁用，等待人工修复

### 3.5 `insufficient_quota`

- 来源：通常由上游直接返回
- 当前含义：
  - 上游账号余额不足、额度耗尽、信用额度不足或账单限制触发
- 常见日志特征：
  - 上游响应状态常见为 `402`
  - `type=insufficient_quota` 或消息里包含 `quota`、`credit`、`balance`
- 典型原因：
  - 供应商账号没钱
  - 当日额度用完
  - 账单达到硬限制
  - 上游项目级配额被耗尽
- 排查优先级：
  1. 到供应商控制台查余额和配额
  2. 查该渠道是否所有模型都一起失败
  3. 查是否只是某个项目或某个组织额度耗尽
- 当前处理策略：
  - 归为上游永久性额度问题
  - 当前请求可切换其他候选
  - 候选耗尽后最终文案收敛为“当前分组可用上游额度不足，请稍后再试”

### 3.6 `bad_response`

- 来源：Router 本地兜底包装
- 当前含义：
  - 上游返回内容异常，Router 无法得到符合预期的响应对象或结构
- 常见日志特征：
  - 上游已经有 HTTP 响应
  - 但响应体为空、结构和适配器预期不一致，或关键字段缺失
- 典型原因：
  - 上游网关返回了非标准结构
  - 某供应商协议和当前适配器不匹配
  - 中间层返回了被截断或被改写的 JSON
- 排查优先级：
  1. 查 `body_preview`
  2. 查上游 `content-type`
  3. 对照对应 adaptor 的响应解析逻辑
  4. 判断是单供应商协议变化还是全局协议处理缺陷
- 当前处理策略：
  - 更偏协议兼容或上游异常
  - 不应直接当成本地计费错误

### 3.7 `bad_response_status_code`

- 来源：Router 本地兜底包装
- 当前含义：
  - 上游返回了非成功状态，但 Router 未能提取出更明确的标准错误码
- 常见日志特征：
  - 有上游 HTTP 状态
  - 有 `body_preview`
  - 但最终没有更细的 `code`
- 典型原因：
  - 非 OpenAI 风格供应商返回体不标准
  - HTML 错误页、反代错误页、网关兜底页
  - 中间层统一错误页覆盖了供应商原始错误
- 排查优先级：
  1. 先看上游 `status`
  2. 再看 `body_preview`
  3. 判断是供应商返回、反代返回还是 CDN 返回
- 当前处理策略：
  - 属于上游异常兜底
  - 具体动作要结合上游状态和消息再判断

### 3.8 `model_not_found`

- 来源：上游常见错误；部分场景也可能由 Router 主动构造
- 当前含义：
  - 当前渠道不支持该模型，或该模型名在上游不存在
- 常见日志特征：
  - 常见于 `404` 或 `400 invalid_request_error`
  - 消息里包含 `model not found`、`does not exist`、`unknown model`
- 典型原因：
  - 模型名填错
  - 供应商尚未开通该模型
  - 当前渠道协议类型或请求构造错误
  - 路由到了不具备该模型能力的渠道
- 排查优先级：
  1. 查该模型是否在渠道支持列表里
  2. 查模型映射是否正确
  3. 查是否为供应商未开通而非模型本身不存在
- 当前处理策略：
  - 更偏能力问题，不应直接禁用整条渠道
  - 适合下沉到 `channel_model` 或运行时模型能力层处理

### 3.9 `unsupported_channel_endpoint`

- 来源：Router 本地包装
- 当前含义：
  - 当前渠道模型不支持请求的具体端点
  - 例如模型能走 `/v1/chat/completions`，但不支持 `/v1/responses`
- 常见日志特征：
  - 常见为 `400`
  - 文案类似：`channel model does not support /v1/responses`
- 典型原因：
  - 端点能力声明不完整
  - 渠道模型端点开关配置错误
  - 调用了模型未开通的模态端点
- 排查优先级：
  1. 查 `channel_model_endpoints`
  2. 查该模型对该端点的启用状态
  3. 查 UI 或 migration 是否错误写入了端点能力
- 当前处理策略：
  - 归为端点能力问题
  - 摘除 `channel + model + endpoint` 能力
  - 不应误伤整条渠道的其他模型或其他端点

### 3.10 `authentication_error`

- 来源：通常由上游直接返回
- 当前含义：
  - 上游认证失败
  - 常见于 Key 不合法、认证头错误、账号权限被撤销
- 常见日志特征：
  - 上游状态常见为 `401`
  - `type=authentication_error`
  - 也可能只出现认证失败消息而没有稳定 `code`
- 典型原因：
  - Key 填错
  - 认证头格式不符合供应商要求
  - 上游账号状态异常
  - 密钥所属组织或项目已失效
- 排查优先级：
  1. 查渠道密钥与认证头是否正确
  2. 到上游控制台确认账号和组织状态
  3. 判断是单模型失败还是全渠道失败
- 当前处理策略：
  - 当前代码会把它视为渠道永久性故障候选
  - 在启用自动禁用时，可能直接禁用渠道

### 3.11 `permission_error`

- 来源：通常由上游直接返回
- 当前含义：
  - 调用方没有权限访问某个资源
  - 对模型场景最常见的是“没有访问某模型的权限”
- 常见日志特征：
  - `type=permission_error`
  - 消息里经常同时包含 `model` 和 `access/permission/not allowed`
- 典型原因：
  - 渠道账号未开通目标模型
  - 当前组织没有该模型权限
  - 某资源范围被上游限制
- 排查优先级：
  1. 看消息是否明确指向模型
  2. 如果是模型范围权限问题，优先按模型能力处理
  3. 如果是整账号权限问题，再考虑渠道级处理
- 当前处理策略：
  - 当前监控逻辑可能把它当作可禁用渠道的故障
  - 但从语义上更偏模型能力问题
  - 排查时不要默认认为整条渠道都坏了

### 3.12 `forbidden`

- 来源：通常由上游直接返回
- 当前含义：
  - 请求被禁止
  - 这是一个较宽泛的错误类型，可能是权限、账号状态、策略限制或组织限制
- 常见日志特征：
  - 常见状态为 `403`
  - `type=forbidden`
  - 消息语义差异较大
- 典型原因：
  - 账号被限制
  - 组织策略限制
  - 权限不足
  - 合规策略或供应商风控
- 排查优先级：
  1. 必须结合消息一起看
  2. 判断是模型级限制、账号级限制还是组织级限制
  3. 不要只凭 `403` 就下结论
- 当前处理策略：
  - 当前监控逻辑允许把它视为可禁用渠道的故障
  - 但实际处理时应先判断根因范围

### 3.13 `billing_error`

- 来源：通常由上游直接返回
- 当前含义：
  - 上游账单、扣费、信用额度、结算限制相关错误
- 常见日志特征：
  - `type=billing_error`
  - 消息常带 `billing`、`credit`、`balance`、`quota`
- 典型原因：
  - 账户欠费
  - 账单超限
  - 付款方式异常
  - 供应商项目级计费限制触发
- 排查优先级：
  1. 去供应商控制台核对账单和余额
  2. 查是否是组织或项目级限制
  3. 查是否会波及该渠道全部模型
- 当前处理策略：
  - 当前归类上更接近 `insufficient_quota`
  - 候选耗尽后通常收敛为“当前分组可用上游额度不足，请稍后再试”

### 3.14 `account_deactivated`

- 来源：通常由上游直接返回
- 当前含义：
  - 上游账号已被停用、冻结或撤销
- 常见日志特征：
  - `code=account_deactivated`
  - 常与 `401/403` 一起出现
- 典型原因：
  - 供应商封禁账号
  - 账号违反平台策略
  - 账号被管理员停用
- 排查优先级：
  1. 去上游控制台确认账号状态
  2. 查是否为组织级封禁
  3. 查同账号下全部模型是否一起不可用
- 当前处理策略：
  - 当前监控逻辑会把它视为可自动禁用渠道的明确故障
  - 这是比一般认证失败更明确的永久性错误

### 3.15 `rate_limit_exceeded`

- 来源：通常由上游直接返回
- 当前含义：
  - 上游限流或某类消费额度超过限制
- 常见日志特征：
  - 常见状态可能是 `429`
  - 也可能出现为 `402`，但 `type=rate_limit_error`、`code=rate_limit_exceeded`
  - 消息中可能带 `limit_type`、`current`、`limit`、`reset_time`
- 典型原因：
  - QPS 限流
  - TPM / RPM 超限
  - 日额度、周额度或月额度超限
  - 供应商把消费封顶也复用到这个错误码
- 排查优先级：
  1. 先看 HTTP 状态和消息
  2. 再看是否带 `reset_time`、`limit_type`
  3. 判断它是“瞬时流量限流”还是“账单/消费上限”
- 当前处理策略：
  - 纯 `429` 语义时，更偏瞬时上游过载
  - 但如果像真实样例那样是 `402 + rate_limit_error + 周消费超限`，语义上更接近额度不足
  - 所以不能只看 `code`，必须同时看状态和消息

## 4. 快速区分表

| 错误码 | 来源 | 是否已发起上游请求 | 是否更偏网络/网关问题 | 是否更偏本地配置/计费问题 |
| --- | --- | --- | --- | --- |
| `upstream_transport_goaway` | Router 本地细分 | 是 | 是 | 否 |
| `do_request_failed` | Router 本地兜底 | 是 | 通常是 | 否 |
| `unsupported_image_billing` | Router 本地校验 | 否 | 否 | 是 |
| `invalid_api_key` | 上游直接返回 | 是 | 否 | 否，偏渠道密钥失效 |
| `insufficient_quota` | 上游直接返回 | 是 | 否 | 否，偏上游余额或额度不足 |
| `bad_response` | Router 本地兜底 | 是 | 通常是 | 否 |
| `bad_response_status_code` | Router 本地兜底 | 是 | 通常是 | 否 |
| `model_not_found` | 上游或 Router | 是或否，取决于场景 | 否 | 否，偏能力配置 |
| `unsupported_channel_endpoint` | Router 本地包装 | 否，通常在本地能力判定阶段 | 否 | 否，偏端点能力声明 |
| `authentication_error` | 上游直接返回 | 是 | 否 | 否，偏渠道认证失效 |
| `permission_error` | 上游直接返回 | 是 | 否 | 否，偏模型或资源权限 |
| `forbidden` | 上游直接返回 | 是 | 否 | 否，偏权限或账号状态 |
| `billing_error` | 上游直接返回 | 是 | 否 | 否，偏上游计费或账单 |
| `account_deactivated` | 上游直接返回 | 是 | 否 | 否，偏账号停用 |
| `rate_limit_exceeded` | 上游直接返回 | 是 | 视场景而定 | 否，有时更像消费上限 |
