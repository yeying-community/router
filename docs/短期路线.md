# 短期路线

本文档从 `里程碑.md` 拆出，用来沉淀未来一个迭代周期内可以开发、可以验收、可以复盘的具体事项。

目标不是罗列愿望，而是把短期建设拆成明确交付物。每个事项都需要能回答：

- 做什么
- 交付到哪里
- 怎么验收
- 哪些边界先不做

## 1. 执行原则

### 1.1 优先补运营闭环

优先开发能让运营从“发现问题”走到“处理完成”的能力。

基本闭环：

1. 看见问题
2. 定位对象
3. 进入明细
4. 执行动作
5. 留下记录

### 1.2 优先补路由可解释性

短期内不以“接入更多模型数量”为主目标。

更重要的是让一次请求能被解释：

- 为什么命中这个模型
- 为什么命中这个渠道
- 为什么 fallback
- 为什么失败
- 成本和扣费口径是什么

### 1.3 每个事项必须可验收

进入开发前必须明确验收方式。

验收方式可以是：

- 页面可见
- 接口返回字段可核对
- 数据库记录可追溯
- 操作后状态变化明确
- 测试用例覆盖核心分支

## 2. P0：运营概览到用户处理闭环

### 2.1 用户分层列表联动

开发事项：

- 系统概览 / 用户页保留高消费、高活跃、长尾、余额风险等核心分层。
- 每个分层卡片提供明确入口，跳转到用户列表并自动带入筛选条件。
- 用户列表顶部展示当前筛选来源和命中用户数。
- 用户列表支持导出当前筛选结果。

验收标准：

- 从系统概览点击任一用户分层后，能进入用户列表。
- 用户列表只展示该分层命中的用户。
- 页面能显示筛选条件摘要和命中数量。
- 导出的用户名单与当前列表筛选结果一致。

暂不做：

- 复杂自定义人群规则编辑器。
- 多维度交叉画像配置。

### 2.2 用户批量动作

开发事项：

- 用户列表支持批量选择。
- 支持至少一个可落地批量动作：批量发放余额、批量分配套餐、批量生成兑换码三选一优先完成。
- 批量动作需要二次确认。
- 操作结果写入业务流水或操作日志。

验收标准：

- 运营可以从筛选后的用户列表选择多个用户。
- 执行批量动作后，目标用户状态发生对应变化。
- 失败用户和成功用户能在结果中区分。
- 用户详情或业务流水中能追溯本次批量动作。

暂不做：

- 批量动作审批流。
- 复杂任务队列可视化。

### 2.3 用户详情运营面板

开发事项：

- 用户详情页补齐运营判断所需信息：余额、生效套餐、最近消费、最近充值 / 兑换、最近请求模型。
- 用户详情页提供常用单用户动作入口：发放余额、分配套餐、查看流水、复制用户 ID。
- 钱包、用户 ID 等基础字段样式与页面其他字段保持一致。

验收标准：

- 运营进入用户详情后，不需要跳转其他页面即可判断用户余额和套餐状态。
- 至少一个单用户运营动作可以在详情页完成。
- 单用户动作完成后，页面状态刷新或给出明确结果提示。
- 基础字段字号、行高、复制按钮策略与同页其他字段一致。

暂不做：

- 用户生命周期自动推荐策略。
- 用户价值评分模型。

## 3. P0：日志与路由解释

### 3.1 请求日志补齐路由字段

开发事项：

- `event_logs` 补齐请求路由和计费估算所需字段。
- 日志库复用主库时也必须执行 log-scope migrations。
- 请求日志记录估算 token、估算器、估算精度、命中渠道、命中模型、端点协议、fallback 状态。
- 写入失败时要有明确错误日志，避免静默丢字段。

验收标准：

- 新环境启动后，`event_logs` 包含新增字段。
- `LogSQLDSN` 为空时，日志表复用主库也能完成日志字段迁移。
- 一次真实请求后，日志中能查到估算 token 和路由命中信息。
- 定向测试覆盖日志迁移路径或迁移函数。

暂不做：

- 外部观测平台集成。
- 全量链路追踪系统。

### 3.2 日志详情路由解释区

开发事项：

- 日志详情页增加路由解释区。
- 展示请求模型、最终模型、命中渠道、供应商、端点协议、fallback 次数、失败原因。
- 对常见失败原因做归类：额度、权限、模型不可用、渠道不可用、上游错误、请求格式。

验收标准：

- 打开一条成功日志，可以看到最终命中的模型和渠道。
- 打开一条失败日志，可以看到主要失败分类和上游错误摘要。
- fallback 请求能展示至少一次 fallback 的尝试结果。
- 页面字段为空时有稳定占位，不出现布局错乱。

暂不做：

- 完整候选渠道打分解释。
- 复杂链路图可视化。

## 4. P0：余额不足自动禁用与恢复

### 4.1 余额不足禁用状态记录

开发事项：

- 渠道因余额不足被自动禁用时，写入 circuit breaker `canceled` 状态。
- 状态 reason 固定为 `insufficient balance`。
- 即使之前没有熔断记录，也要能创建该状态。

验收标准：

- 余额不足触发自动禁用后，渠道状态变为自动禁用。
- `channel_circuit_breaker_states` 中存在 `canceled / insufficient balance` 记录。
- 对应事件写入 `channel_circuit_breaker_events`。
- 测试覆盖无历史熔断记录时的 canceled 状态创建。

暂不做：

- 把余额不足状态改造成独立状态机。
- 自动充值或自动采购。

### 4.2 账务刷新纳入自动禁用渠道

开发事项：

- 自动账务刷新需要包含启用渠道。
- 自动账务刷新也需要包含“因余额不足自动禁用”的渠道。
- 手动禁用渠道不参与自动恢复。
- 其他原因自动禁用的渠道不参与余额恢复。

验收标准：

- 启用渠道会进入自动账务刷新队列。
- `AutoDisabled + canceled / insufficient balance` 渠道会进入自动账务刷新队列。
- 手动禁用渠道不会进入自动恢复路径。
- 测试覆盖启用、余额不足自动禁用、手动禁用三种状态。

暂不做：

- 所有自动禁用原因的统一恢复策略。
- 多原因恢复优先级。

### 4.3 账务可用后自动恢复

开发事项：

- 账务刷新后，如果套餐、日额度、周额度或总额度可用，恢复余额不足自动禁用渠道为启用。
- 恢复动作只针对 `AutoDisabled + canceled / insufficient balance` 渠道。
- 恢复后保留原 circuit breaker `canceled` 记录，作为历史原因。

验收标准：

- 余额不足自动禁用渠道在账务恢复可用后，状态变为启用。
- 手动禁用渠道即使账务可用，也不会被自动启用。
- 非余额不足原因自动禁用的渠道不会被该逻辑恢复。
- 测试覆盖恢复判定的正向和反向场景。

暂不做：

- 恢复后自动发通知的模板优化。
- 恢复后立即执行模型连通性测试。

## 5. P1：套餐可见范围与名单保留

### 5.1 套餐可见范围切换

开发事项：

- 套餐支持全部用户可见和部分用户可见。
- 从全部用户可见切换为部分用户可见时，即使当前名单为空也允许保存。
- 部分用户可见名单为空表示当前没有用户可见，后续可通过新增按钮添加用户。

验收标准：

- 可见范围从全部用户可见切换到部分用户可见时，空名单可以保存。
- 页面不再提示“部分用户可见时必须选择用户”。
- 保存后再次进入页面，显示为部分用户可见且名单为空。

暂不做：

- 按用户分组动态可见。
- 用户标签规则可见。

### 5.2 全部可见时保留历史名单

开发事项：

- 从部分用户可见切换到全部用户可见时，保留已选择用户名单。
- 全部用户可见状态下，名单只是不生效。
- 再切回部分用户可见时，自动恢复之前名单。

验收标准：

- 部分用户可见状态下选择用户 A、B。
- 切换到全部用户可见并保存后，数据库仍保留 A、B。
- 再切回部分用户可见时，页面自动显示 A、B。
- 全部用户可见期间，实际可见范围不受名单限制。

暂不做：

- 多套名单版本管理。
- 名单变更审批。

## 6. P1：路由策略中心第一版

### 6.1 分组模型与渠道可见性

开发事项：

- 后台提供分组可用模型配置入口。
- 后台提供分组可用渠道配置入口。
- `/v1/models` 返回结果与分组模型可见性一致。
- 实际请求路由与分组渠道可见性一致。

验收标准：

- 管理员关闭某分组的某模型后，该分组用户在 `/v1/models` 看不到该模型。
- 该分组用户请求被关闭模型时，返回明确权限或模型不可用错误。
- 管理员移除某分组的某渠道后，实际请求不会命中该渠道。
- 测试覆盖 `/v1/models` 和实际路由的可见性一致性。

暂不做：

- 复杂策略 DSL。
- 多维权重自动调参。

### 6.2 fallback 策略开关

开发事项：

- 支持按分组或全局配置 fallback 开关。
- 日志记录 fallback 是否发生和尝试次数。
- fallback 关闭时，请求失败不再继续尝试备用渠道。

验收标准：

- fallback 开启时，主渠道失败后会尝试可用备用渠道。
- fallback 关闭时，主渠道失败后直接返回错误。
- 日志能区分 fallback 开启、关闭和实际发生状态。

暂不做：

- 按错误类型配置 fallback。
- 跨区域 fallback 策略。

## 7. P1：成本与毛利基础视图

### 7.1 核心财务口径统一

开发事项：

- 明确用户消费、用户充值、套餐收入、渠道成本、模型成本、毛利的计算口径。
- 形成文档并同步到相关页面字段说明。
- 后端接口返回字段命名保持一致。

验收标准：

- 同一周期内，系统概览、用户详情、渠道详情中的消费口径一致。
- 字段命名不混用收入、消费、扣费、成本。
- 文档中能查到每个核心指标的计算来源。

暂不做：

- 财务级对账。
- 税费、退款、发票口径。

### 7.2 系统概览经营指标

开发事项：

- 系统概览展示周期收入、消费、成本、粗略毛利。
- 支持至少今日、近 7 天、近 30 天三个周期。
- 指标卡片能跳转到相关明细或列表。

验收标准：

- 切换周期后，四个经营指标同步刷新。
- 指标为空时显示 0 或明确占位。
- 点击指标能进入对应明细页面或带筛选列表。

暂不做：

- 自定义任意日期范围。
- 复杂利润报表。

## 8. P2：异常待办入口

### 8.1 待处理事项列表

开发事项：

- 系统概览增加待处理事项入口。
- 第一版支持余额风险用户、套餐即将过期、渠道余额不足、模型测试失败四类事项。
- 每条事项可跳转到对应用户、套餐、渠道或模型页面。
- 事项需要有状态：待处理、已处理、忽略。

验收标准：

- 系统能生成至少四类待办中的两类。
- 每条待办可以跳转到明确对象。
- 操作已处理或忽略后，该事项不再出现在默认待处理列表。

暂不做：

- 复杂 SLA 规则。
- 多角色待办分派。

## 9. 优先级与验收节奏

| 优先级 | 事项 | 建议验收方式 |
| --- | --- | --- |
| P0 | 用户分层列表联动 | 页面跳转、筛选结果、导出结果核对 |
| P0 | 用户批量动作 | 批量操作、状态变化、流水记录核对 |
| P0 | 用户详情运营面板 | 页面字段、单用户动作、结果刷新核对 |
| P0 | 请求日志补齐路由字段 | 迁移、真实请求日志、定向测试核对 |
| P0 | 日志详情路由解释区 | 成功、失败、fallback 三类日志核对 |
| P0 | 余额不足自动禁用与恢复 | 状态变化、熔断记录、自动刷新测试核对 |
| P1 | 套餐可见范围与名单保留 | 保存行为、名单保留、权限效果核对 |
| P1 | 路由策略中心第一版 | `/v1/models` 与实际路由一致性核对 |
| P1 | 成本与毛利基础视图 | 指标口径、周期切换、明细跳转核对 |
| P2 | 异常待办入口 | 待办生成、跳转、状态流转核对 |

## 10. 不做清单

短期内建议暂不投入：

- 大规模重做聊天前台
- 模型资讯站或模型排行榜内容站
- 以支持更多模型数量作为主要开发目标
- 复杂工作流编排
- 复杂多级代理商体系
- 未形成扣费闭环的营销活动系统
- 自定义人群规则编辑器
- 复杂财务对账系统
