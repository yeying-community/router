# 文档索引与治理规则

本目录用于沉淀 Router 的对外能力、运营策略和演进决策。  
目标是“文档可执行、可核对、不过度重复”。

## 1. 文档分层（谁是主文档）

- `接口文档.md`：API 能力入口文档，面向前后端联调和运营排障。
- `路由逻辑.md`：Relay 选路、端点协商与失败重试逻辑。
- `上游错误码.md`：上游错误码、Router 本地包装错误码与运行排查口径。
- `模型端点.md`：模型端点治理总览入口，用来串联供应商规范、渠道规范、分组规范与 `/v1/models` 透出规则。
- `表格列宽规范.md`：后台列表页列宽、操作列和水平滚动的统一实现规则。
- `供应商规范.md`：供应商层职责、官方信息来源、migration 边界与模型状态规则。
- `渠道规范.md`：渠道层职责、模型/端点人工配置规则、上游同步与测试边界。
- `分组规范.md`：分组层职责、`group_channels` / `group_model_channels` 规则与用户接口暴露边界。
- `端点协议.md`：Router 当前支持的端点协议、协议差异与供应商协议选择规范。
- `渠道模型兼容策略.md`：渠道模型端点内的细粒度能力声明、请求修正规则与审计设计。
- `模型计费.md`：模型定价解析、复杂计费条件匹配与单位换算规则。
- `扣费逻辑.md`：YYC 语义、金额基线与请求期账本模型。
- `词元计算方案.md`：请求前 token 估算的统一方案文档，包含设计原则、当前实现和后续演进。
- `支付方案.md`：充值/支付链路规范、联调与验收落地。
- `运营文档.md`：运营侧实操说明。
- `待办.md`：仅保留未完成事项，不记录已完成项。

## 2. 治理规则

1. `待办.md` 只写未完成事项；已完成事项从待办删除，不做“已完成列表”堆积。
2. 涉及 API 变更时，必须同步更新：
- `接口文档.md`
- 若是支付/路由/额度相关，还需同步对应专题文档。
3. 专题文档不重复罗列接口明细，接口明细以 `接口文档.md` 为准。
4. 文档中的代码路径统一使用仓库相对路径，避免写绝对路径。
5. 涉及“兼容/历史字段”时，必须明确：
- 当前是否仍生效
- 仅兼容读取还是仍会写入
- 计划何时移除

## 3. 最小核对清单（提交前）

1. 路由是否存在：对照 `internal/transport/http/router/api.go`。
2. 字段是否存在：对照 `internal/admin/model/*` 与 controller 返回结构。
3. 页面行为是否一致：对照 `web/src/pages/*` 和 `web/src/components/*`。
4. 待办是否有冗余：确认“已做完”的事项已移出 `待办.md`。
