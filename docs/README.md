# 文档索引与治理规则

本目录用于沉淀 Router 的对外能力、运营策略和演进决策。  
目标是“文档可执行、可核对、不过度重复”。

## 1. 文档分层（谁是主文档）

- `接口文档.md`：API 能力入口文档，面向前后端联调和运营排障。
- `swagger/openapi.json`：接口机器可读契约（由注释生成）。
- `路由逻辑.md`：Relay 选路、端点协商、失败重试与桥接逻辑。
- `渠道协议.md`：协议选择与上游接入规范。
- `模型计费.md`：模型定价解析、复杂计费条件匹配与单位换算规则。
- `扣费逻辑.md`：YYC 语义、金额基线与请求期账本模型。
- `支付方案.md`：充值/支付链路规范、联调与验收落地。
- `运营文档.md`：运营侧实操说明。
- `待办.md`：仅保留未完成事项，不记录已完成项。

## 2. 治理规则

1. `待办.md` 只写未完成事项；已完成事项从待办删除，不做“已完成列表”堆积。
2. 涉及 API 变更时，必须同步更新：
- `接口文档.md`
- `swagger`（注释生成）
- 若是支付/路由/额度相关，还需同步对应专题文档。
3. 专题文档不重复罗列接口明细，接口明细以 `接口文档.md` 为准。
4. 文档中的代码路径统一使用仓库相对路径，避免写绝对路径。
5. 涉及“兼容/历史字段”时，必须明确：
- 当前是否仍生效
- 仅兼容读取还是仍会写入
- 计划何时移除

## 3. 最小核对清单（提交前）

1. 路由是否存在：对照 `internal/transport/http/router/api.go`。
2. 字段是否存在：对照 `internal/admin/model/*` 与 controller 返回结构。
3. 页面行为是否一致：对照 `web/src/pages/*` 和 `web/src/components/*`。
4. 待办是否有冗余：确认“已做完”的事项已移出 `待办.md`。
