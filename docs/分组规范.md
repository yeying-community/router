# 分组规范

本文档定义分组层的职责、变更边界、静态展开规则，以及与供应商层、渠道层的关系。

## 文档信息

- 最后核对日期：2026-05-17
- 关键代码路径：
  - `internal/admin/model/group_model_binding.go`
  - `internal/admin/model/group_channel.go`
  - `internal/admin/model/group_model_channel.go`
  - `internal/admin/controller/group/handler.go`
  - `internal/admin/repository/group/repository.go`

## 摘要

本层拥有：

1. 分组暴露哪些模型。
2. 分组接入哪些渠道。
3. 分组倍率、启用状态、描述等运营字段。

本层不拥有：

1. 供应商官方模型信息。
2. 渠道模型是否启用。
3. 渠道端点是否启用。

本层修改入口：

1. 管理员保存 `group_models` 与 `group_channels`。
2. `group_model_channels` 只按当前人工配置整理，不能反向改写人工配置。

## 1. 本层职责

分组不是展示字段，而是系统的核心运营边界。分组层负责四件事：

1. 决定某分组对外暴露哪些模型。
2. 决定某分组可接入哪些渠道。
3. 决定该分组的静态路由范围。
4. 决定该分组的业务倍率和配额边界。

一句话：

`分组 = 模型暴露边界 + 渠道路由边界 + 计费边界 + 配额边界`

分组层不负责定义渠道端点真值。

## 2. 与上层的边界

基于最新的供应商规范和渠道规范，分组层必须明确不拥有以下真值：

1. 供应商官方模型信息。
2. 供应商官方价格。
3. 供应商官方端点范围。
4. 渠道模型是否启用。
5. 渠道端点是否启用。
6. 某渠道最近一次上游返回是否包含某模型。
7. 某渠道最近一次端点测试是否成功。

这些都属于供应商层或渠道层。

分组层只拥有：

1. 这个分组要不要暴露某模型。
2. 这个分组要不要接入某渠道。
3. 这个分组内的渠道优先级。
4. 这个分组的业务倍率、启用状态、描述等运营字段。

## 3. 本层关键结构

当前主要结构如下：

1. `users.group`
   - 表达用户属于哪个分组。
   - 是用户访问能力的入口维度。
2. `groups`
   - 表达分组元信息。
   - 主要承载 `name`、`description`、`billing_ratio`、`enabled`、`sort_order`、`source`。
3. `group_models`
   - 表达某分组对外声明提供哪些模型。
   - 同时承载分组模型的 `provider` 主值。
4. `group_channels`
   - 表达某分组接入哪些渠道。
   - 同时承载分组内渠道优先级。
5. `group_model_channels`
   - 由 `group_models + group_channels + channel_models` 刷新得到。
   - 表达分组模型到渠道模型的静态展开关系。

这里要再强调一次：

1. `group_models` 和 `group_channels` 是人工配置。
2. `group_model_channels` 不是人工配置，而是静态展开关系。
3. 静态展开关系可以刷新，但不能反向定义人工配置。

术语说明：

1. 本文档统一使用 `group_channels` 作为分组渠道人工配置的业务名称。
2. 当前数据库表名、结构名、函数名都应统一使用 `group_channels`。
3. 本文档统一使用 `group_model_channels` 作为分组模型静态展开关系的业务名称。
4. 当前实现中的业务命名统一为 `group_model_channels`。

## 4. 当前实现合理性评估

### 4.1 当前整体方向是合理的

基于最新分层语义，分组层当前的三张主表加一张静态展开表的方向是合理的：

1. `group_models` 管“要暴露什么模型”。
2. `group_channels` 管“可用哪些渠道”。
3. `group_model_channels` 管“分组模型静态展开到哪些渠道模型”。

这比把所有语义都塞进一张分组路由表更干净，也更符合供应商层和渠道层已经确定下来的边界。

### 4.2 当前最合理的点

1. 分组层没有直接拥有渠道端点真值。
2. 分组对外能力来自渠道层，而不是直接来自供应商层。
3. `group_model_channels` 是静态展开表，不承担人工配置语义。
4. 分组倍率独立存在于 `groups.billing_ratio`，语义清晰。

### 4.3 当前仍需持续守住的边界

1. 分组层不能因为渠道刷新或供应商变化，自动改 `group_models` 人工配置。
2. 分组层不能把供应商官方端点范围直接暴露给用户。
3. 分组层不能绕过渠道端点真值去扩充 `supported_endpoints`。
4. 管理端不能把 `group_model_channels` 当成人工配置直接编辑。

### 4.4 当前最关键的结论

分组层是“暴露与路由控制层”，不是“渠道能力真值层”。

更具体地说：

1. 分组层可以收缩能力。
2. 分组层不能扩张渠道层没有真实启用的能力。
3. 分组层只能消费渠道层真值，不能自创真值。

## 5. 标准定义

### 5.1 人工配置与静态展开表边界

1. `group_models` 是分组模型人工配置。
2. `group_channels` 是分组渠道人工配置。
3. `group_model_channels` 是静态展开表。

因此：

1. 管理端应修改人工配置表。
2. `group_model_channels` 可以按当前人工配置整体刷新。
3. 静态展开表不能反向成为上游人工配置真值。

进一步明确：

1. 管理员保存分组时，真正保存的是“暴露哪些模型”和“绑定哪些渠道”。
2. 分组保存成功后，系统再根据当前人工配置刷新静态展开关系。
3. `group_channels` 变化后，系统必须同步刷新 `group_model_channels`。
4. `group_models` 变化后，系统必须同步刷新 `group_model_channels`。
5. 刷新 `group_model_channels` 不等于越过管理员单独修改其他层配置。
6. `group_channels` 和 `group_model_channels` 的变化，不得反推修改 `group_models`。

### 5.2 分组层不拥有的配置

分组层明确不拥有：

1. `channel_model_endpoints.enabled`
2. 渠道模型是否真实支持某端点
3. 供应商官方端点范围
4. 渠道最近一次上游返回结果
5. 渠道最近一次端点测试结果
6. 渠道模型人工启停原因
7. 渠道端点人工启停原因

这些都只能作为分组管理端的解释信息，不能写回分组人工配置。

### 5.3 分组静态展开刷新规范

分组层的静态展开刷新任务可以：

1. 根据 `group_models` 与 `group_channels` 整理 `group_model_channels`。
2. 根据渠道模型人工配置，刷新分组模型到渠道模型的静态关系。
3. 维护 `group`、`model`、`channel_id`、`upstream_model`、`provider`、`priority` 这些静态字段。
4. 不在这张表里写入临时连通性、测试状态或上游返回状态。

分组层的静态展开刷新任务不可以：

1. 反向修改 `channel_model_endpoints`。
2. 反向修改 `channel_models` 人工配置。
3. 因分组配置变化而删除渠道层端点真值。
4. 直接把供应商官方端点范围当成分组真实可用端点。
5. 把渠道最近一次上游返回结果直接持久化成分组自己的真值。
6. 把渠道最近一次测试结果直接持久化成分组自己的真值。
7. 根据静态展开结果新增、删除、启用、禁用 `group_models`。

### 5.4 分组人工配置保存规则

管理员保存分组配置时：

1. 保存 `group_models` 是在声明“这个分组想暴露哪些模型”。
2. 保存 `group_channels` 是在声明“这个分组想接入哪些渠道”。
3. 保存成功后，应刷新 `group_model_channels`。
4. 刷新结果只能来自当前分组人工配置。

还需要明确保存失败和保存成功的语义：

1. 分组人工配置一旦保存成功，就持续有效，除非管理员再次主动修改。
2. 渠道临时异常、测试失败、上游波动，不得自动改写分组人工配置。
3. 这些变化由通知机制提示管理员处理，而不是由分组层反复做可用性判定。
4. 即使某个分组模型当前没有任何可展开的渠道承载关系，这条 `group_models` 人工配置也应保留，直到管理员主动删除或关闭。

### 5.5 分组运行时规则

请求处理流程应保持为：

1. 鉴权后拿到用户分组。
2. 根据 `group + model` 在 `group_model_channels` 中找静态路由范围内的渠道。
3. 实际暴露的端点来自这些可达渠道里真实启用端点的汇总结果。
4. 计费时读取该分组业务倍率。

其中第 3 点必须进一步标准化：

1. `/v1/models` 面向的是“当前分组声明且存在静态渠道承载关系的模型”。
2. `/v1/models.supported_endpoints` 面向的是“当前分组下、静态路由范围内、真实已启用端点的汇总结果”。
3. 某个端点只有在至少一个静态路由范围内渠道对应的真实启用端点存在时，才能对外暴露。
4. 供应商官方端点范围、渠道官方范围、测试信息，都不能直接作为用户接口返回值。

对文本生成类模型，`supported_endpoints` 只能从 `/v1/chat/completions`、`/v1/responses`、`/v1/messages` 中取值，并且必须是“供应商模型官方端点范围”和“当前分组可达渠道真实启用端点”的交集。交集为空时，该模型不应作为可调用文本服务对外暴露。

分组层不应在请求时直接读取供应商官方端点范围作为响应依据。

这里还需要明确一条：

1. `group_models` 负责定义“这个分组声明暴露哪些模型”。
2. `group_model_channels` 负责定义“这些模型当前能路由到哪些渠道”。
3. 即使 `group_model_channels` 为空，也不得自动删除对应的 `group_models`。

## 6. 与渠道层的关系

分组层强依赖渠道层真值。

具体来说：

1. 分组使用的模型和渠道范围，来自已保存的渠道人工配置。
2. 渠道临时异常、测试失败、上游波动，由渠道层通知管理员处理。
3. 分组层不能把这些变化回写成自己的配置判断。
4. 如果某渠道已经被 `group_channels` 接入，渠道侧不能静默禁用该渠道。
5. 渠道禁用前应先做分组引用检查；若仍被分组使用，阻止禁用，由管理员先处理分组绑定或分组模型。

分组层对渠道层的正确依赖方式应该是：

1. 消费渠道人工配置。
2. 接收渠道侧通知作为运维信号。
3. 但不把通知和事实记录误当成自己的人工配置。

管理端语义也要统一：

1. 分组管理端以人工配置为主，不反复生成“当前是否可用”的结论。
2. 渠道侧变化由通知和渠道管理页解释，不在分组页制造额外状态语义。
3. 分组管理端不能提供绕过渠道人工配置的强制启用能力。

## 7. 与供应商层的关系

分组层不直接把供应商层当成分组暴露真值。

供应商层只会间接影响分组层：

1. 通过渠道层可配置范围变化，影响未来可被渠道采纳的模型与端点。
2. 通过渠道层已启用能力变化，影响最终分组可路由结果。
3. 通过官方状态、官方价格、官方端点范围，影响管理端解释与校验提示。

但供应商层不会直接决定分组路由真值。

也就是说：

1. 供应商有某模型，不代表分组一定暴露它。
2. 供应商把某模型标记为 `deprecated`，不代表分组自动删除它。
3. 分组最终暴露范围由分组人工配置和渠道人工配置共同决定。

## 8. 用户接口暴露规范

分组层对 `/v1/models` 与 `/v1/models.supported_endpoints` 的输出必须满足：

1. 只暴露当前分组可见模型。
2. 只暴露当前分组可达渠道里真实启用的端点。
3. 不得把供应商官方端点范围直接原样返回给用户。
4. 不得把已被渠道层人工禁用或系统禁用的端点暴露为可用。
5. 文本生成类模型只在 `/v1/chat/completions`、`/v1/responses`、`/v1/messages` 至少存在一个可用交集端点时对外提供服务。

这里的“可见模型”再具体一点：

1. 先看分组人工配置是否声明该模型。
2. 再看是否存在对应的 `group_model_channels` 静态展开关系。
3. 满足后即可稳定出现在用户接口。

如果后续产品需要“先展示、后不可用”的产品语义，也必须单独设计字段，不能污染当前分组暴露接口。

## 9. 管理说明

### 9.1 管理接口

当前分组管理接口位于：

1. `GET /api/v1/admin/groups`
2. `POST /api/v1/admin/group`
3. `PUT /api/v1/admin/group`
4. `DELETE /api/v1/admin/group/:id`
5. `GET /api/v1/admin/group/:id/channels`
6. `PUT /api/v1/admin/group/:id/channels`
7. `GET /api/v1/admin/group/:id/models`
8. `PUT /api/v1/admin/group/:id/models`
9. `PUT /api/v1/admin/group/:id/models/:model`

前端入口：管理员 Header 中“分组”Tab。

接口字段约定：

1. 分组写接口中，分组模型字段统一使用 `models`。
2. `models` 的每一项表达一条分组模型绑定关系，至少包含：
   - `model`
   - `channel_id`
   - `upstream_model`
3. `enabled`、`priority` 作为同一条分组模型绑定关系的附加字段写入。
4. 分组层不再使用 `model_configs` 作为分组模型写接口字段名。

### 9.1.1 管理端展示口径

分组管理端同样必须把三类信息拆开显示：

1. 人工配置
   - `groups` 的启用状态、倍率、描述。
   - `group_models` 声明的暴露模型。
   - `group_channels` 声明的绑定渠道和优先级。
2. 静态展开结果
   - 当前哪些 `group_model_channels` 已生成。
   - 当前哪些模型还没有生成任何渠道映射。
3. 解释信息
   - 渠道侧最近的通知或异常说明。
   - 渠道侧为何建议管理员调整配置。

分组管理端必须遵守：

1. 人工配置可编辑，静态展开结果只读。
2. 静态展开结果不能反向改写人工配置。
3. 解释信息来自渠道侧通知和事实记录，不能保存回分组人工配置。
4. 不能让管理员误以为“渠道有异常”就等于“分组没配置该模型”。

### 9.2 管理注意事项

1. 正在被用户、渠道或当前对外能力使用的分组不允许删除。
2. 仅创建 `groups` 行并不会自动生效。
3. 要让分组真正可用，至少要同时满足：
   - 用户被分配到该分组
   - 分组已绑定渠道
   - 分组已配置模型，且静态展开结果已正确刷新
4. 若修改了 `billing_ratio`，后续请求会按新的业务倍率计费。
5. 管理员维护分组时，应把分组人工配置和渠道侧异常处理分开看待。
6. 如果渠道侧有异常，应优先去渠道管理页或通知中心处理，而不是在分组页反复推断。

### 9.2.1 分组页字段展示要求

分组模型页至少应区分显示：

1. 已声明暴露的模型。
2. 每个模型对应的渠道映射。
3. 哪些模型当前还没有任何渠道映射，但人工配置仍然保留。
4. 需要管理员进一步编辑的缺口项。

分组渠道页至少应区分显示：

1. 已绑定渠道。
2. 每个渠道的优先级。
3. 需要管理员进一步编辑的缺口项。

如果渠道侧存在异常，UI 应引导管理员去渠道页查看原因，而不是在分组页制造新的状态判断。

### 9.3 常见问题

Q1: 为什么用户看不到某个模型？

通常是以下原因之一：

1. `group_models` 没有声明该模型。
2. 分组没有绑定可用渠道。
3. 绑定渠道里没有真实启用该模型。
4. `group_model_channels` 尚未正确刷新。

Q2: 为什么请求报“当前分组下无可用渠道”？

常见原因是：

1. 分组未绑定渠道。
2. 分组绑定了渠道，但渠道未启用对应模型。
3. 相关渠道状态禁用。
4. 运行时路由展开结果为空。
5. 相关渠道虽然有模型，但当前没有真实启用的可达端点。

Q3: 分组只是前端分类吗？

不是。分组直接影响：

1. 后端路由决策。
2. 用户可见模型范围。
3. 业务倍率计费。
4. 配额边界。

## 10. 建议表结构

```sql
CREATE TABLE group_channels (
  "group"      varchar(32)  NOT NULL,
  channel_id   varchar(64)  NOT NULL,
  enabled      boolean      NOT NULL DEFAULT true,
  priority     bigint       NOT NULL DEFAULT 0,
  created_at   bigint       NOT NULL DEFAULT 0,
  updated_at   bigint       NOT NULL DEFAULT 0,
  PRIMARY KEY ("group", channel_id)
);

CREATE INDEX idx_group_channels_channel_id
  ON group_channels (channel_id);

CREATE INDEX idx_group_channels_priority
  ON group_channels ("group", enabled, priority);
```

约束建议：

1. `group_channels` 是分组接入渠道的人工配置关系。
2. `group_models` 是分组模型的人工配置关系。
3. `group_model_channels` 只能为已存在于 `group_channels` 的渠道生成记录。
4. 关闭或删除某个分组渠道绑定时，应同步刷新该分组该渠道对应的静态展开关系。

## 11. 稳健性要求

分组层后续设计必须满足：

1. 分组重建不覆盖渠道层人工真值。
   - 这里的“分组重建”仅指 `group_model_channels` 的静态展开刷新。
2. 分组展示与实际选路使用同一套渠道能力输入。
3. 分组层只做暴露与路由控制，不扩权到渠道端点治理。
4. 分组层不把供应商官方信息误当成分组暴露真值。
5. 分组层不把渠道事实记录误当成分组人工配置。

## 12. 当前实现收口点

当前实现里，`group_model_channels` 的写入入口应持续保持收口，至少需要守住以下几类边界：

1. 分组保存、分组渠道保存、分组模型保存时刷新这张表。
   - 这类入口符合静态展开语义，应保留。
2. 渠道模型人工配置变更时刷新这张表。
   - 这类入口符合静态展开语义，应保留。
3. 渠道手动启用、手动禁用时刷新这张表相关缓存。
   - 这类入口只能做静态展开同步或缓存刷新，不能再写表内状态位。
4. 渠道测试完成后刷新这张表。
   - 这类入口不符合静态展开语义，应移除。

因此，后续改造顺序应明确为：

1. 继续禁止测试、连通性、上游波动触发动态写入这张表。
2. 只允许分组配置变更和渠道人工配置变更触发静态展开刷新。
3. 继续保证管理端不直接编辑这张表。
