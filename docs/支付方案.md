# 支付方案（定稿）

本文档是 Router 充值与支付链路的确定性规范。  
涉及实现边界、接口职责、状态流转、鉴权签名、履约与排障口径，统一以本文为准。

## 1. 适用范围

1. 用户余额充值（`balance_topup`）。
2. 用户套餐购买/续费/升级（`package_purchase` + `operation_type`）。
3. 兑换码核销入账。
4. 外部支付回调、主动查单、取消支付。

## 2. 核心原则

1. 支付与入账解耦。
- 支付链路负责订单与支付状态，不直接承担兑换码核销。
- 兑换链路负责兑换码校验与入账，不负责外部支付创建。

2. 后端统一建单。
- 订单 ID、`transaction_id`、`redirect_url` 仅由 Router 后端生成。
- 前端不拼签名、不拼交易参数。

3. 幂等优先。
- 同商品重复下单优先复用未完成订单。
- 支付状态推进和履约按订单幂等执行，避免重复入账/重复开通套餐。

4. 账本先行。
- 余额变化以余额批次和交易流水为主。
- 业务表互引字段已删除，新链路仅依赖账本交易关系。

## 3. 接口与职责

1. `POST /api/v1/public/user/topup/orders`
- 创建外部支付订单。
- 返回 `id`、`transaction_id`、`status`、`redirect_url`、`created_at`。

2. `GET /api/v1/public/user/topup/orders`
- 当前用户支付订单列表。

3. `GET /api/v1/public/user/topup/orders/:id`
- 当前用户支付订单详情。

4. `POST /api/v1/public/user/topup/orders/:id/refresh`
- 主动查单并补推进状态。

5. `POST /api/v1/public/user/topup/orders/:id/cancel`
- 取消未完成订单。

6. `POST /api/v1/public/topup/callback`
- 外部支付系统回调，推进订单状态。

7. `POST /api/v1/public/user/topup`
- 兑换码核销入账（`code`，兼容 `key`）。

8. `GET /api/v1/public/user/topup/balance/transactions`
9. `GET /api/v1/admin/user/:id/topup/balance/transactions`
- 余额批次交易明细查询（排障主入口）。

## 4. 下单模型（确定口径）

### 4.1 业务类型 `business_type`

1. `balance_topup`：账户余额充值。
2. `package_purchase`：套餐交易。

### 4.2 套餐操作类型 `operation_type`

1. `topup`：余额充值固定值（系统自动写入）。
2. `purchase`：购买套餐。
3. `renew`：续费套餐。
4. `upgrade`：升级套餐。

### 4.3 下单输入约束

1. `balance_topup`
- 可通过 `plan_id` 使用充值档位定价（推荐）。
- 无 `plan_id` 时，按 `amount + currency` 下单；`amount > 0` 必填。
- 若 `quota` 未传或为 `0`，由服务端按汇率折算。

2. `package_purchase`
- `package_id` 必填。
- 金额以后端套餐配置为准，不信任前端金额。
- `operation_type` 可选，不传时按系统推断（默认 `purchase`）。

3. `return_url`
- 可选，服务端会做安全清洗（移除支付敏感参数）。

## 5. 订单状态机（确定口径）

### 5.1 枚举

1. `created`
2. `pending`
3. `paid`
4. `fulfilled`
5. `failed`
6. `canceled`

### 5.2 主要流转

1. 创建订单：进入 `created`。
2. 支付中：可进入 `pending`。
3. 支付成功：进入 `paid`。
4. 履约完成：进入 `fulfilled`。
5. 支付失败：进入 `failed`。
6. 用户取消：进入 `canceled`。

### 5.3 关键约束

1. `paid` 之后才能履约。
2. `fulfilled`、`canceled` 状态不会被 `refresh` 改写。
3. 已支付订单不可取消。

## 6. 支付执行与补偿

1. 支付模式
- 支持 `api` 与 `redirect` 两种模式。
- 生产推荐 `api` 模式（外部支付 API 直接建单与查单）。

2. 重复下单复用
- 同用户、同商品、同关键参数命中 `created/pending/paid` 订单时复用。
- 复用命中 `paid` 时优先触发履约并直接返回结果。

3. 主动查单
- `refresh` 调外部查单接口。
- 外部已支付则补推进本地状态并触发履约。

4. 后台补偿
- 定时任务扫描 `top_up_api` 来源的 `created/pending/paid` 订单。
- 自动查单并补偿推进，降低“支付成功但回调丢失”的人工干预成本。

## 7. 回调与签名

### 7.1 回调鉴权

1. 令牌来源优先级：
- `system_settings.TopUpCallbackToken`
- 环境变量 `TOPUP_CALLBACK_TOKEN`

2. 允许的回调凭证头：
- `X-Topup-Callback-Token`
- `Authorization: Bearer <token>`

3. 回调定位订单支持：
- `order_id`
- `transaction_id`
- `provider_order_id`

### 7.2 对外请求签名（Router -> 支付网关）

签名规则固定如下：

1. 参数按 key 升序。
2. 空值不参与。
3. 不包含 `sign` 本身。
4. 末尾追加 `secret=<TopUpSignSecret>`。
5. 对拼接串做 `sha256`，输出 hex 小写。

说明：
- `operation_type` 必须参与签名。
- 余额充值固定 `operation_type=topup`。

## 8. 履约规则

### 8.1 余额充值履约

1. 订单 `quota` 入账到用户主余额。
2. 同步写入 `user_balance_lots` 批次。
3. 若配置有效期（`validity_days`），生成 `credit_expires_at`。
4. 超期批次通过过期任务扣减，保持余额与批次一致。
5. 余额实际扣减顺序采用“先到期先用”（FEFO），不是“最早充值先用”（FIFO）。
6. 扣减排序规则固定为：
   - `expires_at=0`（永久有效）排最后；
   - 其余按 `expires_at asc`；
   - 同到期时间按 `created_at asc`，再按 `id asc`。
7. 因此用户多次充值时，系统优先消耗“更早到期”的批次；仅在到期时间相同的情况下，才会按更早创建批次优先。

### 8.2 套餐履约

1. `purchase`：开通套餐。
2. `renew`：续期当前套餐。
3. `upgrade`：升级到目标套餐。

#### 8.2.1 升级（`upgrade`）扣费规则（当前实现）

1. 入口与触发：
- `business_type=package_purchase` 且 `operation_type=upgrade` 时，进入升级差价计算。
- `operation_type` 为空时，系统自动推断：无生效套餐=`purchase`，同套餐=`renew`，不同套餐=`upgrade`。

2. 基础金额：
- 先将“当前套餐售价”和“目标套餐售价”都折算为 `YYC`。
- `diffYYC = targetYYC - currentYYC`。
- 若 `diffYYC <= 0`，拒绝升级（无需补差价）。

3. 按剩余周期折算应付差价：
- `periodSeconds = active.expires_at - active.started_at`
- `remainingSeconds = active.expires_at - now`
- `payableYYC = ceil(diffYYC * remainingSeconds / periodSeconds)`
- 若当前套餐模板缺失，回退为“目标套餐全价 `YYC`”。

4. 币种与金额换算：
- 先得到 `payableYYC`，再按目标套餐币种换算应付金额。
- 货币金额向上取到分（2 位小数），并设置最小支付金额 `0.01`。

5. 升级后的生效时间：
- 升级履约时立即生效：`start_at=now`。
- 保留当前周期到期点：`expires_at=当前生效套餐 expires_at`（不额外延长周期）。

6. 边界行为：
- 指定 `upgrade` 但当前无生效套餐时，自动按 `purchase` 处理。
- 指定 `upgrade` 且目标套餐与当前套餐相同，返回错误，要求走 `renew`。

## 9. 兑换码规则

1. 核销入口：`POST /api/v1/public/user/topup`。
2. 请求字段：`code`（兼容 `key`）。
3. 兑换码有效期：`code_validity_days`（`0` 表示永久）。
4. 到账有效期：`credit_validity_days`（`0` 表示永久）。
5. 核销结果返回 `credit_expires_at`，用于前端展示到账额度到期时间。

## 10. 数据边界与去耦

1. 订单主表：`topup_orders`。
2. 兑换主表：`redemptions`。
3. 余额批次：`user_balance_lots`。
4. 余额交易：`user_balance_lot_transactions`。

5. 互引字段策略：
- `topup_orders.redemption_id` 与 `redemptions.topup_order_id` 已删除。
- 充值订单与兑换记录通过余额账本交易明细关联排障，不再通过业务表互引字段关联。
- 查询接口不透出互引字段。

## 11. 管理台展示口径

1. 业务流水-充值：只展示支付订单语义。
2. 业务流水-兑换：只展示兑换入账语义。
3. 排障优先看账本交易流水，不依赖业务表互引字段。

## 12. 明确不采用的旧口径

1. 不再使用 `POST /api/v1/public/user/topup` 的裸整数 `data` 返回。
2. 不再由前端拼接支付签名与交易参数。
3. 不再把支付与兑换混在同一业务语义中描述。

## 13. 联调配置矩阵

### 13.1 Router 侧必配

| 配置项 | 作用 | 典型值 |
| --- | --- | --- |
| `ServerAddress` | 生成回调基址 | `https://router.yeying.pub` |
| `TopUpMode` | 支付接入模式 | `api` |
| `TopUpAPICreateURL` | 外部支付下单地址 | `https://www.tidukongjian.com/.../api.external_pay/create` |
| `TopUpAPIQueryURL` | 外部支付查单地址 | `https://www.tidukongjian.com/.../api.external_pay/query` |
| `TopUpAPIUniacid` | 外部支付 `uniacid` | `1` |
| `TopUpMerchantApp` | 外部商户标识 | `router` |
| `TopUpSignSecret` | Router->支付网关签名密钥 | `router_pay_sign_202604` |
| `TopUpCallbackToken` | 校验支付回调 | `router_pay_callback_202604` |

说明：

1. `TopUpSignSecret` 与支付网关商户配置的 `sign_secret` 必须一致。
2. `TopUpCallbackToken` 与支付网关商户配置的 `callback_token` 必须一致。
3. `ServerAddress` 必须是外网可达 HTTPS 地址。

### 13.2 支付网关侧必配（Tidu 示例）

1. 微信支付参数：`wx_pay_app_id`、`wx_pay_mch_id`、`wx_pay_key`、证书路径。
2. 商户配置 `external_pay_merchants.router` 至少包含：
- `sign_secret`
- `callback_token`
- `callback_url`

建议 `callback_url` 显式配置为：

`https://router.yeying.pub/api/v1/public/topup/callback`

## 14. 联调与验收流程

### 14.1 联调前检查

1. Router：`TopUpMode=api`，建单/查单 URL、签名密钥、回调令牌已配置。
2. Router：至少有一个启用套餐，且配置了 `sale_price` / `sale_currency`。
3. Router：币种目录里有 `CNY`，且 `CNY -> YYC` 汇率正确。
4. 支付网关：商户配置与 Router 的密钥、令牌一致。
5. 网络：网关可访问 Router 回调地址，双方 HTTPS 正常。

### 14.2 联调步骤

1. 套餐购买链路
- 调 `POST /api/v1/public/user/topup/orders`（`business_type=package_purchase`）。
- 跳转 `redirect_url` 完成支付。
- 回调后订单应进入 `fulfilled`，套餐权益生效。

2. 余额充值链路
- 调 `POST /api/v1/public/user/topup/orders`（`business_type=balance_topup`）。
- 支付后订单应进入 `fulfilled`，余额增加。
- 若设定了档位有效期，检查 `credit_expires_at` 是否写入。

3. 失败分支最少覆盖 2 项
- 签名密钥不一致。
- 回调令牌不一致。
- 套餐未配置售价。
- 回调地址不可达。

### 14.3 验收标准

1. 套餐购买成功至少 1 次。
2. 余额充值成功至少 1 次。
3. 两条链路均完成真实支付、回调和履约。
4. 至少验证 2 个失败场景并有可复现记录。

## 15. 排障优先级

1. 先看订单状态和 `status_message`。
2. 再看支付回调日志与查单结果是否一致。
3. 再看余额批次与交易流水是否匹配订单履约。
4. 最后核对汇率、套餐售价和请求签名参数。
