# 充值接口兼容演进方案

## 1. 当前前提

当前项目的升级前提是：

1. 前端和后端会一起升级。
2. 不考虑前后端版本错配。
3. 使用同一个数据库，并允许同步演进数据库结构。

在这个前提下，`POST /api/v1/public/user/topup` 不需要为了“旧前端兼容”保留裸整数返回。

## 2. 当前结论

当前采用的方案是：

1. 保持兑换码入账路径不变：`POST /api/v1/public/user/topup`
2. 直接把 `topup.data` 从整数升级为结构化对象
3. 新增 `POST /api/v1/public/user/topup/orders`，由后端创建外部充值跳转订单
4. 新增 `POST /api/v1/public/topup/callback`，由外部系统推进订单状态
5. 前端不再自行拼接 `transaction_id`
6. 前端充值页、Swagger、接口文档同步升级

这是当前仓库下最直接、成本最低、也最不容易继续做乱的方案。

## 3. 当前接口形状

请求体：

```json
{
  "code": "redeem-xxxx-xxxx"
}
```

兼容请求别名：

- `key`

返回体：

```json
{
  "success": true,
  "message": "",
  "data": {
    "redeemed_yyc": 100000,
    "before_yyc_balance": 500000,
    "after_yyc_balance": 600000,
    "redemption_id": "474c9b7b534047ce9205220990f53087",
    "redemption_name": "新用户补偿",
    "group_id": "474c9b7b534047ce9205220990f53087",
    "group_name": "enterprise",
    "face_value_amount": 10,
    "face_value_unit": "USD",
    "redeemed_at": 1774000000
  }
}
```

## 4. 字段语义

| 字段 | 语义 |
| --- | --- |
| `redeemed_yyc` | 本次兑换成功写入账户的 `YYC` |
| `before_yyc_balance` | 兑换前用户账户总 `YYC` 余额 |
| `after_yyc_balance` | 兑换后用户账户总 `YYC` 余额 |
| `redemption_id` | 兑换码记录 ID |
| `redemption_name` | 兑换码名称或批次名 |
| `group_id` | 兑换码绑定的分组 ID |
| `group_name` | 兑换码绑定的分组名称 |
| `face_value_amount` | 兑换码原始面值 |
| `face_value_unit` | 兑换码原始面值单位，如 `YYC`、`USD`、`CNY` |
| `redeemed_at` | 兑换成功时间（Unix 秒） |

说明：

1. 不回显完整兑换码本身。
2. 当前不额外返回充值日志 ID。
3. 这些字段已经足够支撑用户前端刷新余额、展示充值结果和后续排障。

## 5. 为什么不再保留裸整数

因为当前不存在“新后端 + 旧前端”同时在线的问题，所以继续保留：

```json
{
  "data": 100000
}
```

反而会带来三个问题：

1. 无法表达充值前后余额。
2. 无法表达兑换码名称和兑换时间。
3. 后续如果再补字段，还得继续做额外兼容分支。

在统一升级前提下，直接改成对象更干净。

## 6. 为什么新增 `topup/orders`

当前已经新增：

- `POST /api/v1/public/user/topup/orders`
- `POST /api/v1/public/topup/callback`

原因是外部充值跳转和兑换码入账已经是两条不同业务线，继续混在前端里拼 `transaction_id` 只会让链路越来越乱。

当前这个新路径只负责三件事：

1. 生成内部订单 ID
2. 生成 `transaction_id`
3. 返回最终外部跳转地址 `redirect_url`

它不直接完成支付，也不直接给用户入账。

回调路径的职责则是：

1. 推进订单状态，例如 `pending / paid / fulfilled / failed / canceled`
2. 回填上游订单号、供应商标识、状态说明
3. 在已知内部兑换码时，把订单和兑换码绑定起来

回调令牌优先读取 `system_settings.TopUpCallbackToken`，为空时回退到环境变量 `TOPUP_CALLBACK_TOKEN`。

## 7. 当前两条链路的边界

现在已经明确拆成两条路径：

1. `POST /api/v1/public/user/topup/orders`
   - 外部充值跳转订单
   - 当前只负责生成订单骨架和跳转地址
2. `GET /api/v1/public/user/topup/orders`
   - 当前用户查看自己的外部充值订单列表
   - 用于和兑换入账记录分开展示
3. `POST /api/v1/public/topup/callback`
   - 外部系统推进订单状态
   - 可选回填内部兑换码关联关系
4. `POST /api/v1/public/user/topup`
   - 兑换码核销入账
   - 直接把 `YYC` 写入用户余额
   - 如果兑换码已绑定充值订单，则自动把订单推进到 `fulfilled`

后续如果要补支付回调、订单状态推进、兑换码发放，应继续沿 `topup/orders` 这条线扩展，而不是再往 `topup` 里塞。

## 8. 后续演进建议

### P1

先保持当前结构化返回稳定，确保：

1. 前端充值页完全适配
2. Swagger / OpenAPI 与实现一致
3. 日志和错误提示语义一致

### P2

如果未来充值链路复杂化，继续扩展：

- 充值单状态流转
- 支付回调
- 充值单记录接口
- 外部支付成功后的兑换码发放链路

### P3

如果未来存在第三方客户端稳定接入，再评估是否需要：

1. API 版本化
2. 明确的兼容期
3. 旧路径冻结策略

## 9. 最终结论

在“前后端一起升级”的前提下，当前最佳方案是：

1. 升级 `POST /api/v1/public/user/topup` 的 `data` 为结构化对象
2. 增加 `POST /api/v1/public/user/topup/orders`，把外部充值跳转订单收回后端生成
3. 增加 `POST /api/v1/public/topup/callback`，把订单状态推进收回 Router 统一处理
4. 同步更新前端、Swagger、接口文档
5. 为订单与兑换码关联、支付回调和状态流转预留数据库记录

这就是当前仓库已经采用的路线。
