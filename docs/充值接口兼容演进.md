# 充值接口兼容演进方案

## 1. 当前前提

当前项目的升级前提是：

1. 前端和后端会一起升级。
2. 不考虑前后端版本错配。
3. 使用同一个数据库，并允许同步演进数据库结构。

在这个前提下，`POST /api/v1/public/user/topup` 不需要为了“旧前端兼容”保留裸整数返回。

## 2. 当前结论

当前采用的方案是：

1. 保持兑换码入账路径不变：`POST /api/v1/public/user/topup`
2. 直接把 `topup.data` 从整数升级为结构化对象
3. 新增 `POST /api/v1/public/user/topup/orders`，由后端创建外部充值跳转订单
4. 新增 `POST /api/v1/public/topup/callback`，由外部系统推进订单状态
5. 前端不再自行拼接 `transaction_id`
6. 前端充值页、Swagger、接口文档同步升级

这是当前仓库下最直接、成本最低、也最不容易继续做乱的方案。

## 3. 当前接口形状

请求体：

```json
{
  "code": "redeem-xxxx-xxxx"
}
```

兼容请求别名：

- `key`

返回体：

```json
{
  "success": true,
  "message": "",
  "data": {
    "redeemed_yyc": 100000,
    "before_yyc_balance": 500000,
    "after_yyc_balance": 600000,
    "redemption_id": "474c9b7b534047ce9205220990f53087",
    "redemption_name": "新用户补偿",
    "group_id": "474c9b7b534047ce9205220990f53087",
    "group_name": "enterprise",
    "face_value_amount": 10,
    "face_value_unit": "USD",
    "redeemed_at": 1774000000
  }
}
```

## 4. 字段语义

| 字段 | 语义 |
| --- | --- |
| `redeemed_yyc` | 本次兑换成功写入账户的 `YYC` |
| `before_yyc_balance` | 兑换前用户账户总 `YYC` 余额 |
| `after_yyc_balance` | 兑换后用户账户总 `YYC` 余额 |
| `redemption_id` | 兑换码记录 ID |
| `redemption_name` | 兑换码名称或批次名 |
| `group_id` | 兑换码绑定的分组 ID |
| `group_name` | 兑换码绑定的分组名称 |
| `face_value_amount` | 兑换码原始面值 |
| `face_value_unit` | 兑换码原始面值单位，如 `YYC`、`USD`、`CNY` |
| `redeemed_at` | 兑换成功时间（Unix 秒） |

说明：

1. 不回显完整兑换码本身。
2. 当前不额外返回充值日志 ID。
3. 这些字段已经足够支撑用户前端刷新余额、展示充值结果和后续排障。

## 5. 为什么不再保留裸整数

因为当前不存在“新后端 + 旧前端”同时在线的问题，所以继续保留：

```json
{
  "data": 100000
}
```

反而会带来三个问题：

1. 无法表达充值前后余额。
2. 无法表达兑换码名称和兑换时间。
3. 后续如果再补字段，还得继续做额外兼容分支。

在统一升级前提下，直接改成对象更干净。

## 6. 为什么新增 `topup/orders`

当前已经新增：

- `POST /api/v1/public/user/topup/orders`
- `POST /api/v1/public/topup/callback`
- `POST /api/v1/public/user/topup/orders/{id}/refresh`

原因是外部充值跳转和兑换码入账已经是两条不同业务线，继续混在前端里拼 `transaction_id` 只会让链路越来越乱。

当前这个新路径只负责三件事：

1. 生成内部订单 ID
2. 生成 `transaction_id`
3. 返回最终外部跳转地址 `redirect_url`

它不直接完成支付，也不直接给用户入账。

当前 `POST /api/v1/public/user/topup/orders/{id}/refresh` 的职责是：

1. 主动向外部支付平台查询该订单当前支付状态
2. 当外部平台已支付但业务回调失败或延迟时，补推进 Router 本地订单状态
3. 在状态确认已支付后，幂等执行本地履约

当前 `POST /api/v1/public/user/topup/orders` 还增加了一条约束：

1. 同一用户对同一套餐、或同一固定充值档位重复发起支付时，优先复用已有未完成订单
2. 如果匹配到的旧订单其实已经支付成功，则先补齐本地状态，再直接返回该订单
3. 避免用户因为重复点击而产生多条“已创建”订单

另外当前服务端还增加了轻量补偿 worker：

1. 定时扫描 `top_up_api` 来源、状态仍为 `created / pending / paid` 的订单
2. 主动调用外部支付状态查询接口
3. 如果发现外部已支付，则自动补齐 Router 本地状态与履约
4. 因此“支付成功但用户没有手动刷新记录页”的订单，也会在后台被逐步修正

当前 `POST /api/v1/public/user/topup/orders` 已支持两种业务：

1. `balance_topup`
   - 前端上传人民币金额 `amount`
   - Router 后端按 `CNY -> YYC` 汇率折算应到账额度
   - 再把订单跳转到 tidukongjian 收银台
2. `package_purchase`
   - 前端只上传 `package_id`
   - Router 后端读取套餐配置中的 `sale_price` / `sale_currency`
   - 不信任前端传入金额
   - 若套餐未配置售价则直接失败

回调路径的职责则是：

1. 推进订单状态，例如 `pending / paid / fulfilled / failed / canceled`
2. 回填上游订单号、供应商标识、状态说明
3. 接受 `redemption_id / redemption_code` 兼容参数，但不再新增订单与兑换码互引写入（去耦阶段）

回调令牌优先读取 `system_settings.TopUpCallbackToken`，为空时回退到环境变量 `TOPUP_CALLBACK_TOKEN`。

## 7. 当前两条链路的边界

现在已经明确拆成两条路径：

1. `POST /api/v1/public/user/topup/orders`
   - 外部充值跳转订单
   - 当前只负责生成订单骨架和跳转地址
2. `GET /api/v1/public/user/topup/orders`
   - 当前用户查看自己的外部充值订单列表
   - 用于和兑换入账记录分开展示
3. `POST /api/v1/public/user/topup/orders/{id}/refresh`
   - 当前用户主动刷新某一笔订单的支付状态
   - 用于“继续支付前确认状态”以及“支付成功但回调失败时补单”
4. `POST /api/v1/public/topup/callback`
   - 外部系统推进订单状态
   - `redemption_id / redemption_code` 仅兼容接收，不再写入订单-兑换互引字段
5. `POST /api/v1/public/user/topup`
   - 兑换码核销入账
   - 直接把 `YYC` 写入用户余额
   - 对“历史已绑定充值订单”的兑换码，仍会兼容推进订单到 `fulfilled`

这意味着：

1. 余额充值走“金额驱动”
2. 套餐购买走“商品驱动”
3. 两者复用同一条外部支付订单链路，但商品定价权仍留在 Router 后端
4. 支付回调不是唯一真相源，订单状态可以通过主动查单补齐
5. 下单接口本身会优先复用同商品的未完成订单，减少重复订单噪音

### 7.3 兑换有效期语义（2026-04-13）

兑换链路新增两层有效期：

1. 兑换码有效期：`code_validity_days`（0 表示永久），到期后不可再兑换。
2. 到账额度有效期：`credit_validity_days`（0 表示永久），兑换成功后写入余额批次过期时间，过期自动失效并扣减余额。

后续如果要补支付回调、订单状态推进、兑换码发放，应继续沿 `topup/orders` 这条线扩展，而不是再往 `topup` 里塞。

### 7.1 管理台展示隔离（2026-04-13）

为避免“充值”和“兑换”概念混淆，管理台按业务页面彻底隔离：

1. 业务流水-充值：只展示充值订单语义（用户、状态、来源、额度、订单信息、时间）
2. 业务流水-兑换：只展示兑换入账语义（用户、兑换码、分组、面值、入账时间）
3. 充值列表不再展示“关联兑换”列，兑换列表也不承载充值字段
4. 查询接口不再透出互引字段：充值订单查询不返回 `redemption_id`，兑换记录查询不返回 `topup_order_id`

### 7.2 数据层现状与后续方向

当前为兼容既有回调/幂等逻辑，`topup_orders` 与 `redemptions` 仍保留历史关联字段。  
但后续目标是把追踪关系收敛到统一账本（如余额批次/交易流水），逐步淡出业务表互相持有外键的模式。

当前已新增账本排障接口：

1. `GET /api/v1/public/user/topup/balance/transactions`
2. `GET /api/v1/admin/user/:id/topup/balance/transactions`

接口按 `source_type` 与 `tx_type=credit|consume|expire` 过滤，返回余额批次级交易明细，用于替代依赖 `topup_orders.redemption_id` 与 `redemptions.topup_order_id` 的历史排障方式。

## 8. 后续演进建议

### P1

先保持当前结构化返回稳定，确保：

1. 前端充值页完全适配
2. Swagger / OpenAPI 与实现一致
3. 日志和错误提示语义一致

### P2

如果未来充值链路复杂化，继续扩展：

- 充值单状态流转
- 支付回调
- 充值单记录接口
- 外部支付成功后的兑换码发放链路

### P3

如果未来存在第三方客户端稳定接入，再评估是否需要：

1. API 版本化
2. 明确的兼容期
3. 旧路径冻结策略

## 9. 最终结论

在“前后端一起升级”的前提下，当前最佳方案是：

1. 升级 `POST /api/v1/public/user/topup` 的 `data` 为结构化对象
2. 增加 `POST /api/v1/public/user/topup/orders`，把外部充值跳转订单收回后端生成
3. 增加 `POST /api/v1/public/topup/callback`，把订单状态推进收回 Router 统一处理
4. 同步更新前端、Swagger、接口文档
5. 为订单与兑换码关联、支付回调和状态流转预留数据库记录

这就是当前仓库已经采用的路线。
