# 用户 YYC 模型

> 本文档只描述“用户额度约束”这一层。  
> 通用的 `YYC` / 法币 / 兼容字段语义，统一以 [夜莺币（YYC）账本与金额语义基线](./额度与金额语义基线.md) 为准。

## 1. 业务定义

用户侧额度不是单一余额，而是三层约束同时生效：

1. 用户总余额：用户账户里实际可扣减的 `YYC`。
2. 用户每日额度：用户在当天可直接使用的基础 `YYC`。
3. 用户每月应急额度：当日额度不够时，可自动启用的月度弹性 `YYC`。

对应的技术字段如下：

| 业务概念 | 技术字段 | 语义 |
| --- | --- | --- |
| 用户总余额 | `users.quota` | 用户账户当前总 `YYC` 余额 |
| 用户每日额度 | `users.daily_quota_limit` | 用户当天基础 `YYC` 上限 |
| 用户每月应急额度 | `users.monthly_emergency_quota_limit` | 用户在某些天超出日额度后可继续使用的月度应急 `YYC` |
| 额度重置时区 | `users.quota_reset_timezone` | 日额度和月应急额度的周期边界时区 |

规则：

1. 真实扣费最终仍然落到账户总余额 `users.quota`。
2. 每日额度和每月应急额度负责“准入控制”和“额度归因”，不替代总余额账本。
3. 文档和页面统一称为 `YYC`；只有在说明技术实现时才出现 `quota` 字段名。

## 2. 约束顺序

单次请求进入系统后，按下面的顺序判断是否允许继续：

1. 先检查分组套餐额度。
2. 再检查用户每日额度和每月应急额度。
3. 最后检查用户总余额和令牌余额。

这意味着：

- 某个用户即使总余额还够，只要当天套餐额度或用户日额度已经耗尽，也会被拦截。
- 某个用户即使日额度和应急额度还够，如果总余额不够，也仍然不能继续消费。

## 3. 结算顺序

请求成功后，系统按下面的顺序归因本次消耗：

1. 优先记入用户当日基础额度。
2. 基础额度不够时，超出部分记入当月应急额度。
3. 无论归因落到哪一层，最终都会同时扣减用户总余额 `users.quota`。

一句话：

- 用户总余额是“主账本”。
- 日额度和应急额度是“限额池 + 归因池”。

## 4. 周期规则

- 用户日额度按 `quota_reset_timezone` 的自然日重置。
- 用户月应急额度按 `quota_reset_timezone` 的自然月重置。
- `quota_reset_timezone` 为空时，默认使用 `Asia/Shanghai`。

当前分组套餐额度的计数器已经抽象为可扩展周期模型：

- 当前实际使用 `daily`
- 后续可扩展 `weekly`、`monthly` 等 `counter_type`

## 5. 数据模型

### 5.1 users

用户主记录负责保存额度策略基线：

- `quota`
- `daily_quota_limit`
- `monthly_emergency_quota_limit`
- `quota_reset_timezone`

其中：

- `quota` 是主账本余额。
- 其余字段是限额策略，不是主账本余额本身。

### 5.2 user_quota_counters

按“用户 + 周期类型 + 周期键”累计用户侧限额使用情况。

主键：

- `user_id + counter_type + period_key`

核心字段：

- `counter_type`
  - `daily`
  - `monthly_emergency`
- `period_key`
  - `daily` 使用 `YYYY-MM-DD`
  - `monthly_emergency` 使用 `YYYY-MM`
- `reserved_quota`
- `consumed_quota`

语义：

- `reserved_quota`：已预占但尚未最终结算的 `YYC`
- `consumed_quota`：已确认结算的 `YYC`

### 5.3 group_quota_counters

按“分组 + 用户 + 周期类型 + 周期键”累计套餐额度使用情况。

主键：

- `group_id + user_id + counter_type + period_key`

核心字段：

- `counter_type`
  - 当前为 `daily`
  - 后续可扩展 `weekly` 等
- `period_key`
  - `daily` 使用 `YYYY-MM-DD`
- `reserved_quota`
- `consumed_quota`

语义：

- 这是“某个用户在某个分组套餐下”的额度计数器。
- 它不等价于用户总余额，也不等价于用户自己的日额度池。

## 6. 预占与结算

为了避免并发请求把额度打穿，用户额度采用“预占 -> 成功结算 / 失败释放”模型。

流程：

1. 请求进入时，先按预估消耗预占 `YYC`。
2. 请求失败时，释放本次预占。
3. 请求成功后，按真实消耗重新结算到基础额度 / 应急额度。

说明：

- 预估 `YYC` 和真实消耗可能不一致。
- 当真实消耗高于预估时，系统会在结算阶段继续补记。
- 如果补记后额度池不足，本次真实消耗仍会被记录，后续请求再被拦截。

## 7. 对外接口

### 7.1 用户自己查看

- `GET /api/v1/public/user/quota/daily`
- `GET /api/v1/public/user/quota/summary`

语义：

- `GET /api/v1/public/user/quota/daily`
  - 返回“当前分组套餐”的今日额度快照
- `GET /api/v1/public/user/quota/summary`
  - 返回“用户自己”的额度汇总
  - 包括：`daily`、`monthly_emergency`

快照对象统一同时返回：

- 兼容字段：`limit`、`consumed_quota`、`reserved_quota`、`remaining_quota`
- 显式字段：`yyc_limit`、`yyc_consumed`、`yyc_reserved`、`yyc_remaining`

### 7.2 管理员查看某个用户

- `GET /api/v1/admin/user/:id/quota/summary`
- `GET /api/v1/admin/group/:id/quota/daily?user_id=...`

语义：

- `GET /api/v1/admin/user/:id/quota/summary`
  - 查看指定用户自己的额度汇总
- `GET /api/v1/admin/group/:id/quota/daily?user_id=...`
  - 查看指定用户在某个分组下的今日套餐额度快照

上述快照对象与用户侧一致，也统一同时返回：

- `yyc_limit`
- `yyc_consumed`
- `yyc_reserved`
- `yyc_remaining`

## 8. 日志归因

消费日志会额外记录两类用户额度归因：

- `user_daily_quota`
- `user_emergency_quota`

它们的业务语义分别是：

- 本次请求命中了多少用户基础额度 `YYC`
- 本次请求命中了多少用户月应急额度 `YYC`

对应的显式别名为：

- `yyc_user_daily`
- `yyc_user_emergency`

这些字段用于解释一次消费是怎么分摊到“用户基础额度”和“用户应急额度”的，不表示额外再扣了一次钱。

## 9. 一句话总结

用户侧额度模型可以概括为：

- 总余额负责真实扣费
- 分组套餐负责套餐级准入
- 用户日额度负责日常基础限制
- 用户月应急额度负责少量超额缓冲
- 所有额度统计最终统一折算并记录为 `YYC`
