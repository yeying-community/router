# 用户额度模型

## 目标

用户额度由三层共同约束：

1. 用户总余额 `users.quota`
2. 用户每日额度上限 `users.daily_quota_limit`
3. 用户每月应急额度 `users.monthly_emergency_quota_limit`

其中：

- `users.quota` 是总余额，真实扣费仍然从这里结算。
- `daily_quota_limit` 是用户当天可直接使用的基础额度。
- `monthly_emergency_quota_limit` 是用户在某些天超过每日额度时可自动启用的月度弹性额度池。

## 结算顺序

请求到来时，按如下顺序判断：

1. 先检查分组套餐日额度。
2. 再检查用户日额度与月应急额度。
3. 最后检查用户总余额与令牌额度。

消费结算时，按如下顺序落账：

1. 优先消耗用户当日基础额度。
2. 当基础额度不足时，超出部分自动消耗当月应急额度。
3. 真实费用仍然统一扣减 `users.quota`。

## 重置规则

- 用户日额度按 `quota_reset_timezone` 的自然日重置。
- 用户月应急额度按 `quota_reset_timezone` 的自然月重置。
- `quota_reset_timezone` 为空时默认 `Asia/Shanghai`。

## 数据表

### users

新增字段：

- `daily_quota_limit`
- `monthly_emergency_quota_limit`
- `quota_reset_timezone`

### user_daily_quota_counters

按用户 + 业务日累计：

- `reserved_quota`
- `consumed_quota`

### user_monthly_emergency_quota_counters

按用户 + 业务月累计：

- `reserved_quota`
- `consumed_quota`

## 预占与结算

为避免并发请求穿透额度，用户额度采用“预占 -> 结算/释放”模式：

- 请求进入时先预占预计额度。
- 请求失败时释放预占额度。
- 请求成功后按真实消耗重新分配到“基础额度/应急额度”。

说明：

- 预估额度和真实消耗可能不完全一致。
- 当真实消耗高于预估时，系统会在结算阶段优先补记到应急额度；若应急额度也不足，则仍会记录真实消耗，后续请求会被继续拦截。

## 接口

### 用户自己查看

- `GET /api/v1/public/user/quota/summary`

返回：

- `daily`: 今日基础额度快照
- `monthly_emergency`: 本月应急额度快照

### 管理员查看某个用户

- `GET /api/v1/admin/user/:id/quota/summary`

## 日志

消费日志新增两个字段：

- `user_daily_quota`
- `user_emergency_quota`

用于表示本次请求分别消耗了多少“用户基础额度”和“用户应急额度”。
