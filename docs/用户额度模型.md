# 用户 YYC 模型

> 本文档只描述“用户额度约束”这一层。  
> 通用的 `YYC` / 法币 / 兼容字段语义，统一以 [夜莺币（YYC）账本与金额语义基线](./额度与金额语义基线.md) 为准。

## 1. 业务定义

用户侧不再把“套餐额度”和“用户余额”视为同一层含义。

当前实现里，请求会先判定本次消费来源，再进入对应的扣减链路：

1. 套餐额度：用户在某个分组下有生效套餐时，优先消耗套餐额度。
2. 用户总余额：当没有生效套餐，或者套餐额度已用尽时，回落到用户账户余额 `YYC`。
3. 用户每日额度 / 套餐应急额度：当前仅保留为配置字段与快照接口，不参与 relay 请求准入、预占和结算。

对应的技术字段如下：

| 业务概念 | 技术字段 | 语义 |
| --- | --- | --- |
| 套餐额度 | `group_quota_counters` + `user_package_subscriptions` | 当前用户在当前分组下的套餐额度计数器，当前已落地 `daily` |
| 用户总余额 | `users.quota` | 用户账户当前总 `YYC` 余额 |
| 用户每日额度 | `users.daily_quota_limit` | 保留的用户侧 `YYC` 限额配置/快照字段，当前不参与 relay 准入 |
| 套餐应急额度 | `package_emergency_quota_limit` | 保留的用户侧 `YYC` 限额配置/快照字段，当前不参与 relay 准入 |
| 额度重置时区 | `users.quota_reset_timezone` | 保留给用户侧限额快照使用的周期边界时区 |

规则：

1. 先判定本次请求走“套餐模式”还是“余额模式”。
2. 套餐模式命中时，只扣套餐额度，不扣用户总余额 `users.quota`。
3. 余额模式命中时，真实扣费最终落到账户总余额 `users.quota`。
4. 用户每日额度和套餐应急额度当前不参与请求准入控制和额度归因，不替代总余额账本。
5. 文档和页面统一称为 `YYC`；只有在说明技术实现时才出现 `quota` 字段名。

## 2. 约束顺序

单次请求进入系统后，按下面的顺序判断是否允许继续：

1. 先判断当前分组下是否存在生效套餐。
2. 如果存在生效套餐，优先尝试预占套餐额度。
3. 如果套餐额度足够，则本次请求进入“套餐模式”。
4. 如果套餐额度不足，则自动回落到“余额模式”。
5. 余额模式下，再检查用户总余额和令牌额度。

这意味着：

- 有生效套餐且套餐额度够用时，不会再因为用户总余额不足而被拦截。
- 套餐额度耗尽但用户仍有余额时，请求会自动切到余额模式继续执行。
- 没有生效套餐时，请求直接走余额模式。

## 3. 结算顺序

请求成功后，系统按“本次请求实际命中的模式”结算：

1. 套餐模式：
   - 结算分组套餐额度
   - 若存在令牌额度限制，则同步扣减令牌额度
   - 不扣用户总余额 `users.quota`
2. 余额模式：
   - 扣减用户总余额 `users.quota`
   - 若存在令牌额度限制，则同步扣减令牌额度

一句话：

- 套餐模式结算到套餐额度。
- 余额模式结算到用户余额。
- 用户日额度和套餐应急额度当前不参与实时扣减。

## 4. 周期规则

- 用户日额度快照按 `quota_reset_timezone` 的自然日边界计算。
- 套餐应急额度快照当前按 `quota_reset_timezone` 的自然月边界计算。
- `quota_reset_timezone` 为空时，默认使用 `Asia/Shanghai`。

当前分组套餐额度的计数器已经抽象为可扩展周期模型：

- 当前实际使用 `daily`
- 后续可扩展 `weekly`、`monthly` 等 `counter_type`

## 5. 数据模型

### 5.1 users

用户主记录负责保存额度策略基线：

- `quota`
- `daily_quota_limit`
- `package_emergency_quota_limit`
- `quota_reset_timezone`

其中：

- `quota` 是主账本余额。
- 其余字段是限额策略，不是主账本余额本身。

### 5.2 user_quota_counters

按“用户 + 周期类型 + 周期键”保留用户侧限额快照计数。

主键：

- `user_id + counter_type + period_key`

核心字段：

- `counter_type`
  - `daily`
  - `package_emergency`
  - `period_key`
  - `daily` 使用 `YYYY-MM-DD`
  - `package_emergency` 使用 `YYYY-MM`
- `reserved_quota`
- `consumed_quota`

语义：

- `reserved_quota`：历史兼容字段
- `consumed_quota`：历史兼容字段
- 该计数器当前保留为历史/兼容数据与快照来源，不再参与 relay 实时生效。

### 5.3 group_quota_counters

按“分组 + 用户 + 周期类型 + 周期键”累计套餐额度使用情况。

主键：

- `group_id + user_id + counter_type + period_key`

核心字段：

- `counter_type`
  - 当前为 `daily`
  - 后续可扩展 `weekly` 等
- `period_key`
  - `daily` 使用 `YYYY-MM-DD`
- `reserved_quota`
- `consumed_quota`

语义：

- 这是“某个用户在某个分组套餐下”的额度计数器。
- 它不等价于用户总余额，也不等价于用户自己的日额度池。
- 当请求命中套餐模式时，真实消耗会结算到这里，而不会同时扣用户余额。

## 6. 预占与结算

为了避免并发请求把额度打穿，系统统一采用“预占 -> 成功结算 / 失败释放”模型。

流程：

1. 请求进入时，先判定套餐模式或余额模式。
2. 套餐模式下，先预占套餐额度。
3. 余额模式下，不再预占用户日额度 / 应急额度，只预扣余额 / 令牌额度。
4. 请求失败时，释放本次预占，并归还预扣部分。
5. 请求成功后，按真实消耗结算到套餐额度或用户余额。

说明：

- 预估 `YYC` 和真实消耗可能不一致。
- 当真实消耗高于预估时，系统会在结算阶段继续补记。
- 如果补记后额度池不足，本次真实消耗仍会被记录，后续请求再被拦截。

## 7. 对外接口

### 7.1 用户自己查看

- `GET /api/v1/public/user/quota/daily`
- `GET /api/v1/public/user/quota/summary`

语义：

- `GET /api/v1/public/user/quota/daily`
  - 返回“当前分组套餐”的今日额度快照
- `GET /api/v1/public/user/quota/summary`
  - 返回“用户自己”的额度汇总
  - 包括：`daily`、`package_emergency`
  - `package_emergency` 表达的是“当前生效套餐带来的应急额度月快照”

快照对象统一同时返回：

- 兼容字段：`limit`、`consumed_quota`、`reserved_quota`、`remaining_quota`
- 显式字段：`yyc_limit`、`yyc_consumed`、`yyc_reserved`、`yyc_remaining`

### 7.2 管理员查看某个用户

- `GET /api/v1/admin/user/:id/quota/summary`
- `GET /api/v1/admin/group/:id/quota/daily?user_id=...`

语义：

- `GET /api/v1/admin/user/:id/quota/summary`
  - 查看指定用户自己的额度汇总
  - `package_emergency` 表达的是“当前生效套餐带来的应急额度月快照”
- `GET /api/v1/admin/group/:id/quota/daily?user_id=...`
  - 查看指定用户在某个分组下的今日套餐额度快照

上述快照对象与用户侧一致，也统一同时返回：

- `yyc_limit`
- `yyc_consumed`
- `yyc_reserved`
- `yyc_remaining`

## 8. 日志归因

消费日志仍然会记录总消耗 `Quota/YYCAmount`，并保留两类历史兼容字段：

- `billing_source`
- `user_daily_quota`
- `user_emergency_quota`

`billing_source` 的取值固定为：

- `package`
- `balance`

语义：

- `package`：本次消费由套餐额度承担
- `balance`：本次消费由用户余额承担

它们当前保留用于兼容展示；在现行实现下默认记为 `0`。

对应的显式别名为：

- `yyc_user_daily`
- `yyc_user_emergency`

这些字段当前不再参与一次消费的实际分摊计算，也不表示额外再扣了一次钱；它们只是兼容旧接口形状。

当请求命中套餐模式时：

- `billing_source = package`
- `Quota/YYCAmount` 仍然表示本次真实消耗
- `user_daily_quota` / `user_emergency_quota` 为 `0`
- 套餐额度的消耗体现在 `group_quota_counters`

当请求命中余额模式时：

- `billing_source = balance`
- `user_daily_quota` / `user_emergency_quota` 当前为 `0`

## 9. 一句话总结

用户侧额度模型可以概括为：

- 先判定本次请求走套餐还是余额
- 套餐优先，套餐够用时不扣余额
- 套餐不足时，自动回落到余额
- 用户日额度和套餐应急额度当前不约束 relay 请求
- 所有消费最终统一折算并记录为 `YYC`
