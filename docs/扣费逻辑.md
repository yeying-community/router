# 扣费逻辑

本文档是 Router 当前版本“额度使用 + 金额语义 + 请求期扣减”的统一说明。

边界：

- 支付下单、回调、订单状态、充值履约：见 [支付方案（定稿）](./支付方案.md)
- 模型单价解析与复杂计费条件：见 [模型计费](./模型计费.md)

## 1. 核心基线

1. 平台内部统一记账单位是 `夜莺币`，简称 `YYC`（Yeying Coin）。
2. 上游模型价格按官方币种维护（如 `USD`、`CNY`）。
3. 请求结算时，先算原币种金额，再折算为 `YYC` 入账。
4. 旧字段名 `quota` 系列继续保留兼容，但业务语义统一解释为 `YYC`。

## 2. 术语与字段语义

### 2.1 术语

| 术语 | 含义 |
| --- | --- |
| `YYC` | 平台内部统一记账单位 |
| `amount` | 面向展示或支付的法币金额 |
| `price` | 上游供应商单价 |
| `currency` | 价格/金额所属币种 |

### 2.2 兼容字段与显式字段（当前口径）

| 兼容字段 | 显式字段 | 统一语义 | 典型返回接口 |
| --- | --- | --- | --- |
| `quota` | `balance_amount` / `charge_amount` / `credit_amount` | `YYC` | `quota`、`balance_amount`（用户/列表）；`charge_amount`、`credit_amount`（按各业务接口） |
| `used_quota` | `used_amount` | 已消费 `YYC` | `used_quota`、`used_amount`（用户/令牌/日志等） |
| `remain_quota` | `remaining_amount` | 剩余 `YYC` | 令牌相关接口 |
| `daily_quota_limit` | 无 | 每日额度（`YYC`） | 套餐/订阅相关接口 |
| `package_emergency_quota_limit` | 无 | 应急额度（`YYC`） | 套餐/订阅相关接口 |
| `consumed_quota` | `consumed_amount` | 已结算 `YYC` | 额度快照接口 |
| `reserved_quota` | `reserved_amount` | 已预占 `YYC` | 额度快照接口 |
| `remaining_quota` | `remaining_amount` | 剩余可用 `YYC` | 额度快照接口 |
| `user_daily_quota` | `user_daily_charge_amount` | 本次由套餐日额度承担的 `YYC` | 消费日志接口 |
| `user_emergency_quota` | `user_emergency_charge_amount` | 本次由套餐应急额度承担的 `YYC` | 消费日志接口 |

规则：

1. 显式金额字段仅在已标注接口返回；不假设全量接口统一返回。
2. 兼容字段继续保留，且数值必须与显式字段一致。
3. `amount` 只表示法币，不表示 `YYC`。

### 2.3 金额快照字段

| 字段 | 语义 | 单位 |
| --- | --- | --- |
| `billing_input_amount` | 输入侧法币金额快照 | `billing_currency` |
| `billing_output_amount` | 输出侧法币金额快照 | `billing_currency` |
| `billing_amount` | 本次法币总金额快照 | `billing_currency` |
| `billing_charge_rate` | 计费时汇率快照 | `YYC / currency` |
| `billing_charge_amount` | 本次最终 `YYC` | `YYC` |
| `estimated_charge_amount` | 请求前预扣估算 `YYC` | `YYC` |
| `billing_charge_delta_amount` | 最终实结 `YYC` - 预扣估算 `YYC` | `YYC` |
| `estimated_prompt_tokens` | 请求前估算输入 token | token |
| `estimated_output_tokens` | 请求前估算输出计费数量 | token |
| `billing_prompt_token_delta` | 上游真实输入 token - 请求前估算输入 token | token |
| `billing_output_token_delta` | 上游真实输出 token - 请求前估算输出计费数量 | token |

## 3. 价格折算与记账

### 3.1 折算流程

```text
上游价格（price + currency + unit）
  -> 按请求用量计算法币金额
  -> 按币种汇率折算为 YYC
  -> 按 YYC 扣减余额 / 套餐额度 / 令牌额度
```

### 3.2 统一公式（按当前代码）

```text
raw_charge_amount = amount(currency) × charge_rate(currency) × group_channel_ratio
charge_amount = ceil(raw_charge_amount)
```

说明：

1. `amount(currency)` 由 `price_unit` 决定：
- `per_1k_tokens` / `per_1k_chars`：`amount = quantity × price / 1000`
- `per_image` / `per_video` / `per_second` / `per_minute` / `per_request` / `per_task`：`amount = quantity × price`
2. `group_channel_ratio` 来自本次命中的 `group + channel_id` 业务倍率。
3. `charge_amount` 最终按 `YYC` 整数向上取整。
4. 当本次有用量但价格换算后为 0（且非免费分组）时，最小记 1 `YYC`。
5. 当 `group_channel_ratio=0` 时，记账结果为 0 `YYC`。

## 4. 请求期扣减模型

### 4.1 两种扣费来源

| 模式 | 触发条件 | 扣减对象 | `billing_source` |
| --- | --- | --- | --- |
| 套餐模式 | 当前分组命中生效套餐，且套餐额度/次数/并发允许 | `group_quota_counters` + `user_quota_counters(package_emergency)`，或 `user_package_usage_counters` | `package` |
| 余额模式 | 无生效套餐，或套餐额度不足且允许回落 | `users.quota`（并同步余额批次交易） | `balance` |

说明：

- 内部存在 `package_fallback_balance` 场景；日志对外仍归因为 `balance`。
- YYC 套餐模式下若配置令牌限制，令牌额度会同步扣减。
- 请求次数套餐模式按 `user_id + group_id` 匹配生效订阅；可用供应商、模型和渠道由分组配置决定，套餐自身不再按供应商或模型匹配。
- 请求次数套餐命中后按请求数扣减，不扣令牌额度。
- 请求次数套餐的并发控制分两层：`max_concurrency_per_user` 限制“单个用户自己的这份订阅”可同时预占的请求数；`max_concurrency_per_package` 限制“这份套餐订阅整体”可同时预占的请求数。当前实现里订阅按用户开通，因此两者常常配置成相同值。

### 4.2 准入与预占

1. 先计算本次预估 `YYC`。
2. 先按 `user_id + group_id` 尝试请求次数套餐；命中后检查周期剩余次数与并发限制。
3. 请求次数套餐预占成功则走套餐模式；失败且 `allow_balance_fallback=true` 时回落余额，否则直接拒绝。
4. 未命中请求次数套餐时，判断当前分组是否存在 YYC 生效套餐。
5. 若有 YYC 套餐：尝试 `ReservePackageQuota`。
6. YYC 套餐预占成功则走套餐模式；额度不足时自动回落余额模式。
7. 余额模式下检查主余额与令牌额度。

### 4.3 结算与回滚

1. 请求成功：
- YYC 套餐模式先结算日额度，再结算套餐应急额度。
- 请求次数套餐模式结算 1 次请求，失败或中断时释放预占。
- 余额模式扣减 `users.quota` 并写余额交易。
2. 请求失败：
- 释放预占，并归还预扣余额/令牌。

### 4.4 预估与实扣差异（当前实现）

1. 准入阶段按“预估 `YYC`”预占（避免并发超扣）。
2. 结算阶段按真实 usage 重算 `YYC`，并对“预估 - 实际”做差额回补或补扣。
3. 余额模式在“主余额远大于预估值”（`user_quota > 100 × preConsumedQuota`）时，会跳过数据库层预扣，最终按实耗落账。
4. YYC 套餐模式只要预占成功，就始终先在套餐计数器结算；套餐不足才会在准入阶段回落到余额模式。
5. 请求次数套餐按请求维度计数，不随实际 token/图片/视频用量变化调整次数。

图片请求补充口径：

1. 文本、音频、视频等当前默认按“1 次请求 = 1 次扣减”。
2. 图片生成 / 图片编辑请求按图片张数扣减：
   - `n=1` 记 1 次
   - `n=4` 记 4 次
3. 图片请求在准入阶段会按 `n` 预占次数，成功后按相同数量结算；失败则释放预占。

## 5. 周期与计数器口径

1. 分组套餐日额度：`group_quota_counters(counter_type=daily)`，按自然日。
2. 套餐应急额度：`user_quota_counters(counter_type=package_emergency)`，按自然月。
3. 请求次数套餐：`user_package_usage_counters(metric=request_count)`，周期由套餐 `period_type` 决定，常用月周期。
4. 周期边界时区：`quota_reset_timezone`，空值回落 `Asia/Shanghai`。
5. 分组日额度策略来自“当前分组生效 YYC 套餐”；无生效套餐时该分组视图口径为 unlimited（`limit=0`）。

请求次数套餐补充口径：

1. `period_limit` 是单个周期的请求总次数，不随 token、图片张数、视频时长等实际 usage 再换算。
2. 一次请求在准入成功后会先占用 1 个“并发槽位”；请求结束后再结算这 1 次请求，并释放并发槽位。
3. 如果并发已满：
   - `allow_balance_fallback=true`：可转入余额模式。
   - `allow_balance_fallback=false`：直接拒绝。
4. 当前实现里套餐订阅按用户开通，因此：
   - `max_concurrency_per_user` 可理解为“这个用户最多同时跑多少个按次套餐请求”。
   - `max_concurrency_per_package` 可理解为“这份套餐实例最多同时跑多少个按次套餐请求”。
5. 如果暂时没有共享套餐实例诉求，建议将 `max_concurrency_per_user` 与 `max_concurrency_per_package` 配成相同值，降低理解和运营成本。

## 6. 余额批次口径

余额模式下，批次扣减采用 FEFO（先到期先用），不是 FIFO（最早充值先用）。

排序规则：

1. `expires_at=0`（永久有效）排最后。
2. 其余按 `expires_at asc`。
3. 同到期时间按 `created_at asc`，再 `id asc`。

## 7. 对外接口口径

### 7.1 快照接口

- `GET /api/v1/public/user/quota/daily`
- `GET /api/v1/public/user/quota/summary`
- `GET /api/v1/admin/user/:id/quota/summary`
- `GET /api/v1/admin/group/:id/quota/daily?user_id=...`

快照对象统一提供：

- `limit / limit_amount`
- `consumed_quota / consumed_amount`
- `reserved_quota / reserved_amount`
- `remaining_quota / remaining_amount`

### 7.2 已移除旧快照列

1. `users.daily_quota_limit`、`users.package_emergency_quota_limit` 已从用户表迁移移除。
2. 运行时额度策略统一来自“生效套餐 + 额度快照计数器”，不从用户表读取额度上限。

## 8. 页面与文案规范

1. 页面术语统一使用 `YYC`，不再用 `quota` 作为产品文案。
2. 若同时展示法币，采用双值展示：`1250 YYC（约 $0.0025）`。
3. 兑换比率建议使用“`USD -> YYC`”表达，不使用“单位美元额度”。

## 9. 当前边界（代码现状）

1. 日志 `billing_source` 当前只落两类：`package`、`balance`（不落 `package_fallback_balance`）。
2. 请求次数套餐准入基于 `ReserveRequestPackage`（周期次数 + 并发预占）；YYC 套餐准入基于 `ReservePackageQuota`（分组日额度 + 套餐应急额度）；余额准入基于主余额与令牌额度。
3. 计价与记账保留双快照：法币金额（`billing_*_amount`）与 `YYC`（`billing_charge_amount`）。
4. 采购成本底线已经接入预扣和结算，但默认策略 `official_markup = 1`、`target_margin = 0`、`risk_buffer = 0` 不改变没有采购成本配置时的现有扣费。
5. 当前 Router 采用同端点转发，端点协议由供应商模型、渠道端点配置和路由共同对齐；供应商 adaptor 只处理供应商特有差异，不作为跨协议重估入口。
6. 采购成本归因在消费日志写入后异步补全，成本未配置使用 `none`，不能解释为零成本。
