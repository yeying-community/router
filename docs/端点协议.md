# 端点协议

## 这篇文档只回答什么

这篇文档只回答三类问题：

1. Router 当前支持哪些端点协议。
2. 各端点协议在请求体、响应体、鉴权头、流式行为上的主要差异。
3. 渠道和供应商通常适合声明哪类端点协议。

这篇文档不展开回答：

1. 某个请求最终会选中哪个渠道。
2. 请求失败后是否会切换渠道、重试多少次、如何计费。
3. 渠道模型、渠道端点、分组模型的人工配置边界。

关联文档：

- 接口路径、字段和返回口径见 [接口文档.md](接口文档.md)
- 路由、选路、重试、转发过程见 [路由逻辑.md](路由逻辑.md)
- 渠道端点人工配置边界见 [渠道规范.md](渠道规范.md)

## 1. 结论

当前 Router 采用同端点转发规则。

当前规则很简单：

1. 下游调用什么端点协议，Router 就按该端点协议接收请求。
2. 渠道必须显式声明自己支持哪些端点。
3. 只有当下游端点和渠道已启用端点一致时，Router 才会把请求转发给该渠道。
4. 如果当前渠道没有这个端点能力，Router 直接报错。

因此，这篇文档重点是：

- Router 当前支持哪些端点协议
- 各端点协议之间有什么差异
- 供应商/渠道通常支持哪种端点协议
- 管理端配置时应如何选择

## 2. 什么是端点协议

这里的“端点协议”指的是：

- 下游请求进入 Router 时使用的 API 入口
- Router 转发到上游时使用的 API 入口
- 这套入口所要求的请求体、响应体、鉴权头、流式行为

在当前实现里，文本类最关键的三种端点协议是：

- `/v1/chat/completions`
- `/v1/responses`
- `/v1/messages`

图像、音频、嵌入等也各自有独立端点协议，例如：

- `/v1/images/generations`
- `/v1/images/edits`
- `/v1/audio/speech`
- `/v1/audio/transcriptions`
- `/v1/embeddings`

## 3. 当前 Router 的协议规则

### 3.1 同端点转发

对文本类请求，当前规则是：

- `/v1/chat/completions` 只转发到支持 `/v1/chat/completions` 的渠道
- `/v1/responses` 只转发到支持 `/v1/responses` 的渠道
- `/v1/messages` 只转发到支持 `/v1/messages` 的渠道

不会再做：

- `responses -> chat/completions`
- `messages -> chat/completions`
- `chat/completions -> responses`

### 3.2 端点能力来自配置

Router 是否允许转发，不靠运行时猜测，而是看渠道的端点能力配置。

因此：

- 供应商层提供“官方支持哪些端点”的参考信息
- 渠道层决定“这个具体渠道启用了哪些端点”
- 分组层只消费渠道已经启用的能力

### 3.3 协议和模型品牌不是一回事

不能因为模型品牌是 Claude，就默认它必须走 `/v1/messages`。

判断标准永远是：

- 上游官方文档怎么定义接口
- 这个渠道实际暴露的端点是什么
- 这个渠道当前启用了什么端点

## 4. 文本端点协议

### 4.1 `/v1/chat/completions`

这是 OpenAI 较早、也最常见的文本协议。

典型特征：

- 请求主字段是 `messages`
- 常见鉴权是 `Authorization: Bearer`
- 常见流式返回是 chat completion chunk

适用场景：

- OpenAI 兼容网关
- 各类自称“OpenAI compatible”的上游
- 大量历史模型和客户端 SDK

### 4.2 `/v1/responses`

这是 OpenAI 当前更完整的新文本协议。

典型特征：

- 请求主字段通常是 `input`
- 更适合统一承载文本、工具、多模态能力
- 图片生成工具、结构化输出等能力通常优先围绕这个协议扩展

适用场景：

- OpenAI 新协议
- 明确支持 `responses` 的 OpenAI 兼容网关
- 需要统一处理文本、工具、多模态输入的模型

### 4.3 `/v1/messages`

这是 Anthropic 风格文本协议。

典型特征：

- 请求主字段是 `messages`
- 常见鉴权是 `x-api-key`
- 需要 `anthropic-version`
- 上游文本主接口通常就是 `/v1/messages`

适用场景：

- Anthropic 官方接口
- 明确暴露 Anthropic 协议的 Claude 渠道

### 4.4 三者差异

| 端点协议 | 常见供应商风格 | 典型请求主字段 | 常见鉴权 | 当前 Router 转发规则 |
| --- | --- | --- | --- | --- |
| `/v1/chat/completions` | OpenAI 兼容旧协议 | `messages` | `Authorization: Bearer` | 只转发到 chat 渠道 |
| `/v1/responses` | OpenAI 新协议 | `input` | `Authorization: Bearer` | 只转发到 responses 渠道 |
| `/v1/messages` | Anthropic 协议 | `messages` | `x-api-key` | 只转发到 messages 渠道 |

## 5. 非文本端点协议

除了文本协议，当前 Router 还支持多类非文本端点协议。它们本身就是独立协议，不参与文本端点切换。

常见端点包括：

- `/v1/images/generations`
- `/v1/images/edits`
- `/v1/audio/speech`
- `/v1/audio/transcriptions`
- `/v1/audio/translations`
- `/v1/embeddings`
- `/v1/moderations`
- `/v1/realtime`
- `/v1/videos`

这些端点的共同原则是：

- 下游请求哪个端点，就要求渠道启用了哪个端点
- Router 只做该端点协议内的必要处理
- Router 不会用另一个端点协议替代当前请求端点

## 6. 供应商协议选择

在渠道新增/编辑页面里，`协议` 字段应理解为：

**上游 API 协议类型**

它控制的内容包括：

- 默认 Base URL
- 请求 URL 规则
- 鉴权头规则
- 模型列表同步方式
- 某些协议专属请求头

它不等于：

- 模型品牌
- 渠道名称
- 供应商名称

### 6.1 `openai`

应选择 `openai` 的典型情况：

- 上游要求 `Authorization: Bearer`
- 文本接口是 `/v1/chat/completions` 或 `/v1/responses`
- 模型列表通常走 `GET /v1/models`

即使这个上游卖的是 Claude、Gemini、Qwen，只要它暴露的是 OpenAI 兼容 API，协议仍然应选 `openai`。

### 6.2 `anthropic`

应选择 `anthropic` 的典型情况：

- 上游要求 `x-api-key`
- 文本主接口是 `/v1/messages`
- 请求头需要 `anthropic-version`

只有当上游真实暴露 Anthropic 协议时，才应选 `anthropic`。

## 8. 管理端配置建议

新增渠道时，建议按下面顺序判断协议：

1. 先看上游官方文档要求的鉴权头。
2. 再看上游主文本接口路径。
3. 再看模型列表接口如何访问。
4. 最后才看它主要卖哪些模型。

配置时应遵循：

1. 优先按上游接口规范选协议。
2. 不要按模型名称选协议。
3. 一个渠道如果暴露的是 OpenAI 兼容 API，就配置为 `openai`。
4. 只有真实暴露 Anthropic 协议时，才配置为 `anthropic`。

## 9. 端点能力配置建议

在渠道模型和渠道端点配置里，应把“端点协议”理解为能力开关，而不是自动推断结果。

因此：

1. 渠道支持哪个端点，就显式开启哪个端点。
2. 不支持的端点不要勉强勾选。
3. 下游要走 `/v1/responses`，就必须保证至少有一个渠道对该模型启用了 `/v1/responses`。
4. 下游要走 `/v1/messages`，就必须保证至少有一个渠道对该模型启用了 `/v1/messages`。

## 10. 文档范围

这篇文档只描述：

- 当前 Router 支持的端点协议
- 端点协议之间的差异
- 协议选择和端点能力配置规则

这篇文档不讨论：

- 文本协议之间的自动互转
- 跨端点自动转换
- 历史命名或旧实现口径
