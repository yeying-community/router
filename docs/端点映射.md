# 端点映射

## 1. 结论

在新增渠道时，`协议`不是展示字段，也不是“模型品牌”字段。

`协议`定义的是：Router 应该按哪一套上游 API 规则与该渠道通信。
a. 它决定请求发到哪里。
b. 它决定鉴权头怎么构造。
c. 它决定请求体、响应体怎么转换。
d. 它决定获取模型列表时如何请求上游。

因此，判断协议的标准不是“这个渠道主要卖什么模型”，而是“这个渠道真实遵循哪套 API 协议”。

## 2. 协议控制什么

| 控制项 | 说明 | 代码位置 |
| --- | --- | --- |
| 协议名归一化 | 例如 `openai-compatible` 最终归一为 `openai` | `internal/relay/channel/protocol.go` |
| 适配器选择 | `protocol -> channelProtocol -> apiType -> adaptor`，最终决定走哪套转发实现 | `internal/relay/channel/helper.go` `internal/relay/adaptor.go` |
| 默认 Base URL | 未填写地址时，按协议补默认上游地址 | `internal/relay/channel/protocol.go` `internal/relay/channel/url.go` |
| 请求 URL 规则 | 不同协议的上游路径不同，例如 `anthropic` 固定请求 `/v1/messages` | `internal/relay/adaptor/openai/adaptor.go` `internal/relay/adaptor/anthropic/adaptor.go` |
| 鉴权头规则 | 例如 `openai` 用 `Authorization: Bearer`，`anthropic` 用 `x-api-key` | `internal/relay/adaptor/openai/adaptor.go` `internal/relay/adaptor/anthropic/adaptor.go` |
| 请求/响应转换 | 决定请求是否要桥接、响应如何回写给下游 | `internal/relay/controller/text.go` `internal/relay/adaptor/openai/adaptor.go` `internal/relay/adaptor/anthropic/adaptor.go` |
| 获取模型方式 | 后台“获取支持模型”时，按协议构造请求头和模型列表请求 | `internal/admin/controller/channel/model_runtime.go` |
| 部分默认模型配置 | 渠道模型默认 endpoint、价格解析等会参考协议 | `internal/admin/model/channel_model.go` `internal/relay/controller/text.go` |

## 3. 协议不直接控制什么

| 不控制项 | 说明 |
| --- | --- |
| 渠道名称、渠道标识 | 这些是管理信息，不由协议决定 |
| 你填写的地址、密钥具体值 | 协议只决定如何使用它们，不决定具体填什么 |
| 这个渠道启用了哪些模型 | 由 `channel_models` 配置决定 |
| 模型归属哪个供应商 | 由模型配置决定，不由协议决定 |
| 哪些分组可以使用这个渠道 | 由分组与渠道能力绑定决定 |
| 渠道优先级、分组路由顺序 | 由分组/能力/优先级配置决定 |

## 4. 常用协议对照表

当前最常用、最容易混淆的是 `openai` 和 `anthropic`。

| 协议 | 适用判断 | 默认 Base URL | 典型鉴权 | 常见上游端点 | 获取模型 |
| --- | --- | --- | --- | --- | --- |
| `openai` | 上游声称自己是 OpenAI 兼容接口；请求通常走 Bearer；常见文本端点为 `/v1/chat/completions` 或 `/v1/responses` | `https://api.openai.com` | `Authorization: Bearer <key>` | `/v1/chat/completions` `/v1/responses` `/v1/images/generations` `/v1/audio/speech` | `GET /v1/models` + Bearer |
| `anthropic` | 上游要求 `x-api-key`，文本主接口是 `/v1/messages` | `https://api.anthropic.com` | `x-api-key: <key>` | `/v1/messages` | `GET /v1/models` + `x-api-key` + `anthropic-version` |

## 5. 配置判断规则

新增渠道时，建议按下面顺序判断协议：

1. 先看上游官方文档要求的鉴权头。
2. 再看上游主文本接口路径。
3. 再看模型列表接口如何访问。
4. 最后才看它主要提供哪些模型。

换句话说：

- 如果上游是 OpenAI 兼容接口，即使它卖的是 Claude，也应优先选择 `openai`。
- 如果上游明确要求 `x-api-key` 且文本接口是 `/v1/messages`，应选择 `anthropic`。
- “模型品牌是 Claude” 不等于 “协议一定是 Anthropic”。

## 6. 两个典型例子

### 6.1 应选 `openai`

渠道提供的模型可能包含：
- `gpt-5.1`
- `claude-sonnet-4-6`
- `gemini-2.5-pro`

但它的接入方式是：
- `Authorization: Bearer <key>`
- 文本接口走 `/v1/responses`
- 模型列表走 `GET /v1/models`

这种情况，本质上是 OpenAI 兼容网关，应选 `openai`。

### 6.2 应选 `anthropic`

渠道提供的模型是 Claude，且要求：
- `x-api-key: <key>`
- 文本接口走 `/v1/messages`
- 模型列表访问时需要 `anthropic-version`

这种情况，应选 `anthropic`。

## 7. 与当前 Router 实现的对应关系

### 7.1 协议归一化
- `openai-compatible` 会归一为 `openai`
- 未识别或为空时，默认回落 `openai`

参考：
- `internal/relay/channel/protocol.go`
- `internal/admin/model/channel.go`

### 7.2 请求转发
- Router 在请求进入时，把渠道协议写入 `meta.ChannelProtocol`
- 再由 `meta.ChannelProtocol` 推导 `meta.APIType`
- 最后选择对应 adaptor 发起上游请求

参考：
- `internal/relay/meta/relay_meta.go`
- `internal/relay/channel/helper.go`
- `internal/relay/adaptor.go`

### 7.3 模型同步
- 后台点击“获取支持模型”时，并不是统一按 Bearer 请求
- `anthropic` 会走专门的请求头逻辑

参考：
- `internal/admin/controller/channel/model_runtime.go`

## 8. 管理侧的实际建议

在渠道新增/编辑页面，`协议`字段应该理解为“上游 API 协议类型”，而不是“供应商类型”或“模型类型”。

因此，配置时应遵循：

1. 优先按上游接口规范选协议。
2. 不要按模型名称选协议。
3. 一个渠道如果同时暴露多种模型，但底层仍是一套 OpenAI 兼容 API，就仍然配置为 `openai`。
4. 只有当该渠道真实要求 Anthropic 风格接口时，才配置为 `anthropic`。

## 10. 目前可以落地的配置建议

为了避免新增渠道时选错协议，可以直接按下面规则执行：

| 判断问题 | 选择建议 |
| --- | --- |
| 上游要求 `Authorization: Bearer`，文本接口是 `/v1/chat/completions` 或 `/v1/responses` | 选 `openai` |
| 上游要求 `x-api-key`，文本接口是 `/v1/messages` | 选 `anthropic` |
| 上游虽然卖 Claude，但接口文档是 OpenAI 兼容格式 | 仍然选 `openai` |
| 上游是厂商专有协议，鉴权和路径都不是 OpenAI 兼容 | 选该厂商专用协议 |
| 不能确认时 | 先看官方接口文档，不要按模型名猜 |
