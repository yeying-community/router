# 渠道规范

本文档定义渠道层的职责、变更边界，以及与供应商层、分组层的关系。

## 摘要

本层拥有：

1. 渠道模型是否启用、停用、映射到哪个上游模型。
2. 渠道模型价格覆盖、供应商归属、模型类型等人工维护字段。
3. 渠道端点是否启用。

本层不拥有：

1. 供应商官方模型信息。
2. 供应商官方价格与官方端点范围。
3. 分组是否对外暴露某模型。

本层修改入口：

1. 管理员保存渠道模型与渠道端点。
2. 刷新、测试、上游返回只能写事实记录，不能改人工配置。

## 1. 本层职责

渠道层负责两件事：

1. 定义“该渠道接入哪些模型”。
2. 定义“这些模型的哪些端点当前真的允许参与运行”。

当前实现里：

1. 渠道模型保存“该渠道接入哪些模型”。
2. 渠道端点保存“这些模型的哪些端点当前启用”。
3. 渠道端点是当前端点启用真值层。

同时需要明确：

1. 渠道必须感知供应商层面的变化。
2. 渠道也必须感知最近一次上游返回结果的变化。
3. 但这些变化都不能静默覆盖人工配置。
4. 人工配置一旦保存成功，就持续有效，除非管理员再次主动修改。
5. 供应商信息和最近一次上游返回结果变化后，系统只负责记录和通知，不替管理员修改配置。

## 2. 本层拥有的配置

渠道层拥有的真值包括：

1. 渠道模型是否选中、是否停用。
2. 渠道模型对应的上游模型。
3. 渠道模型对应的供应商和模型类型。
4. 渠道价格覆盖
5. 渠道端点是否启用。

只要这些字段是管理员显式维护的，就不能被上层自动任务静默覆盖。

这条原则要再说清楚：

1. 人工配置是真值。
2. 真值只由管理员保存动作修改。
3. 供应商信息变化和最近一次上游返回结果变化，只能影响校验结果、状态提示和通知内容，不能直接改写人工配置。

## 3. 渠道模型变化

渠道自身支持的模型不是静态的，需要先明确模型变化来源。

### 3.1 渠道模型会怎么变化

渠道中的模型集合，至少会因为下面几类原因变化：

1. 管理员手动新增模型。
2. 管理员手动关闭模型。
3. 渠道刷新后，上游返回了新的模型集合。
4. 上游不再返回某个模型。
5. 供应商层补充了该模型的价格、端点范围、模型状态等官方信息。

### 3.2 哪些变化属于渠道自己感知到的变化

渠道层需要区分两类变化：

1. 上游真实变化
   - 例如渠道刷新后，实际从上游拿到了新的模型列表，或者原有模型不再返回。
2. 供应商信息变化
   - 例如供应商新增了模型、更新了价格、调整了端点范围、把模型状态改成 `deprecated`。

这两类变化都会影响渠道的“可选范围”和“解释信息”，但不应直接等同于“人工配置已变更”。

## 4. 当前实现现状

当前代码历史上存在两类问题：

1. 刷新会把上游拉取结果直接回写到渠道模型。
2. 端点快照会被整体重建，容易覆盖人工端点开关。

这两类行为都不符合本文档的目标语义，后续必须收口为：

1. 刷新和测试只记录事实。
2. 人工配置只由管理员保存动作修改。

## 5. 标准定义

### 5.1 总原则

1. 渠道中的模型和端点，以管理员保存的人工配置为真值。
2. 人工配置保存前，必须检查供应商官方信息和最近一次上游返回结果。
3. 人工配置保存成功后，该配置持续有效，除非管理员再次主动修改。
4. 供应商官方信息和最近一次上游返回结果，只能影响校验结果、状态提示和通知内容，不能静默覆盖人工配置。

### 5.2 配置前校验规则

管理员启用模型前，后端必须检查两类信息：

1. 供应商官方信息里是否存在该模型。
2. 最近一次上游返回里是否包含该模型。

管理员启用端点前，后端必须检查两类信息：

1. 供应商官方端点范围里是否包含该端点。
2. 最近一次端点测试结果里是否有明确的成功结果。

规则如下：

1. 供应商和上游都满足时，允许保存。
2. 供应商缺失时，不允许启用。
3. 最近一次上游返回未包含时，不允许启用。
4. 端点不在供应商官方端点范围内时，不允许启用。
5. 端点没有最近一次成功测试结果时，不允许启用。

### 5.3 保存后规则

人工配置保存成功后，规则应明确为：

1. 保存成功后，模型和端点即视为启用状态。
2. 启用状态持续有效，直到管理员再次主动修改。
3. 后续供应商变化或最近一次上游返回结果变化，不自动关闭、不自动删除。
4. 系统只记录变化并通知管理员。

### 5.4 禁用渠道规则

管理员手动禁用渠道前，后端必须先检查该渠道是否仍被分组使用。

规则如下：

1. 如果渠道未被任何分组使用，允许直接禁用。
2. 如果渠道仍被任一分组接入，阻止禁用。
3. 阻止禁用时，系统必须明确返回受影响分组，以及该渠道当前承载的分组模型。
4. 不允许出现“渠道已禁用，但分组仍保留悬空引用”的静默状态。
5. 如果后续需要支持联动处理，必须走显式的单独操作，不能把级联移除作为普通禁用的隐式副作用。

### 5.5 渠道如何感知供应商变化

渠道感知供应商变化，至少应包括以下几类：

1. 感知供应商新增模型。
2. 感知供应商模型状态变化，例如从 `active` 变成 `deprecated`。
3. 感知供应商价格变化。
4. 感知供应商端点范围变化。

感知后的处理规则必须明确：

1. 供应商新增模型时，渠道应把它展示为“可选模型”，而不是自动启用。
2. 供应商模型状态变成 `deprecated` 时，渠道应能看到该状态，并通知管理员，但不能自动删除该模型。
3. 供应商价格变化时，渠道可以刷新展示口径；如果渠道已配置价格覆盖，则不能自动覆盖。
4. 供应商端点范围变化时，渠道应更新“可选端点范围”，但不能直接覆盖管理员已启用或已禁用的端点。

### 5.6 渠道如何感知上游模型变化

渠道感知上游模型变化，是指渠道刷新后拿到的实际上游模型列表变化。

规则应明确为：

1. 上游新增模型时，渠道应记录为最近一次上游返回包含该模型，并通知管理员。
2. 上游不再返回某模型时，渠道应通知管理员。
3. 上游不再返回某模型，不等于立即删除该模型。
4. 上游不再返回某模型，也不等于立即关闭该模型。
5. 是否保留、关闭、删除该模型，应由管理员确认。

### 5.7 刷新规范

渠道刷新允许做的事：

1. 发现新的上游返回模型。
2. 发现新的官方端点范围变化。
3. 更新事实记录和辅助信息。
4. 感知上游模型集合变化。
5. 感知供应商价格、状态、端点范围变化。
6. 生成通知或提示信息。

渠道刷新不允许做的事：

1. 静默打开用户已关闭的模型。
2. 静默打开用户已关闭的端点。
3. 静默删除用户已启用或已禁用的端点记录。
4. 用整体重建覆盖人工状态来源。
5. 因上游不再返回某模型，就静默删除该模型。
6. 因供应商把模型改成 `deprecated`，就静默删除该模型。

### 5.8 migration 规范

渠道层相关 migration 可以：

1. 回填缺失字段。
2. 补齐新结构所需的初始字段值。
3. 为已存在的渠道模型补展示、校验、通知所需元数据。
4. 同步供应商层已确认的官方价格、官方端点范围、模型状态，用于渠道展示、校验和通知判断。

渠道层相关 migration 不可以：

1. 代替管理员做模型启停决策。
2. 代替管理员做端点启停决策。
3. 因供应商信息变化而删除现有渠道端点真值。
4. 因供应商模型状态变化而删除现有渠道模型。

### 5.9 自动禁用规范

渠道层允许存在系统自动禁用，但必须满足：

1. 自动禁用和人工禁用可区分。
2. 自动禁用原因可解释。
3. 管理员可恢复。
4. UI 与后端读取同一套状态定义。

换句话说，后续不能只有一个“是否启用”字段，却同时承载：

1. 人工关闭
2. 系统关闭
3. 范围内但未启用
4. 不再推荐

这些语义必须能被区分。

## 6. 与供应商层的关系

渠道层依赖供应商层提供：

1. 官方模型信息
2. 官方价格
3. 官方端点范围

首先要明确：

1. 渠道必须感知供应商层面的变化。
2. 供应商新增模型时，渠道应能看到这个模型进入可配置范围。
3. 供应商模型下线或状态变化时，渠道应能看到该变化。
4. 供应商价格或端点范围变化时，渠道应能看到该变化。

但渠道层必须坚持：

1. 官方范围或上游返回结果不等于已启用。
2. 供应商信息修正不等于渠道配置变更。
3. 供应商模型下线不等于渠道配置自动删除。
4. 感知供应商变化，不等于自动替管理员修改渠道模型和渠道端点开关。

## 7. 与分组层的关系

渠道层向分组层输出：

1. 哪些模型在该渠道上存在。
2. 哪些模型当前在该渠道上可用。
3. 哪些端点当前真实启用。

分组层消费这些真值来生成用户可见能力，但不得反向改写渠道层真值。

## 8. 通知与展示

渠道层后续必须实现以下通知和展示能力：

管理端展示也要明确，不能要求管理员靠推断操作。当前渠道管理端至少应直接展示：

1. 最近一次上游返回。
2. 最近一次测试结果。
3. 不能启用的原因。

### 8.1 管理端展示口径

渠道管理端必须把三类信息分开显示，不能混成一类：

1. 人工配置
   - 渠道模型是否启用。
   - 渠道端点是否启用。
   - 渠道价格覆盖、模型别名、访问地址等人工维护字段。
2. 当前对外范围
   - 当前哪些模型和端点真正参与对外请求。
   - 当前哪些端点真实可被下游请求命中。
3. 解释信息
   - 最近一次上游返回。
   - 最近一次测试结果。
   - 不能启用的原因。
   - 供应商官方状态、官方端点范围、官方价格变化提示。

界面规则必须明确：

1. 人工配置字段要可编辑。
2. 当前对外范围要可见，但不能伪装成可编辑人工配置。
3. 解释信息只读，不得保存回渠道人工配置。
4. 不能让管理员把“测试失败”误解成“已自动关闭”。
5. 不能让管理员把“最近一次上游未返回”误解成“配置已经删除”。

### 8.2 模型页展示要求

渠道模型页至少要直接展示：

1. 模型名称、别名、供应商、价格、类型。
2. 该模型当前是否已被渠道人工配置选中。
3. 最近一次上游返回状态。
4. 当前不能启用的原因。

如果模型当前不可启用，UI 应直接解释原因，而不是只把开关置灰。

### 8.3 端点页展示要求

渠道端点页至少要直接展示：

1. 模型 + 端点。
2. 当前端点是否已启用。
3. 最近一次测试结果。
4. 最近一次测试时间。
5. 最近一次测试错误。
6. 当前不能启用的原因。

端点开关语义必须保持：

1. 开关表示渠道人工配置里的端点启用状态。
2. 最近一次测试结果只是解释信息，不直接等价于开关状态。
3. 没有成功测试记录时，不允许新启用，但不代表已保存的启用状态会被静默取消。

1. 每条端点至少要能解释以下状态：
   - 范围内但未启用
   - 人工启用
   - 人工禁用
   - 系统禁用
   - 模型未选中
   - 最近一次上游返回未包含

2. 模型本身至少要能解释以下来源：

1. 上游当前返回
2. 上游当前不再返回
3. 供应商仍标记为可用
4. 供应商已标记为 `deprecated`
5. 管理员已启用
6. 管理员已停用

3. 系统至少要支持以下通知场景：

1. 上游不再返回已启用模型
2. 上游不再返回已启用端点
3. 供应商把已启用模型标记为 `deprecated`
4. 供应商调整已启用模型的端点范围
5. 已启用模型连续测试失败

## 9. 建议数据结构

如果后续按“语义最干净、扩展最稳”的方向演进，建议把渠道层拆成三类数据：

### 9.1 人工配置

人工配置只回答“管理员想怎么配”，属于真值。

渠道模型至少需要表达：

1. 渠道是否启用该模型。
2. 该模型对应的上游模型名。
3. 该模型归属的供应商。
4. 该模型的类型。
5. 该模型是否有价格覆盖。

端点配置至少需要表达：

1. 该端点是否被管理员启用。
2. 该端点是否被系统临时禁用。
3. 系统禁用原因是什么。

### 9.2 最近一次上游返回结果

最近一次上游返回结果只回答“这个渠道最近一次从上游实际返回了什么”。

模型返回结果至少需要表达：

1. 最近一次刷新时，上游返回里是否包含这个模型。
2. 最近一次同步时间。
3. 最近一次同步任务。
4. 最近一次同步错误。

端点测试结果至少需要表达：

1. 最近一次测试的是哪个端点。
2. 最近一次测试时间。
3. 最近一次测试是否成功。
4. 最近一次错误。

这里还需要明确两张测试相关表的职责：

1. `channel_tests`
   - 保存测试历史明细。
   - 用于测试页面、任务排查、测试产物下载。
   - 可以保留同一模型、同一端点的多轮测试结果。
2. `channel_model_endpoint_test_results`
   - 保存 `channel_id + model + endpoint` 维度的最近一次端点测试事实。
   - 用于端点页面展示最近一次测试状态。
   - 用于端点启用前校验“最近一次测试是否成功”。

两张表之间的关系必须固定为：

1. `channel_tests` 是历史明细表。
2. `channel_model_endpoint_test_results` 是最近一次端点测试事实表。
3. 测试任务完成后，必须先写历史明细，再同步刷新最近一次端点测试事实。
4. 新增最近一次端点测试事实表后，migration 必须把历史明细里的最新结果回填进去。
5. 端点页和端点启用校验只读取最近一次端点测试事实，不直接扫描历史明细。

## 10. 后端规则清单

### 10.1 保存前校验

模型保存前，后端必须检查：

1. 供应商官方信息里是否存在该模型。
2. 最近一次上游返回里是否包含该模型。

端点保存前，后端必须检查：

1. 供应商官方端点范围里是否包含该端点。
2. 最近一次端点测试结果里，是否存在该端点且测试成功。

拦截规则：

1. 供应商信息缺失，不允许启用。
2. 最近一次上游返回结果缺失，不允许启用。
3. 端点不在供应商官方端点范围内，不允许启用。

### 10.2 保存后处理

保存成功后，后端只做两件事：

1. 保留人工配置。
2. 记录后续供应商变化和最近一次上游返回结果变化，并通知管理员。

后端不做的事：

1. 不因上游缺失自动关闭模型。
2. 不因上游缺失自动关闭端点。
3. 不因供应商 `deprecated` 自动关闭模型。
4. 不因供应商端点范围变化自动删除端点配置。

### 10.3 刷新任务可写范围

渠道刷新任务允许写：

1. 最近一次上游返回结果。
2. 候选模型范围相关的事实信息。
3. 候选端点范围相关的事实信息。
4. 与最近一次上游返回结果直接相关的提示信息。

渠道刷新任务不允许写：

1. 人工启用状态。
2. 人工关闭状态。
3. 人工维护的模型选择结果。
4. 人工维护的端点启停结果。

端点测试任务允许写：

1. 端点测试结果。
2. 最近一次测试时间。
3. 最近一次测试错误。
4. 最近一次端点测试事实。

### 10.4 供应商变化可写范围

供应商变化同步到渠道时，允许写：

1. 供应商官方价格。
2. 供应商官方端点范围。
3. 供应商官方状态。
4. 渠道中的解释信息和通知内容。

供应商变化同步到渠道时，不允许写：

1. 人工模型启停。
2. 人工端点启停。
3. 已保存的渠道价格覆盖。

### 10.5 删除规则

渠道中的模型和端点删除，必须区分“人工删除”和“系统最近一次上游返回未包含”。

规则应定为：

1. 上游同步不到，不等于自动删除。
2. 供应商已 `deprecated`，不等于自动删除。
3. 删除必须由管理员主动执行，或者由后续单独定义的系统策略执行。

## 11. 稳健性要求

渠道层后续设计必须满足：

1. 用户手工开关不被 refresh 覆盖。
2. 用户手工开关不被 migration 覆盖。
3. 用户手工开关不被供应商信息修正覆盖。
4. 渠道能够感知供应商变化，也能够感知上游模型变化。
5. 感知变化后的自动处理范围必须明确，不能靠隐式覆盖实现。
6. 系统负责校验、记录和通知，不负责替管理员收缩已保存配置。
