# 套餐收敛方案

这篇文档定义 Router 当前采用的用户态套餐模型，以及后续演进时需要保持的边界。

相关文档：

- 套餐语义与并发说明见 [套餐说明.md](./套餐说明.md)
- 请求准入、YYC / 按次扣减见 [扣费逻辑.md](./扣费逻辑.md)
- 接口字段口径见 [接口文档.md](./接口文档.md)
- 支付与套餐购买链路见 [支付方案.md](./支付方案.md)

## 1. 核心结论

Router 的用户资产模型是：

```text
一个用户
= 多个不同槽位的生效套餐
+ 一份独立余额账本
```

其中：

- 用户只感知“套餐”和“余额”
- 用户可以同时拥有多个不同套餐槽位的生效套餐
- 同一套餐槽位内只允许一个生效套餐，同槽位购买立即替换当前生效套餐
- 续费不创建待生效计划，而是立即延长当前生效套餐到期时间
- 不同套餐槽位互不替换，购买时视为新增权益
- 余额始终独立存在，可被套餐策略用作兜底

套餐槽位由后台字段确定：

```text
group_id + package_type + quota_metric
```

`group_id` 仍是运行时边界，用于绑定模型、供应商、渠道、倍率和路由策略；用户侧不把它作为可选择的产品概念。

管理员新增或编辑套餐时不手工填写槽位。选择分组后，该分组的模型范围就是套餐适用范围；同一分组、同一套餐类型和同一扣减口径会被系统视为同一槽位。充值额度不是套餐槽位，只通过分组说明到账余额适用于哪些模型范围。

## 2. 为什么采用多套餐槽位

单一套餐模型会让不同供应商、不同额度形态和不同模型范围互相排斥，限制用户同时购买多类权益。

多套餐槽位模型的好处：

1. 不同模型分组可以并存，用户可以同时购买多类服务能力
2. 同一槽位内仍然保持清晰的续费、升级、降级边界
3. 运行时按模型和分组查找权益，不需要把所有套餐强行压成一个全局状态
4. 后台 `group` 能继续承担技术边界，前台只展示套餐资产

## 3. 用户接口口径

核心接口：

```text
GET /api/v1/public/user/package/subscription
GET /api/v1/admin/user/:id/package/subscription
```

返回字段：

- `has_active_packages`
- `active_packages`

不再返回：

- `current_package`
- `next_packages`
- `subscription`
- `subscriptions`

请求次数套餐对象内返回 `usage`，包含：

- `period_key`
- `limit_amount`
- `consumed_amount`
- `reserved_amount`
- `remaining_amount`
- `unlimited`

## 4. 写入与替换规则

### 4.1 新购

目标槽位无生效套餐时，购买目标套餐生成新的生效套餐。

其他槽位已有套餐不受影响。

### 4.2 续费

目标槽位已有同套餐生效时，续费立即延长当前生效套餐到期时间，不影响其他槽位。

### 4.3 升级

目标槽位已有不同套餐生效时，升级默认立即生效。

升级差价按同槽位生效套餐的剩余周期价值折算。

### 4.4 降级

降级立即生效，同槽位当前套餐会被目标套餐替换。

### 4.5 转换

当前实现不再把不同套餐类型天然视为“转换”。如果 `package_type` 或 `quota_metric` 不同，它们属于不同槽位；购买目标套餐按新增权益处理。

需要产品上支持“跨槽位转换”时，应另行定义迁移动作和权益清算规则，不能复用普通套餐购买推断。

## 5. 购买预览推断

`POST /api/v1/public/user/topup/package/preview` 的自动推断规则：

1. 目标槽位无生效套餐：`purchase`
2. 目标槽位生效套餐与目标套餐相同：`renew`
3. 目标槽位生效套餐与目标套餐不同：`upgrade`

显式传入 `downgrade` 时，预览按同槽位立即替换计算。

显式传入 `convert` 但目标槽位无生效套餐时，按 `purchase` 处理。

## 6. 前台展示规则

用户侧套餐状态页展示：

1. 生效套餐列表
2. 每个套餐的模型范围、额度、并发、周期、到期时间和请求次数用量

报价页展示：

1. 可购买套餐
2. 目标套餐相对同槽位生效套餐的动作标签
3. 新购、续费、升级等预览和应付金额

不展示用户可理解为“唯一套餐”的状态。

## 8. 后续注意事项

1. 新接口不要恢复 `current_package`、`subscription`、`subscriptions`
2. 新写入逻辑必须保持同槽位互斥、不同槽位并存
3. 预览和支付履约必须使用同一槽位规则
4. 如果将来要支持跨槽位转换，需要新增明确业务动作和迁移规则
