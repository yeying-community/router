# 夜莺币（YYC）账本与金额语义基线

## 1. 结论

基于当前代码与计费链路，最佳方案是：

1. 继续保留统一内部主账本单位。
2. 这个内部主账本单位，业务上统一命名为 `夜莺币`，简称 `YYC`，全称 `Yeying Coin`。
3. 上游模型价格继续按供应商官方币种和官方价格维护。
4. 请求结算时，先按供应商币种算出价格，再折算成 `YYC` 入账。
5. 数据库字段名、旧接口字段名中的 `quota` 暂时保留，只作为技术兼容名；业务语义统一解释为 `YYC`。

一句话概括：

- 上游计价单位是法币价格。
- 平台内部结算单位是 `YYC`。
- `quota` 是 `YYC` 的历史技术名，不再作为对外产品术语使用。

## 2. 为什么继续保留 YYC 主账本

继续使用统一主账本，而不是直接用法币做内部账本，有三个直接收益：

1. 不同供应商支持的计价币种不一样，内部统一成 `YYC` 后，余额、限额、令牌、兑换码、日志都可以共用一套账本。
2. `YYC` 使用整数，适合高频预占、并发结算、聚合统计。
3. 平台可以把“供应商成本币种”和“平台运营额度”分离，后续支持促销、赠送、套餐、折扣时更稳定。

因此，当前方向不是删除内部单位，而是把内部单位正式产品化。

## 3. 术语统一

### 3.1 业务术语

| 术语 | 含义 |
| --- | --- |
| 夜莺币 / YYC | 平台内部统一记账单位 |
| 金额 / amount | 面向人类展示的法币金额 |
| 价格 / price | 上游供应商官方定价 |
| 币种 / currency | 价格所属币种，如 `USD`、`CNY` |

### 3.2 技术兼容术语

| 旧技术名 | 新业务语义 |
| --- | --- |
| `quota` | `YYC` |
| `used_quota` | 已消费 `YYC` |
| `remain_quota` | 剩余 `YYC` |
| `daily_quota_limit` | 每日 `YYC` 上限 |
| `monthly_emergency_quota_limit` | 每月应急 `YYC` 上限 |
| `QuotaPerUnit` | 某一法币到 `YYC` 的兑换比率配置雏形 |

规则：

1. 对外产品、文档、页面文案，统一使用 `YYC`。
2. 代码和历史接口中的 `quota` 字段，解释为 `YYC`。
3. 只有在兼容说明和开发实现里，才继续出现 `quota` 这个词。

### 3.3 对外字段固定基线表

| 兼容字段 | 显式字段 | 统一语义 | 适用对象 |
| --- | --- | --- | --- |
| `quota` | `yyc_balance` / `yyc_amount` / `yyc_value` | `YYC`，具体含义由对象决定 | 用户余额 / 日志金额 / 兑换码面额 |
| `used_quota` | `yyc_used` | 已消费 `YYC` | 用户 / 令牌 / 渠道 / 仪表盘聚合 |
| `remain_quota` | `yyc_remain` | 剩余 `YYC` | 令牌 |
| `daily_quota_limit` | `yyc_daily_limit` | 每日 `YYC` 上限 | 用户 / 分组 |
| `monthly_emergency_quota_limit` | `yyc_monthly_emergency_limit` | 每月应急 `YYC` 上限 | 用户 |
| `consumed_quota` | `yyc_consumed` | 已结算消耗 `YYC` | 用户/分组额度快照 |
| `reserved_quota` | `yyc_reserved` | 已预占未结算 `YYC` | 用户/分组额度快照 |
| `remaining_quota` | `yyc_remaining` | 剩余可用 `YYC` | 用户/分组额度快照 |
| `user_daily_quota` | `yyc_user_daily` | 本次请求命中的用户日额度 `YYC` | 日志 |
| `user_emergency_quota` | `yyc_user_emergency` | 本次请求命中的用户月应急额度 `YYC` | 日志 |

规则：

1. 新接口优先直接返回显式 `yyc_*` 字段。
2. 旧接口继续保留兼容字段，直到明确宣布破坏性升级。
3. 如果同一对象已经存在兼容字段和显式字段，两者数值必须完全一致。

### 3.4 金额字段固定基线表

| 字段 | 语义 | 单位 |
| --- | --- | --- |
| `billing_input_amount` | 输入部分原始法币金额快照 | `billing_currency` |
| `billing_output_amount` | 输出部分原始法币金额快照 | `billing_currency` |
| `billing_amount` | 本次请求原始法币总金额快照 | `billing_currency` |
| `billing_yyc_rate` | 计费时法币到 `YYC` 的折算比率 | `YYC / currency` |
| `billing_yyc_amount` | 本次请求最终折算后的 `YYC` 金额 | `YYC` |

规则：

1. `amount` 只表示法币金额，不表示 `YYC`。
2. `yyc_*` 只表示 `YYC`，不表示法币金额。
3. 不允许在新增接口里继续新增一个含义不明的 `quota` 字段来表达法币。

### 3.5 兼容例外表

| 接口 / 场景 | 当前形状 | 原因 | 处理原则 |
| --- | --- | --- | --- |
| OpenAI 兼容 billing / usage 视图 | 返回 OpenAI 兼容对象 | 该类接口优先满足协议兼容，不适合混入 `yyc_*` 字段 | 在协议说明中解释内部仍然按 `YYC` 记账，不改响应形状 |
| 旧数据库列名 | 仍保留 `quota` | 涉及迁移成本与兼容面较大 | 业务语义统一解释为 `YYC`，逐步在接口层加显式别名 |

## 4. 当前现状

### 4.1 当前实际账本仍然是 `quota`

从代码看，当前链路已经不是“倍率估价”，而是：

1. 先从供应商模型目录读取官方价格。
2. 按价格单位和币种计算成本。
3. 立刻折算成内部 `quota`。
4. 后续余额、令牌、分组限额、用户限额、日志、报表都只记录 `quota`。

所以当前真实状态是：

- 价格层：已法币化
- 账本层：仍是 `quota`

这说明主账本思路没有问题，只是术语和多币种模型还没有正式化。

### 4.2 当前问题

当前问题不在于“有没有内部主账本”，而在于：

1. `quota` 这个名字太技术化，业务难理解。
2. 当前多币种支持仍然比较粗，只是折成同一个内部单位，没有独立的币种目录和兑换比率基线。
3. `QuotaPerUnit` 现在更像“默认缩放系数”，还不是正式的多币种兑换体系。
4. 文档、页面、接口没有明确说明“quota 就是 YYC”。

## 5. YYC 主账本方案

### 5.1 核心规则

平台内部统一使用 `YYC` 结算：

```text
上游价格（price + currency + unit）
  -> 按请求用量计算法币金额
  -> 按币种兑换比率折算为 YYC
  -> 使用 YYC 扣减余额 / 令牌 / 分组限额 / 用户限额
```

### 5.2 入账公式

统一公式：

```text
yyc_cost = price_amount × quantity × yyc_per_currency_unit(currency) × business_ratio
```

其中：

- `price_amount`：供应商官方价格
- `quantity`：本次请求实际计费数量
- `yyc_per_currency_unit(currency)`：该币种每 1 单位法币可兑换多少 YYC
- `business_ratio`：平台业务加成，例如分组倍率

示例：

- `gpt-5.4`
- 输入单价：`0.00125 USD / 1K tokens`
- 输入 tokens：`2000`
- `USD -> YYC` 比率：`500000`
- 分组倍率：`1`

则：

```text
法币成本 = 0.00125 × 2000 / 1000 = 0.0025 USD
YYC 成本 = 0.0025 × 500000 = 1250 YYC
```

### 5.3 结算对象

以下对象统一使用 `YYC`：

- 用户总余额
- 用户每日额度
- 用户月应急额度
- 分组每日额度
- 令牌剩余额度
- 兑换码面额
- 渠道累计消耗
- 日志聚合与报表

## 6. 多币种支持设计

### 6.1 目标

系统需要正式支持“供应商价格是多币种，平台账本是单一 YYC”的模式。

也就是说：

- 供应商价格保留原币种
- 平台账本不直接使用原币种
- 平台通过兑换比率把不同币种统一折成 `YYC`

### 6.2 建议的币种目录

建议新增正式的币种配置模型，例如：

### billing_currencies

| 字段 | 含义 |
| --- | --- |
| `code` | 币种代码，如 `USD`、`CNY`、`EUR` |
| `name` | 币种名称 |
| `symbol` | 显示符号，如 `$`、`¥`、`€` |
| `minor_unit` | 最小货币精度，如 `2` |
| `yyc_per_unit` | 每 1 单位该币种可兑换多少 `YYC` |
| `status` | 是否启用 |
| `source` | 汇率/运营配置来源 |
| `updated_at` | 更新时间 |

### 6.3 兑换比率规则

建议采用下面的规则：

1. `YYC` 永远是平台内部结算主单位。
2. 每个启用币种都需要配置 `yyc_per_unit`。
3. 供应商价格进入系统后，必须能找到该币种对应的 `yyc_per_unit`，否则不能参与计费。
4. 价格结算时使用“计费时快照”的兑换比率，而不是事后动态重算。

### 6.4 当前实现与目标实现的关系

当前已有：

- `QuotaPerUnit`
- `currency`
- `price_unit`

建议演进为：

1. `QuotaPerUnit` 逐步退化为 `USD -> YYC` 的默认兼容配置。
2. 真正的多币种兑换比率，统一进入币种目录表。
3. 结算逻辑优先查币种目录，不再写死 `USD/CNY` 的特殊处理。

当前实现补充：

- `QuotaPerUnit` 只驱动 `USD -> YYC` 的系统默认值兼容。
- `CNY -> YYC` 作为独立币种配置维护，不再按固定 `USD/CNY=7.0` 跟随联动更新。

## 7. API 与兼容规则

### 7.1 现状兼容

短期内，既有接口字段名不改：

- `quota`
- `used_quota`
- `remain_quota`
- `daily_quota_limit`
- `monthly_emergency_quota_limit`

但文档和页面统一解释为：

- 这些字段的值，单位都是 `YYC`

### 7.2 新增接口和新文档规则

后续新增接口，建议遵循以下规则：

1. 业务展示字段优先使用 `yyc` 语义命名，如：
   - `balance_yyc`
   - `daily_limit_yyc`
   - `used_yyc`
2. 兼容接口继续保留旧字段。
3. 如果要返回法币展示值，单独增加：
   - `amount`
   - `currency`
4. 不允许同一字段既表示 `YYC` 又表示法币。

### 7.3 页面展示规则

页面文案建议统一：

1. 内部额度显示为 `YYC`
2. 如果同时展示法币，则写成：
   - `1,250 YYC`
   - `约 $0.0025`
3. 不再使用“quota”作为页面术语
4. 不再使用“单位美元额度”这种难理解文案，改成：
   - `USD -> YYC 兑换比率`

## 8. 日志与审计建议

虽然主账本继续用 `YYC`，但为了后续审计，建议在日志层补齐法币快照字段。

建议后续消费日志增加：

- `billing_price_unit`
- `billing_currency`
- `billing_group_ratio`
- `billing_input_quantity`
- `billing_output_quantity`
- `billing_input_amount`
- `billing_output_amount`
- `billing_amount`
- `billing_yyc_rate`
- `billing_yyc_amount`

这样既保留 `YYC` 主账本，又能完整追溯“这笔消耗来自哪个币种、按什么价格、按什么业务倍率、按什么兑换比率折算”。

## 9. 分阶段落地建议

### P0：术语统一

1. 文档统一改称 `夜莺币 / YYC`
2. 页面文案逐步去掉 `quota`
3. 在开发文档中明确：`quota` 只是 `YYC` 的兼容字段名

### P1：币种目录

1. 增加正式的币种配置表
2. 支持配置启用币种和 `yyc_per_unit`
3. 结算逻辑由“写死默认换算”切到“查币种目录”

### P2：日志快照

1. 消费日志记录法币金额和 `YYC` 金额双快照
2. 图表和报表继续以 `YYC` 聚合
3. 详情页可同时展示原始法币成本

### P3：接口收口

1. 新接口优先使用 `yyc` 命名
2. 旧接口继续兼容 `quota`
3. 后台和用户台逐步只展示 `YYC`

## 10. 最终基线

最终应当统一为：

- 业务上：平台额度单位叫 `夜莺币（YYC）`
- 计费上：供应商价格按原币种维护
- 结算上：所有价格统一折算为 `YYC`
- 账本上：用户、令牌、分组、兑换码、日志全部以 `YYC` 记账
- 兼容上：旧代码和旧接口的 `quota` 字段继续存在，但语义统一解释为 `YYC`
