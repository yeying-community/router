# 分组说明

本文档说明 Router 中“分组（group）”的作用、数据结构、运行机制与管理建议。

补充说明：
- 本文档主要描述当前实现。
- 关于下一阶段的定价分层方案，见 [模型定价](./模型定价.md)。

## 1. 分组是什么

分组不是展示字段，而是系统的核心策略维度，用于：
- 控制用户可用模型范围
- 控制请求可路由到哪些渠道
- 控制计费倍率（组倍率）

一句话：`分组 = 权限边界 + 路由边界 + 计费边界`。

## 2. 分组在系统中的作用

### 2.1 路由控制
- 用户发起模型请求时，系统先读取用户所属分组。
- 再按 `group + model` 从能力表中选择可用渠道。
- 若该分组下没有可用渠道，会直接返回“当前分组下无可用渠道”。

### 2.2 模型可见性控制
- 用户在前端可见的模型列表不是全量，而是按用户分组过滤后的结果。
- 同一套系统中，不同分组可见模型可以不同。

### 2.3 计费倍率控制
- 实际扣费 = `基础价格 × 分组业务倍率`。
- 基础价格来自供应商默认价或渠道覆盖价。
- 分组倍率来自 `groups.billing_ratio`，可用于实现会员、企业、内部等差异化计费策略。

## 3. 核心数据结构

### 3.1 用户分组
- 表：`users`
- 字段：`group`
- 含义：用户所属分组（如 `enterprise`、`internal`）

### 3.2 分组目录
- 表：`groups`
- 主要字段：`id`、`name`、`description`、`billing_ratio`、`enabled`、`sort_order`、`source`
- 用途：管理端展示与维护分组元信息

### 3.3 分组与渠道绑定
- 接口：`GET/PUT /api/v1/admin/group/{id}/channels`
- 含义：管理员在分组管理页显式选择该分组可使用哪些渠道
- 结果：系统会按该分组绑定的渠道与渠道已选模型生成能力映射

### 3.4 能力映射（路由关键）
- 表：`group_model_channels`
- 关键维度：`group + model + channel_id`
- 含义：某分组是否可以通过某渠道使用某模型
- 来源：分组绑定渠道后，根据渠道已选模型批量刷新

### 3.5 分组倍率
- 字段：`groups.billing_ratio`
- 用途：定义每个分组的业务倍率（默认通常为 1）

## 4. 请求处理流程（简化）

1. 鉴权后拿到用户 ID  
2. 根据用户 ID 查询 `users.group`  
3. 根据 `group + model` 在 `abilities` 中选渠道（含优先级、可用状态）  
4. 发起上游请求  
5. 按 `基础价格 × 分组业务倍率` 计算消耗并记账

## 5. 管理接口（管理员）

当前分组管理接口位于：
- `GET /api/v1/admin/groups`：分页返回分组目录（支持 `page`、`page_size`、`keyword`）
- `POST /api/v1/admin/group`：新增分组
- `PUT /api/v1/admin/group`：更新分组（含启用状态）
- `DELETE /api/v1/admin/group/:id`：删除分组
- `GET /api/v1/admin/group/:id/channels`：读取该分组绑定的渠道
- `PUT /api/v1/admin/group/:id/channels`：更新该分组绑定的渠道

前端入口：管理员 Header 中“分组”Tab。

## 6. 约束与注意事项

- 正在被用户、渠道或能力映射使用的分组不允许删除。
- 仅创建分组目录并不会自动赋权；要让分组可用，需确保：
  - 用户被分配到该分组
  - 分组已绑定到合适的渠道
  - 能力映射已正确刷新
- 若修改了 `billing_ratio`，后续请求会按新的业务倍率计费。

## 7. 推荐实践

- 保持分组数量适中，建议按业务等级或业务线分层：`free / pro / enterprise` 或 `internal / external`。
- 分组命名使用稳定英文标识（展示名称可中文）。
- 新分组上线前做三项检查：
  - 用户是否已分配
  - 渠道是否已绑定
  - 模型列表是否符合预期

## 8. 常见问题

### Q1: 为什么用户看不到某个模型？
通常是该用户分组在 `abilities` 中没有对应的 `group + model` 可用记录，或相关渠道被禁用。

### Q2: 为什么请求报“当前分组下无可用渠道”？
该分组下没有满足条件的启用渠道，常见原因是分组未绑定渠道、模型不匹配、或渠道状态禁用。

### Q3: 分组只是前端分类吗？
不是。分组直接影响后端路由决策、模型可见性与业务倍率计费。
