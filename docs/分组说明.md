# 分组说明

本文档说明 Router 中“分组（group）”的作用、数据结构、运行机制与管理建议。

## 1. 分组是什么

分组不是展示字段，而是系统的核心策略维度，用于：
- 控制用户可用模型范围
- 控制请求可路由到哪些渠道
- 控制计费倍率（组倍率）

一句话：`分组 = 权限边界 + 路由边界 + 计费边界`。

## 2. 分组在系统中的作用

### 2.1 路由控制
- 用户发起模型请求时，系统先读取用户所属分组。
- 再按 `group + model` 从能力表中选择可用渠道。
- 若该分组下没有可用渠道，会直接返回“当前分组下无可用渠道”。

### 2.2 模型可见性控制
- 用户在前端可见的模型列表不是全量，而是按用户分组过滤后的结果。
- 同一套系统中，不同分组可见模型可以不同。

### 2.3 计费倍率控制
- 实际扣费倍率 = `模型倍率 × 分组倍率`。
- 分组倍率来自 `GroupRatio` 配置，可用于实现会员、企业、内部等差异化计费策略。

## 3. 核心数据结构

### 3.1 用户分组
- 表：`users`
- 字段：`group`
- 含义：用户所属分组（如 `default`、`vip`）

### 3.2 渠道分组
- 表：`channels`
- 字段：`group`
- 含义：该渠道允许哪些分组使用（支持多个分组，逗号分隔）

### 3.3 能力映射（路由关键）
- 表：`abilities`
- 关键维度：`group + model + channel_id`
- 含义：某分组是否可以通过某渠道使用某模型
- 渠道新增/编辑后会根据 `channels.group` 与 `channels.models` 更新能力映射

### 3.4 分组目录（管理面）
- 表：`groups`
- 主要字段：`name`、`display_name`、`description`、`enabled`、`sort_order`、`source`
- 用途：管理端展示与维护分组元信息

### 3.5 分组倍率
- 配置项：`GroupRatio`
- 用途：定义每个分组的倍率（默认通常为 1）

## 4. 请求处理流程（简化）

1. 鉴权后拿到用户 ID  
2. 根据用户 ID 查询 `users.group`  
3. 根据 `group + model` 在 `abilities` 中选渠道（含优先级、可用状态）  
4. 发起上游请求  
5. 按 `模型倍率 × 分组倍率` 计算消耗并记账

## 5. 管理接口（管理员）

当前分组管理接口位于：
- `GET /api/v1/admin/group`：返回启用分组名称列表（兼容旧下拉）
- `GET /api/v1/admin/group/catalog`：返回分组目录完整信息
- `POST /api/v1/admin/group`：新增分组
- `PUT /api/v1/admin/group`：更新分组（含启用状态）
- `DELETE /api/v1/admin/group/:name`：删除分组

前端入口：管理员 Header 中“分组”Tab。

## 6. 约束与注意事项

- `default` 为保底分组，不允许删除。
- 正在被用户、渠道或能力映射使用的分组不允许删除。
- 仅创建分组目录并不会自动赋权；要让分组可用，需确保：
  - 用户被分配到该分组
  - 渠道配置包含该分组
  - 能力映射已正确刷新
- 若修改了 `GroupRatio`，应确认新分组已同步到 `groups` 目录。

## 7. 推荐实践

- 保持分组数量适中，建议按业务等级分层：`default / vip / svip / enterprise`。
- 分组命名使用稳定英文标识（展示名称可中文）。
- 新分组上线前做三项检查：
  - 用户是否已分配
  - 渠道是否已覆盖
  - 模型列表是否符合预期

## 8. 常见问题

### Q1: 为什么用户看不到某个模型？
通常是该用户分组在 `abilities` 中没有对应的 `group + model` 可用记录，或相关渠道被禁用。

### Q2: 为什么请求报“当前分组下无可用渠道”？
该分组下没有满足条件的启用渠道（可能是渠道分组未配置、模型不匹配、或渠道状态禁用）。

### Q3: 分组只是前端分类吗？
不是。分组直接影响后端路由决策、模型可见性与计费倍率。
