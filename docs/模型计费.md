# 模型计费

本文档描述 Router 当前版本已经落地并在运行时生效的模型计费实现。

目标：

1. 说明模型价格如何被解析与覆盖。
2. 说明 `provider_model_price_components.condition` 的真实匹配规则。
3. 说明不同模态在运行时如何换算为 `YYC`。

## 1. 运行时计费链路

单次请求进入 relay 后，模型计费按以下顺序执行：

1. 解析本次命中的分组渠道倍率 `group_channels.billing_ratio`（默认 `1`）。
2. 解析基础模型价格（`ResolveChannelModelPricing`）：
- 先取 `provider_models` 默认价格（`source=provider_default`）。
- 再应用 `channel_models` 渠道覆盖价（`source=channel_override`，仅覆盖大于 0 的 `input_price/output_price`）。
3. 按请求类型尝试命中价格组件（`provider_model_price_components`）：
- 文本：`ResolveTextRequestPricing`
- 图片：`ResolveImageRequestPricing`
- 音频：`ResolveAudioRequestPricing`
- 视频：`ResolveVideoRequestPricing`
4. 使用最终 `price_unit + currency + group_channel_ratio` 计算本次消耗，向上取整到 `YYC` 整数。

说明：

- 若命中价格组件，`source` 会变为 `provider_component`。
- 价格组件命中后，可能覆盖前一步的渠道覆盖价，这是当前代码行为。

## 2. 数据模型与职责

### 2.1 `provider_models`

职责：模型默认价格真源。

关键字段：

- `provider`
- `model`
- `type`
- `input_price`
- `output_price`
- `price_unit`
- `currency`

### 2.2 `provider_model_price_components`

职责：在特定条件下覆盖默认价格。

关键字段：

- `provider`
- `model`
- `component`
- `condition`
- `input_price`
- `output_price`
- `price_unit`
- `currency`
- `sort_order`

唯一键由主键保证：`provider + model + component + condition`。

### 2.3 `channel_models`

职责：渠道级绝对覆盖价。

关键字段：

- `channel_id`
- `model` / `upstream_model`
- `input_price`
- `output_price`
- `price_unit`
- `currency`
- `selected`

### 2.4 `group_channels`

职责：某个渠道在某个分组内的业务倍率。

关键字段：

- `billing_ratio`

## 3. 价格组件匹配规则（`condition`）

匹配函数：`providerPriceComponentMatches`。

语法：

```text
key=value;key=value
```

规则：

1. 分号 `;` 分隔多个条件。
2. 每段必须是 `key=value`，否则整条条件视为不匹配。
3. 键和值匹配时均转小写并去空格。
4. 所有键都命中才匹配成功（AND 关系）。
5. 空字符串 `condition=""` 视为该 component 的默认规则。

### 3.1 匹配优先级

同一模型同一 component 下，组件会先经过标准化排序，再按顺序“首条命中即返回”。

排序规则：

1. `sort_order` 非 0 的排前面，且值越小优先级越高。
2. `sort_order=0` 视为未设置，排在后面。
3. 同优先级再按 `component`、`condition` 字典序。

建议：

- 需要明确优先级时，务必填写 `sort_order`，避免同类规则命中顺序不可读。

## 4. 当前已生效的 component 与条件键

以下均为“当前代码真实传入的匹配属性”，不是规划。

### 4.1 文本

- component: `text`
- 条件键：`endpoint`
- endpoint 值来源：上游真实请求端点（例如 `/v1/responses`、`/v1/chat/completions`、`/v1/messages`）

示例：

```text
endpoint=/v1/responses
```

### 4.2 图片

- component: `image_generation`
- 条件键：`size`、`quality`

示例：

```text
quality=hd;size=1024x1024
```

说明：

- 命中图片组件后，图片链路会将旧的 `imageCostRatio` 置为 `1`，直接按组件价格计费。
- 图片组件会直接使用 `component.input_price` 作为输入价格（即使为 `0`）。

### 4.3 音频

- component: `audio_input`（转录/翻译）
- component: `audio_output`（TTS）
- 条件键：当前无（调用时 attrs 为 `nil`）

建议写法：

```text
component=audio_input, condition=""
component=audio_output, condition=""
```

### 4.4 视频

- component: `video_generation`
- 条件键：`resolution`、`size`、`quality`

示例：

```text
resolution=720p
quality=high;size=1280x720
```

### 4.5 已定义但当前未接入请求计费匹配

- `realtime_text`
- `realtime_audio`

说明：

- 这两个 component 常量已定义并可入库，但当前请求计费路径没有对应的 `Resolve...RequestPricing` 调用。

## 5. 计价单位与换算

支持的 `price_unit`：

- `per_1k_tokens`
- `per_1k_chars`
- `per_image`
- `per_video`
- `per_second`
- `per_minute`
- `per_request`
- `per_task`

金额换算规则：

1. `per_1k_tokens` / `per_1k_chars`：`amount = quantity * price / 1000`
2. 其余单位：`amount = quantity * price`
3. `charge_amount = amount * charge_rate * group_channel_ratio`
4. 账本入账 `YYC` 使用向上取整（`ceil`）

## 6. 各模态 quantity 口径（当前实现）

### 6.1 文本

- 输入 quantity：`prompt_tokens`
- 输出 quantity：`completion_tokens`

### 6.2 图片

- quantity：`image_count * imageCostRatio`
- 当命中 `image_generation` 组件时，`imageCostRatio` 强制为 `1`

补充说明：

- 这套 quantity 口径只对“按张 / 按次”图片计费天然成立。
- 当某些图片模型被录入为 `per_1k_tokens` 一类 token 单位时，当前实现会把 `quantity=1` 直接带入 token 计费公式，得到明显偏低的金额。
- `gpt-image-2` 当前就存在这个问题，因此不能把“图片数量”与“token 数量”混为同一个 quantity 槽位。

### 6.3 音频

- `audio_output`（TTS）：按输入文本字符数计量
- `audio_input`（转录/翻译）：按输出文本 token 数计量

### 6.4 视频

- `per_second`：优先 `request.seconds`，否则解析 `request.duration`（如 `10s`），再否则为 `1`
- `per_minute`：同上，但秒数除以 `60`
- 其他单位：quantity 固定为 `1`

## 7. 图片计费专项：`gpt-image-2` 问题与改造方案

这一节描述图片计费链路的现状问题，以及后续改造方向，便于持续评估。

### 7.1 已确认过的问题

以 `gpt-image-2` 为例，旧实现曾存在以下组合问题：

1. 模型默认价格被配置为图片模型，但 `price_unit=per_1k_tokens`。
2. 图片控制器仍按 `image_count * imageCostRatio` 生成 quantity。
3. 当 `image_generation` 组件命中后，`imageCostRatio` 会被强制改成 `1`。
4. 最终计费层把这个 `1` 当成“1 token”参与 `per_1k_tokens` 公式。

结果就是：

```text
quantity = 1
amount = 1 * price / 1000
```

这会把一张图片请求错误地当成“1 个 token”计费，金额会显著偏低。  
因此，`gpt-image-2` 这类模型不能继续沿用图片链路旧的统一 quantity 口径。

### 7.2 问题根因

当前图片链路把三类不同语义混在了一起：

1. 按张计费：例如 `per_image`
2. 按次计费：例如 `per_request` / `per_task`
3. 按 token 或其他 usage 计费：例如 `per_1k_tokens`

但运行时只保留了一个 quantity：

```text
image_count * imageCostRatio
```

这对前两类还能勉强成立，但对第三类不成立，因为：

- `image_count` 不是 token 数
- 图片请求返回体通常也不会天然给出可直接复用的文本 token usage
- 即使上游某些模型内部按 token 计价，Router 也不能把“1 张图”直接替代成“1 token”

### 7.3 参考 `new-api` 后得到的结论

对照 `new-api` 的实现，可以确认图片相关计费至少要拆成两大家族：

1. 传统图片端点
- 例如 `/v1/images/generations`
- 更接近“按张 / 按次 / 按固定规格”计费

2. `responses` 内嵌图片生成工具调用
- 例如 `/v1/responses` 中触发 `image_generation`
- 本质更接近“文本主链路 + 图片工具附加费用”

`new-api` 的关键做法不是把所有图片都塞进同一种 quantity，而是：

1. 文本主链路继续按 token 结算
2. 图片工具调用额外识别 `image_generation`
3. 再叠加独立的图片工具费用

这说明 Router 后续也应该把“传统图片端点计费”和“responses 图片工具计费”拆开，而不是继续复用同一套图片 quantity。

### 7.4 Router 后续建议的图片计费分类

后续应先给图片模型或图片请求显式分类，再决定如何结算：

1. `per_image`
- 按图片张数计费
- quantity 语义可以继续使用 `image_count`

2. `per_call`
- 按请求次数或任务次数计费
- quantity 固定为 `1`

3. `token_based`
- 只能在存在可信 usage 时按 token 结算
- 不能再复用 `image_count`

4. `unsupported`
- 当前无法可靠估算，也无法可靠结算
- 应拒绝接入，或显式标记为未支持

这里的重点不是“模型名属于哪一类”，而是“模型 + endpoint + 计价单位 + 是否有可信 usage”共同决定属于哪一类。

### 7.5 当前实现状态

当前已经完成第一轮止血：

1. `ComputeImageBillingSnapshot` 不再允许 token 单位图片模型落回 `image_count * imageCostRatio`。
2. `/v1/images/generations` 对 token 单位图片模型已拆成两条本地估算路径：
   - 旧版 `gpt-image-*`（不含 `gpt-image-2`）：
     - prompt token：按输入 prompt 本地估算
     - image output token：按 `size + quality + n` 查本地固定表
   - `gpt-image-2`：
     - prompt token：按输入 prompt 本地估算
     - output amount：按官方公开的常见 `size + quality` 示例价格估算
     - output quantity：仅用于日志与统一结算结构，按 `output_amount / output_price` 反推出等价 quantity，不表示上游真实 usage
   - input price：从 `text` 组件读取
   - output price：从 `image_generation` 组件读取
3. 日志会落下计费来源字段：
   - `billing_usage_source=local_estimate`
   - `billing_estimate_source=traditional_image_tokens_local` 或 `gpt_image_2_examples_local`
   - `billing_settlement_mode=local_estimate_final`

这意味着 `gpt-image-2 + /v1/images/generations` 当前可以生成并计费，但它不是上游 usage 实结，而是“本地估算即最终结算”。其中金额优先对齐官方公开价格，quantity 只是为了兼容现有结算结构而反推出来的等价值。如果上游未来返回可信 usage，应再切换到 usage 优先的实结口径。

仍保持保守的场景：

- 非 `gpt-image-*` 的 token 单位图片模型不自动套用这套估算。
- `size` / `quality` 不在本地 token 表内时直接报错。
- 缺少 `text` 输入价格或输出价格时直接报错，不使用默认值兜底。

### 7.6 后续拆分方向

当前不应再让所有图片请求都走一个统一公式。后续继续拆成三条明确路径：

1. `ComputeImagePerImageQuota`
2. `ComputeImagePerCallQuota`
3. `ComputeTraditionalImageTokenBasedQuota`

其中：

- `per_image` 只消费图片张数
- `per_call` 只消费调用次数
- `token_based` 必须显式依赖可信 usage，不能默认回退到 `1`

### 7.7 `responses` 图片工具计费

对于 `/v1/responses` 中的 `image_generation`：

1. 文本部分继续按文本主链路计费
2. 图片生成部分作为独立附加费用识别
3. 附加费用可按 `size`、`quality` 等条件命中 component

这样可以避免把 `responses` 里的图片生成硬塞进传统 `/v1/images/generations` 的计费模型。

当前已经能从 OpenAI `responses` 响应里识别实际发生的 `image_generation_call`，并在安全场景下叠加图片工具费用。不能安全判断时，日志中的 `billing_settlement_mode` 会保持 `responses_image_tool_pending`，用于后续排查和补齐。

### 7.8 对配置层的约束建议

在实现改造前，配置侧需要遵循以下约束：

1. `per_image` 只用于确实按张计价的图片模型或图片 component。
2. `per_request` / `per_task` 只用于确实按次计价的图片模型。
3. 不要再把“没有 usage 的传统图片端点”简单录成 `per_1k_tokens` 后直接上线。
4. 若某模型官方口径更接近“工具调用附加费”，应优先走 `responses` 工具型方案，而不是伪装成传统图片端点。

### 7.9 观测与测试要求

后续落地时，图片链路至少应补齐以下观测字段：

- `image_billing_mode`
- `image_pricing_source`
- `image_usage_source`
- `image_generation_call`
- `image_generation_call_price`

至少覆盖以下测试场景：

1. `dall-e-3` 这类按张计费模型，验证 `per_image`
2. `gpt-image-2 + /v1/images/generations`，验证不会再落到错误的 token quantity 路径
3. `/v1/responses + image_generation`，验证“文本主链路 + 图片附加费”口径
4. `size` / `quality` component 命中顺序与金额快照

### 7.10 当前结论

当前版本中，图片链路的统一 quantity 模型并不适合所有图片模型。  
特别是 `gpt-image-2` 这类“配置成 token 单位、但请求又是传统图片端点”的组合，必须停止继续套用旧的 `image_count * imageCostRatio` 逻辑。

当前评估基线应以“先区分计费模式，再决定是否允许上线”为准，而不是默认所有图片模型都能复用同一套结算公式。已经支持的 token 估算路径，也必须在日志中明确标记为本地估算，不能伪装成上游 usage 实结。

## 8. 配置建议（避免无效规则）

1. 条件键、条件值、endpoint 路径统一用小写。
2. 文本链路只写 `endpoint=...`，避免录入当前不会传入的键。
3. 音频链路当前不要写非空 `condition`，否则无法命中。
4. 视频链路优先使用 `resolution/size/quality` 三类键。
5. 相同 `provider+model+component+condition` 不要重复定义（数据库主键也会约束）。
6. 需要控制命中优先级时设置 `sort_order`，不要依赖默认排序。

## 9. 当前整体进展评估

这一节从“价格来源是否统一”“预扣是否本地化”“最终结算是否使用真实 usage”三个维度，评估当前重构在各模态上的完成度。

### 9.1 文本

当前状态：

- 基础模型价格解析已统一走 `ResolveChannelModelPricing`
- endpoint 维度组件价格已接入 `ResolveTextRequestPricing`
- 请求前预扣已切换到本地估算器 `tokenestimate.Estimate(...)`
- 请求后结算优先使用上游 usage，缺失时再本地兜底

完成度判断：

- 这是当前完成度最高的一条主链路
- 已基本达到“本地预扣 + 真实 usage 结算”的目标状态
- 仍缺少“预估值 vs 实际值偏差”的持久化观测

### 9.2 音频

当前状态：

- 基础模型价格解析已统一走 `ResolveChannelModelPricing`
- 已支持 `audio_input` / `audio_output` component
- TTS 当前按输入字符数计费
- 转录 / 翻译当前按 token 型规则预扣

存在问题：

- TTS 更接近规则型计费，而不是 usage 型结算
- 转录 / 翻译虽然已有后续处理，但没有完全收敛到与文本相同的“预扣 / 实结”结构
- 音频链路的 usage 来源、估算来源、结算来源还没有统一观测

完成度判断：

- 已进入重构主链路
- 但尚未完全实现“文本同等级别”的一致性

### 9.3 图片

当前状态：

- 基础模型价格解析已统一走 `ResolveChannelModelPricing`
- 已支持 `image_generation` component（`size` / `quality`）
- 已识别出“传统图片端点”和“responses 图片工具调用”必须拆开处理
- 已阻止最明显的 `image_count -> token` 错误路径
- `/v1/images/generations` 对 `gpt-image-*` 已支持本地 token 估算计费
- 成功的图片请求会记录消费日志；即使最终 quota 为 0，也会保留请求日志，避免成功请求从历史中消失

存在问题：

- 传统图片端点当前仍主要依赖本地规则估算
- 图片响应结构当前没有统一接入 usage 结算链路
- `gpt-image-2` 这类“token 单位 + 图片端点”组合目前是本地估算最终结算，还不是上游 usage 实结
- `responses + image_generation` 已能识别实际图片工具调用并在安全场景叠加费用，但复杂工具组合仍会保持 pending 标记

完成度判断：

- 方案已经明确
- 实现已经完成第一轮止血和观测补齐，仍需继续向 usage 优先结算收束
- 这是当前风险最高、优先级最高的模态

### 9.4 视频

当前状态：

- 基础模型价格解析已统一走 `ResolveChannelModelPricing`
- 已支持 `video_generation` component
- quantity 已按 `per_second` / `per_minute` / `per_video` 等规则拆分

存在问题：

- 当前更接近“按请求参数规则计费”
- 并未形成稳定的 usage 型结算链路
- 视频任务型接口的“结算时机”和“最终消耗来源”还没有独立抽象

完成度判断：

- 已具备基础计费能力
- 仍属于规则型实现，不是完整的预扣 / 实结双阶段模型

### 9.5 Realtime

当前状态：

- 测试链路与端点选择能力已在推进
- `realtime_text` / `realtime_audio` component 常量已定义

存在问题：

- 运行时计费路径还没有完整接入 realtime 专项 pricing resolve
- 也没有形成稳定的 realtime usage 结算模型

完成度判断：

- 目前仍以测试与路由能力建设为主
- billing 尚未真正展开

## 10. 当前阶段的关键结论

当前重构已经形成两个非常不同的成熟度层级：

1. 文本主链路
- 已基本完成“本地估算预扣 + 上游 usage 实际结算”
- 是当前最稳定、最接近目标形态的实现

2. 非文本模态（音频 / 图片 / 视频 / realtime）
- 都已经接入了统一的“基础价格解析”框架
- 但“估算来源、usage 来源、最终结算来源”的统一程度仍明显不足

因此，当前不能简单认为“全链路都已完成 new-api 式重构”。  
更准确的说法是：

- **价格解析骨架已统一**
- **文本预扣 / 实结模型已基本落地**
- **非文本模态仍处于持续重构阶段**

## 11. 下一阶段优先级

建议按以下顺序继续推进：

1. 图片
- 优先完成传统图片端点与 `responses + image_generation` 的拆分收口
- 补齐 usage 来源 / 估算来源 / 结算来源的显式观测
- 明确哪些图片模型允许上线，哪些仍应保持保守或禁止

2. 音频
- 梳理 TTS、转录、翻译三条口径
- 明确哪些链路应继续走规则型结算，哪些可切换为 usage 型结算

3. 视频
- 抽象任务型接口的最终结算时机
- 避免继续把视频简单等同为静态 quantity 规则

4. Realtime
- 在测试和路由能力稳定后，再补专门的计费与 usage 处理

## 12. 需要持续补齐的观测项

无论哪种模态，后续都建议补齐以下观测字段：

- `pricing_source`
- `usage_source`
- `estimate_source`
- `settlement_mode`
- `preconsumed_quota`
- `final_consumed_quota`

其中最重要的是要能区分：

1. 这是“本地估算值”
2. 这是“上游真实 usage 值”
3. 这是“最终落账结算值”

当前进展补充：

- 图片链路已先把这四个字段落到消费日志：
  - `billing_pricing_source`
  - `billing_usage_source`
  - `billing_estimate_source`
  - `billing_settlement_mode`
- 文本链路额外记录估算观测字段：
  - `estimated_prompt_tokens`
  - `billing_estimate_estimator`
  - `billing_estimate_precision`
- 现阶段主要用于区分：
  - 价格来自模型基础价还是组件命中价
  - 当前是否只有本地估算、没有上游 usage
  - 最终落账是否仍是“本地估算即最终结算”
- 文本链路也已开始补齐同口径字段：
  - 普通文本请求可标记为 `usage_final`
  - `/v1/responses` 中携带 `image_generation` 工具的请求，会标记为 `responses_image_tool_pending`
  - 这个标记表示：当前仍主要依据文本 usage 结算，图片工具附加费尚未单独拆账
- 当前进一步收口到：
  - Router 已能从 `responses` 响应里识别实际发生的 `image_generation_call`
  - 当请求中只有一个 `image_generation` 工具定义，且该工具模型可按 `per_image` / `per_request` / `per_task` 安全计费时，已可在文本 usage 之外叠加图片附加费
  - 对 `gpt-image-*` 这类 token 单位工具模型，也可在支持的 `size` / `quality` 范围内按本地图片输出 token 表叠加输出费用
  - 这类请求的 `billing_settlement_mode` 会收敛到 `usage_plus_image_fee`
  - 若存在多个图片工具定义、或无法安全判断具体工具计费参数，仍保持 `responses_image_tool_pending`
- 其他模态后续应按同样口径逐步补齐，而不是继续只靠 `content` 文本描述排查。
