# 渠道模型兼容策略

## 1. 背景

Router 同时连接两类不完全一致的系统：

1. 下游客户端：Chat、Codex CLI、第三方 SDK、浏览器调试工具等。
2. 上游渠道：官方供应商、兼容接口、中转渠道、本地模型服务等。

即使客户端和上游都声称支持同一个协议，也可能存在细节差异。例如：

- 同为 Anthropic `/v1/messages`，官方支持图片 URL，但部分兼容渠道只在 base64 图片下真正可用。
- 同为 OpenAI `/v1/responses`，不同渠道对旧参数、流式响应、工具调用、图片/PDF 输入的支持程度不同。
- 同一模型在不同渠道上的能力不完全一致，不能只按模型名判断。
- 上游返回的错误信息可能被兼容层包装，不能直接反映真实根因。

如果 Router 只是简单透传，这些差异会被暴露给客户端，导致每个客户端都需要理解渠道细节。Router 的价值应当体现在：在可解释、可测试、可审计的前提下，把下游标准请求调整为当前上游渠道实际能正确处理的请求。

## 2. 设计目标

1. 把渠道差异显式建模，避免把兼容逻辑散落在 controller 和 adaptor 的条件分支里。
2. 以 `channel + model + endpoint` 为最小策略粒度，描述某个渠道模型在某个端点下的真实能力与必要修正。
3. 所有自动修正必须可解释：能看到命中的规则、修改了什么、为什么修改。
4. 所有策略必须可测试：模型测试链路能验证策略前后的请求与上游响应。
5. 客户端继续以 `/v1/models` 暴露的 `owned_by` 和 `supported_endpoints` 做端点选择，Router 在选中具体渠道后应用渠道策略。
6. 策略动作保持有限枚举，不引入任意脚本或不可审计的动态逻辑。

## 3. 非目标

1. 不把 Router 做成通用脚本执行引擎。
2. 不用兼容策略替代协议适配器。协议级 URL、鉴权、基础请求/响应转换仍属于 adaptor。
3. 不用策略掩盖真实不支持的能力。不能明确修正的输入应返回清晰错误，而不是继续盲目转发。
4. 不改变现有端点协商原则。请求 `/v1/messages` 就只在上游支持 `/v1/messages` 时路由，不隐式回退到 `/v1/responses` 或 `/v1/chat/completions`。
5. 不把所有策略都放进 `channel.config`。`channel.config` 继续用于渠道运行参数，兼容策略应独立建模。

## 4. 现有能力基础

当前代码已经具备几块基础：

| 能力 | 当前载体 | 说明 |
| --- | --- | --- |
| 模型归属 | `provider_models`、`/v1/models.owned_by` | 用于告诉客户端模型供应商归属 |
| 渠道模型 | `channel_models` | 维护模型启用、上游模型名、模型类型、主端点 |
| 端点真值 | `channel_model_endpoints` | 维护 `channel + model + endpoint` 是否启用 |
| 端点测试 | `channel_tests` | 记录端点测试结果、请求与响应产物，供管理端验收与排查使用 |
| 端点协商 | `resolveChannelTextUpstream` | 根据下游请求端点和渠道模型端点能力决定上游端点 |
| 请求转换 | `getRequestBody`、adaptor `ConvertRequest` | 当前转换逻辑主要硬编码在控制器和适配器里 |

缺失的是独立的“端点内能力与修正规则”层。例如 `/v1/messages` 支持本身只能说明这个端点可用，不能说明该端点是否支持 `image.url`、`image.base64`、`pdf.url`、`tools`、`stream` 等更细能力。

## 5. 分层模型

建议把模型能力和兼容策略分成五层：

| 层级 | 示例载体 | 职责 |
| --- | --- | --- |
| 供应商基线层 | `provider_models`、后续供应商能力表 | 描述模型官方或基线能力，例如是否是文本/图片/音频模型 |
| 协议适配层 | adaptor | 描述 OpenAI、Anthropic、Gemini 等协议的 URL、鉴权、基本格式转换 |
| 渠道模型层 | `channel_models` | 描述该渠道暴露了哪些模型、上游模型名是什么 |
| 渠道模型端点层 | `channel_model_endpoints` | 描述该渠道模型在某端点是否可路由 |
| 渠道模型端点策略层 | 新增策略表 | 描述该端点内部的细粒度能力和必要请求修正 |

端点层回答“能不能走这个端点”。策略层回答“走这个端点时，哪些输入能直接走，哪些输入需要明确修正”。

## 6. 数据模型建议

新增表：`channel_model_endpoint_policies`。

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | varchar(64) | 策略 ID |
| `channel_id` | varchar(64) | 渠道 ID |
| `model` | varchar(255) | Router 对外模型名 |
| `endpoint` | varchar(255) | 归一化端点，例如 `/v1/messages` |
| `enabled` | bool | 策略是否启用 |
| `capabilities` | json/text | 该渠道模型端点的细粒度能力声明 |
| `request_policy` | json/text | 请求侧修正规则 |
| `response_policy` | json/text | 响应侧修正规则，首期可为空 |
| `reason` | text | 为什么需要该策略，必须填写 |
| `source` | varchar(32) | `manual`、`test_detected`、`imported` |
| `last_verified_at` | bigint | 最近验证时间 |
| `updated_at` | bigint | 更新时间 |

主唯一键建议为：

```text
channel_id + model + endpoint
```

首期不建议做多条策略叠加。一个 `channel + model + endpoint` 对应一份策略，降低执行顺序复杂度。

## 7. 策略 JSON 结构

建议 `capabilities` 只描述事实，不描述动作：

```json
{
  "input_text": true,
  "input_image_url": false,
  "input_image_base64": true,
  "input_pdf_url": false,
  "input_pdf_file": false,
  "tools": true,
  "stream": true,
  "non_stream": true
}
```

建议 `request_policy` 描述有限枚举动作：

```json
{
  "actions": [
    {
      "type": "image_url_to_base64",
      "when": {
        "field": "content.image.source.type",
        "equals": "url"
      },
      "limits": {
        "max_bytes": 5242880,
        "allowed_content_types": ["image/png", "image/jpeg", "image/webp", "image/gif"]
      },
      "reason": "该渠道 /v1/messages 对 image url 返回 200 但模型不可见，base64 已验证可用"
    },
  ]
}
```

策略必须满足：

1. `type` 只能是代码内定义的枚举。
2. 每个 action 必须有 `reason`。
3. 所有资源拉取动作必须有大小、类型和超时限制。
4. 不支持的 action 直接拒绝加载或拒绝执行。

## 8. 首批动作枚举

首期建议只实现请求侧动作。

| 动作 | 说明 | 适用场景 |
| --- | --- | --- |
| `rename_fields` | 字段重命名 | 兼容少量字段名差异 |
| `set_fields` | 设置固定字段 | 某些渠道要求固定版本、模式或开关 |
| `set_headers` | 设置或覆盖上游请求头 | 某些渠道需要额外 beta header |
| `image_url_to_base64` | 下载图片 URL 并转为 base64 输入 | Anthropic 兼容渠道不实际支持图片 URL |

暂缓实现：

| 动作 | 暂缓原因 |
| --- | --- |
| 任意 JSONPath 表达式 | 容易失控，调试成本高 |
| 任意脚本 | 不可审计，安全风险高 |
| 跨端点自动转换 | 会破坏当前严格端点协商语义 |
| 响应语义重写 | 容易影响计费、流式解析和错误判断，后续单独设计 |

## 9. 运行时流程

请求进入 Router 后建议按如下顺序处理：

1. 解析请求并得到下游端点。
2. 按现有路由逻辑选择具体渠道。
3. 根据 `channel + model + endpoint` 解析上游端点。
4. 加载对应策略。
5. 在构造上游请求体和请求头之前应用请求策略。
6. 记录策略命中日志。
7. 发送上游请求。
8. 响应按现有 adaptor 流程回写客户端。

伪代码：

```go
upstreamMode, upstreamPath := resolveChannelTextUpstream(meta, originModel, actualModel)
policy := LoadEndpointPolicy(meta.ChannelId, originModel, upstreamPath)
requestBody := BuildRequestBody(...)
requestBody, requestHeaders, report := ApplyRequestPolicy(policy, requestBody, requestHeaders)
LogPolicyReport(report)
resp := adaptor.DoRequest(ctx, meta, requestBody)
```

策略应用点应靠近 `getRequestBody`。原因是当前 `messages -> messages` 和 `responses -> responses` 有直通分支，如果策略应用在 adaptor `ConvertRequest` 之后，直通请求可能绕过策略。

## 10. 日志与可解释性

每次命中策略应记录结构化日志，至少包含：

| 字段 | 说明 |
| --- | --- |
| `trace_id` | 请求 trace |
| `channel_id` | 渠道 |
| `origin_model` | 下游模型名 |
| `actual_model` | 上游模型名 |
| `endpoint` | 上游端点 |
| `policy_id` | 策略 ID |
| `actions` | 命中的动作列表 |
| `changed_fields` | 修改字段摘要 |
| `reason` | 动作原因 |
| `input_summary` | 输入摘要，不能记录完整 base64 |

示例：

```text
[endpoint_policy] channel_id=hanbbq-3 model=claude-opus-4-6 endpoint=/v1/messages policy_id=... actions=image_url_to_base64 changed_fields=messages[0].content[0].source reason="..."
```

日志中不得完整输出图片 base64、文件内容、密钥、UCAN token。

## 11. 测试与验证

现有模型测试已经能按端点执行，并记录请求与响应。兼容策略上线后，应扩展测试维度：

1. 文本基础测试：确认端点仍可用。
2. 图片 URL 测试：验证上游是否能直接识别 URL 图片。
3. 图片 base64 测试：验证 base64 是否可用。
4. PDF 测试：验证 PDF 输入格式是否可用。
5. 工具调用测试：验证 tools/tool_choice 是否可用。
6. 流式测试：验证 `stream=true`。
7. 非流式测试：验证 `stream=false`。

测试结果应区分：

| 状态 | 含义 |
| --- | --- |
| `supported` | 原生支持或策略修正后可稳定支持 |
| `unsupported` | 明确不可用 |
| `requires_policy` | 原生不可用，但有明确策略可修正 |
| `unknown` | 未测试或结果无法判定 |

首期可不改 `channel_tests.status` 枚举，只在策略表和测试产物里记录 `requires_policy`。等 UI 和聚合逻辑准备好后再提升为正式状态。

## 12. `/v1/models` 暴露建议

短期继续保持：

```json
{
  "id": "claude-opus-4-6",
  "owned_by": "anthropic",
  "supported_endpoints": ["/v1/messages"]
}
```

不建议立刻把所有细粒度策略暴露给普通客户端。原因是客户端只需要知道可用端点，具体渠道策略应由 Router 承担。

后续如果客户端需要更细能力，可增加可选字段：

```json
{
  "supported_inputs": {
    "/v1/messages": ["text", "image"]
  }
}
```

该字段语义必须是“客户端可用能力”，而不是“上游原生能力”。例如上游不支持 URL 图片但 Router 能转 base64，则对客户端可以表现为支持图片。

## 13. 管理端交互建议

渠道模型详情页建议增加“端点策略”区域：

1. 展示每个端点的细粒度能力。
2. 展示启用的策略动作。
3. 展示最近验证时间和最近一次测试结果。
4. 支持手动编辑 `reason`。
5. 支持下载测试产物。
6. 对资源拉取类动作展示大小限制、超时限制、允许 MIME 类型。

管理端不要提供任意 JSON 编辑作为唯一入口。可以保留高级 JSON 视图，但常用策略应通过明确表单配置。

## 14. 当前 Claude 图片问题的策略表达

问题事实：

1. 官方 Anthropic `/v1/messages` 支持 `source.type = "url"` 图片。
2. 当前部分兼容渠道对 URL 图片返回 `HTTP 200`，但模型回复“看不到图片”。
3. 同一图片转为 base64 后，在部分渠道上可以正常识别。

建议策略：

```json
{
  "capabilities": {
    "input_text": true,
    "input_image_url": false,
    "input_image_base64": true,
    "stream": true,
    "non_stream": true
  },
  "request_policy": {
    "actions": [
      {
        "type": "image_url_to_base64",
        "limits": {
          "max_bytes": 5242880,
          "timeout_ms": 10000,
          "allowed_content_types": ["image/png", "image/jpeg", "image/webp", "image/gif"]
        },
        "reason": "该渠道 Anthropic 兼容层不可靠支持 source.type=url，base64 已验证可用"
      }
    ]
  },
  "reason": "2026-04-30 直连上游验证：国外 Wikimedia 图片 URL 与 WebDAV 图片 URL 均返回 200 但模型不可见，base64 可识别"
}
```

执行效果：

1. 客户端仍可发送图片 URL。
2. Router 命中该渠道模型端点策略后，下载图片并改写为 Anthropic base64 图片块。
3. 如果图片下载失败、大小超限、MIME 类型不允许，则 Router 返回明确错误。
4. 日志记录本次命中的策略与改写摘要。

## 15. 错误处理

策略执行失败时不要继续盲目转发，应返回明确错误。

建议错误码：

| code | 场景 |
| --- | --- |
| `endpoint_policy_failed` | 策略执行失败的通用错误 |
| `unsupported_input` | 输入命中了不支持且不可转换的能力 |
| `media_fetch_failed` | 下载远程媒体失败 |
| `media_too_large` | 媒体超过策略限制 |
| `unsupported_media_type` | MIME 类型不允许 |
| `invalid_policy` | 策略配置非法 |

错误信息应包含：

1. 渠道模型端点。
2. 命中的策略动作。
3. 失败原因。
4. 可操作建议。

不要把上游模糊错误直接包装成“渠道不可用”。

## 16. 安全边界

资源拉取类策略必须有安全限制：

1. 只允许 `http` 和 `https`。
2. 禁止访问内网地址、回环地址、链路本地地址和 metadata 地址。
3. 设置最大下载大小。
4. 设置下载超时。
5. 校验 `Content-Type` 和实际内容类型。
6. 限制重定向次数。
7. 不记录完整文件内容或 base64。

这些限制应作为 `image_url_to_base64`、`pdf_url_to_file` 等动作的统一基础能力。

## 17. 实施步骤

建议分阶段实施。

### 17.1 第一阶段：请求侧策略框架

1. 新增 `channel_model_endpoint_policies` 表和 model。
2. 在渠道缓存中加载策略。
3. 在 relay meta 中保存命中的策略。
4. 在 `getRequestBody` 附近增加策略执行入口。
5. 实现 `image_url_to_base64`。
6. 增加结构化日志。
7. 为 `messages -> messages` 直通路径增加单元测试。

### 17.2 第二阶段：管理端与测试

1. 管理端展示端点策略。
2. 模型测试支持图片 URL、图片 base64 两类测试。
3. 测试产物记录策略前请求、策略后请求、上游响应。
4. 支持根据测试结果提示建议策略，但不自动启用。

### 17.3 第三阶段：能力透出与自动化

1. 评估 `/v1/models` 是否增加 `supported_inputs`。
2. 支持策略命中统计。
3. 支持策略最近失败原因。
4. 支持批量复制策略到同渠道同类模型。

## 18. 验收标准

第一阶段完成后至少满足：

1. 不配置策略时，现有请求行为保持不变。
2. 配置 `image_url_to_base64` 后，Anthropic `/v1/messages` 图片 URL 请求能被改写为 base64 请求。
3. 改写后的请求不泄露完整 base64 到日志。
4. 图片下载失败时返回明确错误。
5. 端点协商仍严格遵守 `supported_endpoints`。
6. 单元测试覆盖直通请求和策略改写请求两类路径。

## 19. 设计原则总结

1. `supported_endpoints` 是端点可用性的真值，不表达端点内部所有输入能力。
2. 兼容策略是 Router 对具体渠道现实差异的显式声明。
3. 策略必须有限、可解释、可测试、可审计。
4. 能明确修正的输入由 Router 修正；不能明确修正的输入由 Router 明确拒绝。
5. 客户端应尽量面向 Router 标准能力编程，渠道差异由 Router 承担。
