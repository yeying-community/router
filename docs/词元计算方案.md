# 词元计算方案

本文档是 Router 当前“词元（token）计算”设计的统一文档，合并了原先拆开的“方案”和“实施设计”两部分内容。

文档目标：

- 明确请求前预扣依赖的本地估算策略
- 明确最终结算与本地估算的边界
- 明确供应商隔离、协议隔离和精度分层原则
- 明确当前代码落点、实现约束和下一步演进方向

边界：

- `YYC` 记账、法币折算、预占/结算账本：见 [扣费逻辑](./扣费逻辑.md)
- 模型价格、价格单位、复杂价格组件：见 [模型计费](./模型计费.md)

## 1. 核心结论

Router 的词元计算主策略是：

1. **本地估算优先**
   - 请求前预扣 / 预占不依赖远程 `count_tokens`
   - 主路径使用本地估算器完成输入 token 估算

2. **最终结算仍以上游 usage 为准**
   - 请求完成后优先采用上游真实 usage
   - 本地估算主要服务于“请求进入前的额度控制”和“请求开始时的预扣”

3. **远程 `count_tokens` 不再作为主路径依赖**
   - 某些上游即使支持官方 `count_tokens`，也不作为预扣主链路的必要条件
   - 是否支持远程 count endpoint，只影响未来可选能力，不影响核心可用性

4. **供应商独立估算，统一结果输出**
   - 不把不同供应商请求先改写成统一请求 DTO 再估算
   - 各供应商估算器读取各自原生请求
   - 上层只统一消费 `EstimateResult`

5. **估算精度必须显式分层**
   - OpenAI 文本类模型可做到相对精确
   - Anthropic / Gemini / 其他供应商文本类模型先采用启发式估算
   - 多模态输入大量场景只能给出粗估，不应伪装成“上游权威 token 数”

## 2. 为什么切换到本地估算优先

旧思路里，Anthropic `/v1/messages/count_tokens` 被视为重要主路径。但在真实渠道环境里，这条路线有明显问题：

1. **渠道能力不稳定**
   - 官方 Anthropic 支持 `/v1/messages/count_tokens`
   - 第三方 Anthropic 兼容供应商不一定支持
   - 即使同为 `/v1/messages`，不同渠道能力也不同

2. **会把 Router 的核心预扣能力绑定到远程接口可用性**
   - 上游没开放 count endpoint
   - 上游接口限流
   - 上游偶发错误
   - count endpoint 与真正推理 endpoint 可用性不一致

3. **预估和实际请求可能不走同一渠道**
   - 某个支持 count endpoint 的渠道可以帮忙估算
   - 但实际推理请求可能路由到另一个不支持该接口的渠道
   - 这种结果只能叫“参考估算”，不能叫“实际目标渠道的官方 token 数”

4. **本地估算覆盖面更广**
   - 不仅覆盖 Anthropic
   - 也能统一覆盖 OpenAI、Gemini、OpenAI-compatible 供应商
   - 更适合 Router 这种聚合网关场景

因此，新方案明确：

- **预扣主路径必须脱离远程 count endpoint**
- **远程 count endpoint 只能作为未来可选能力，不作为基础依赖**

## 3. 与最终结算的边界

词元计算必须区分两个阶段。

### 3.1 请求前预扣 / 预占

目标：

- 判断请求是否允许进入上游
- 估算本次请求需要预留多少额度
- 在并发条件下避免“明显透支后再失败”

特点：

- 可以接受有界误差
- 不要求与最终 usage 绝对一致
- 必须稳定、快速、独立于远程 count endpoint

### 3.2 请求后真实结算

目标：

- 按最终上游 usage 做真实扣费
- 对“预估值 - 实际值”做差额回补或补扣

特点：

- 优先信任上游返回 usage
- 本地估算只作为缺失 usage 时的兜底，不作为默认替代

因此，词元计算方案的目标不是“用本地估算完全替代真实 usage”，而是：

- 保证预扣主路径稳定可用
- 保证最终结算口径清晰
- 让估算来源、精度和回退行为可观察

## 4. 设计原则

### 4.1 供应商隔离

词元估算按供应商原生请求分别实现：

- OpenAI / Responses：读取结构化 request
- Anthropic `messages`：读取原始 `raw body`
- Gemini：后续补原生实现

禁止把跨供应商 bridge 当成词元估算的基础输入层。

bridge 可以用于：

- 请求转发兼容
- 协议转换
- 上游格式适配

但不应用于：

- 词元估算输入
- 词元校验输入
- 预扣口径输入

### 4.2 统一结果，不统一请求

允许不同供应商各自维护不同抽取路径，但统一输出：

```go
type EstimateResult struct {
    PromptTokens int
    Source       string
    Precision    Precision
    Estimator    string
}
```

统一的是结果协议，不是请求对象。

### 4.3 精度显式标记

建议约定：

- `precision=exact`
  - OpenAI tokenizer 主路径

- `precision=heuristic`
  - Anthropic / Gemini / unknown 文本启发式估算
  - OpenAI 图片规则估算

- `precision=coarse`
  - 文件 / 视频 / 通用多模态占位估算

来源示例：

- `local_openai_tokenizer`
- `local_anthropic_heuristic`
- `local_gemini_heuristic`
- `local_unknown_heuristic`

## 5. 估算策略

### 5.1 OpenAI 文本模型

策略：

- 使用真实 tokenizer
- 按模型名选择 codec
- 若模型 codec 未命中，则回退到默认 OpenAI tokenizer 基座

适用范围：

- `gpt-*`
- `o1*`
- `o3*`
- `o4*`
- `chatgpt*`

精度：

- `exact`

### 5.2 Anthropic / Gemini / 其他非 OpenAI 文本模型

策略：

- 使用启发式估算器
- 按字符类型加权：
  - 英文单词
  - 数字串
  - CJK
  - 标点
  - 数学符号
  - URL 分隔符
  - emoji
  - 空格 / 换行

适用范围：

- `claude*`
- `gemini*`
- 其他不适合使用 OpenAI tokenizer 的文本模型

精度：

- `heuristic`

### 5.3 工具与结构化输入

本地估算不只统计用户文本，还应尽量把真正进入模型上下文的结构化内容纳入估算：

- message role
- name
- tool name
- tool description
- tool schema / parameters
- tool call input
- tool result content

这些内容必须由各供应商原生抽取器分别处理，例如：

- Anthropic 直接读取 `system`、`messages[].content[]`、`tool_use`、`tool_result`
- OpenAI 直接读取 `messages`、`responses.input`、`tool_calls`
- Gemini 直接读取其原生 content parts

### 5.4 媒体输入

当前阶段：

- OpenAI 图片输入可逐步细化
- 其他图片 / 音频 / 视频 / 文件先接受保守估算
- 多模态场景先接受分层精度，不追求一步到位精确

## 6. 当前实现落点

当前文本预扣主链路在：

- [internal/relay/controller/text.go](/Users/liuxin2/Workspace/opensource/router/internal/relay/controller/text.go)

```text
RelayTextHelper

  -> reserveRelayQuota

  -> convertTextRequestForUpstream
  -> getRequestBody

  -> adaptor.DoResponse
```

当前实现特点：

1. 预扣入口已经从旧 `getPromptTokens` 切换到 `tokenestimate.Estimate(...)`
2. Anthropic `/v1/messages` 的校验、估算已经直接基于原始 body
3. Anthropic bridge 已后移到真正需要协议转换时才执行
4. 最终结算逻辑仍保持不变，继续优先使用上游 usage

## 7. 当前代码结构

当前模块：

```text
internal/tokenestimate/
  anthropic.go
  estimate.go
  heuristic.go
  openai_request.go
  provider_router.go
  structured_request.go
  structured_request_estimator.go
  types.go
```

职责划分：

- `provider_router.go`
  - 供应商 / family 分发
  - `resolveEstimateModel(...)`
  - `estimateByProvider(...)`

- `anthropic.go`
  - Anthropic 原始 `messages` body 解析
  - `system` / `tool_use` / `tool_result` / `tools` 抽取
  - Anthropic heuristic 估算

- `structured_request.go`
  - 结构化 request 抽取
  - 仅适用于 request-backed estimators

- `structured_request_estimator.go`
  - 结构化 request 的估算封装
  - 供 OpenAI / generic structured fallback 使用

- `openai_request.go`
  - OpenAI tokenizer 计数
  - 结构化 request 的 OpenAI 精确估算

- `heuristic.go`
  - provider family 识别
  - heuristic 权重与规则

- `estimate.go`
  - 公共入口

## 8. 当前控制器实现约束

### 8.1 Anthropic `messages`

控制器前半段只保留原生元信息：

- `model`
- `max_tokens`
- `stream`
- `rawBody`

在预扣之前：

- 原生校验
- 原生估算
- 预扣

在真正要进入协议转换时：

- 才 bridge 成 `GeneralOpenAIRequest`

这意味着 bridge 现在只负责：

- 转发兼容
- 协议转换

而不再负责：

- 校验输入
- 估算输入
- 预扣输入

### 8.2 OpenAI / Responses

当前仍使用 `GeneralOpenAIRequest` 结构化路径进行抽取和估算。

这条路径是当前的保守做法，后续可以继续细化，但不需要回退到“所有供应商先统一改写再估算”的模式。

## 9. 估算器选择规则

当前阶段不做复杂 registry，先做简单分发。

### 9.1 OpenAI family

匹配：

- `gpt-`
- `o1`
- `o3`
- `o4`
- `chatgpt`

结果：

- `source=local_openai_tokenizer`
- `precision=exact`
- `estimator=openai_exact`

### 9.2 Anthropic family

匹配：

- `claude`

结果：

- `source=local_anthropic_heuristic`
- `precision=heuristic`
- `estimator=anthropic_heuristic`

要求：

- 必须提供 `RawBody`
- 不允许回退到 bridge 后的 `Request`

### 9.3 Gemini family

匹配：

- `gemini`

结果：

- `source=local_gemini_heuristic`
- `precision=heuristic`
- `estimator=gemini_heuristic`

### 9.4 Unknown family

其他模型：

- `source=local_unknown_heuristic`
- `precision=heuristic`
- `estimator=unknown_heuristic`

## 10. 当前实现与目标状态的差距

当前方案已经完成：

1. 本地估算接入预扣主链路
2. Anthropic 估算脱离 bridge 输入
3. Anthropic 校验脱离 bridge 输入
4. bridge 后移到真正需要协议转换的位置
5. `internal/tokenestimate` 按 provider / request type 拆分目录职责

仍未完成或只完成一部分的内容：

1. Gemini 原生请求估算器仍未补齐
2. 多模态细粒度估算仍较弱
3. 改写后重估还没有接入
4. 持久化记录“预估值 vs 最终 usage”偏差还未落地
5. `CountToken` 类对外 endpoint 仍未设计为独立能力

## 11. 实施约束

1. **禁止把远程 `count_tokens` 当成预扣主路径**
2. **禁止静默伪装精度**
3. **禁止把本地估算直接等同于最终 usage**
4. **禁止把跨供应商 bridge 当成词元估算基础输入层**
5. **禁止让 Anthropic 再回退到 bridge 后 Request 估算**
6. **OpenAI / Anthropic / Gemini 应独立演进，不共享估算输入语义**

## 12. 测试要求

建议至少覆盖：

1. OpenAI chat 纯文本
2. OpenAI responses 结构化输入
3. OpenAI tools 对估算结果有影响
4. Anthropic raw `messages` 纯文本
5. Anthropic `tool_use` / `tool_result` / `system`
6. Anthropic 缺失 `RawBody` 时直接报错
7. Unknown model fallback

当前已经有对应测试落在：

- `internal/tokenestimate`
- `internal/relay/adaptor/anthropic`
- `internal/relay/controller`

## 13. 后续演进建议

按收益和风险排序，下一步优先级建议如下：

1. **补 Gemini 原生请求估算器骨架**
   - 先占位 provider-specific 文件和分发入口
   - 后续逐步从 structured fallback 迁到原生解析

2. **补“预估值 vs 最终 usage”偏差观测**
   - 在日志或账本快照里保留：
     - `estimated_prompt_tokens`
     - `final_prompt_tokens`
     - `estimator`
     - `source`
     - `precision`

3. **支持改写后重估**
   - 在关键请求改写点后增加重估入口
   - 当估算值显著上升时允许补预扣

4. **细化多模态估算**
   - OpenAI 图片规则继续完善
   - 其他媒体输入逐步从 coarse 走向 heuristic

5. **将远程 count endpoint 设计为独立能力**
   - 管理后台测试指定渠道 count endpoint
   - 对外能力需明确区分：
     - `official`
     - `local_estimate`
     - `unsupported`

## 14. 完成标准

词元方案在当前阶段可以认为达标，当且仅当满足以下条件：

1. 文本主链路预扣已走 `tokenestimate.Estimate(...)`
2. `/v1/chat/completions`、`/v1/responses`、`/v1/messages` 三条文本路径都能走本地估算
3. OpenAI 文本走 tokenizer
4. Anthropic 文本走原生 raw-body heuristic
5. Anthropic 校验和估算都不依赖 bridge
6. bridge 只保留在转发兼容链路
7. 估算来源与精度等级可观察
8. 最终 usage 结算逻辑保持不变
