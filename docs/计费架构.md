# 计费架构

本文档定义 Router 后续计费体系的总架构。

目标不是替代现有运行逻辑说明，而是把下面几件事彻底分开：

1. 用户买的是什么
2. 一次请求应该扣多少
3. Router 实际向上游付出了多少成本
4. 如何在国内外模型混用、上游售卖形式复杂的情况下稳定保持利润

边界：

- 当前请求期预扣、结算、套餐/余额扣减口径，见 [扣费逻辑](./扣费逻辑.md)
- 当前模型价格、价格组件和复杂条件计费口径，见 [模型计费](./模型计费.md) 与 [模型定价](./模型定价.md)
- 本文档只定义总账本架构、成本层和销售层的关系

## 1. 核心结论

Router 的计费必须拆成三层：

1. `销售层`
   用户看到和购买的是 Router 自己的产品，例如套餐、充值、兑换码。
2. `计价层`
   Router 根据模型、用量、分组策略和利润目标，计算本次应该向用户收多少 `YYC`。
3. `采购层`
   Router 实际从上游渠道买到的是余额、日额度、周额度、月额度、套餐次数、专属资源等。

这三层必须独立：

- 用户资产不等于上游余额
- 官方价格不等于真实采购成本
- 上游采购成本不直接等于用户售价

一句话：

```text
用户侧按 YYC 结算
官方价格做稳定锚点
上游采购成本做保本底线
最终销售价由 Router 自己决定
```

## 2. 总体原则

### 2.1 用户侧统一只卖 `YYC`

用户购买的资产统一为 `YYC`，不再表达为“美元额度”。

套餐和充值只是 `YYC` 的两种销售形式：

- 套餐：给用户一组具有范围和周期约束的 `YYC` 权益
- 充值：给用户一组可直接扣减的 `YYC` 余额

这样国内外模型可以共用一套用户资产体系，而不会把上游币种语义带到用户侧。

### 2.2 官方价格是锚点，不是唯一真值

官方价格的作用：

- 统一模型间比较口径
- 作为缺少采购成本明细时的参考基线
- 作为前台定价稳定性的锚点

但官方价格不能单独承担利润控制职责，因为 Router 实际采购成本可能完全不同。

### 2.3 上游采购成本必须单独建模

上游渠道售卖形式很多：

- 余额充值
- 日额度
- 周额度
- 月额度
- 套餐包
- 模型专属包
- 端点专属包

这些都必须落成统一的采购资源模型，才能算真实毛利。

### 2.4 最终售价必须同时满足两件事

1. 与官方价格体系保持大体一致，用户能理解
2. 不低于采购成本底线，平台不会亏损

因此 Router 不应只按官方价格收费，也不应把上游瞬时成本直接暴露给用户。

## 3. 三层模型

### 3.1 销售层

销售层回答的问题是：

```text
用户买到了什么资产？
```

销售层对象建议只有三类：

1. `YYC 余额`
2. `YYC 套餐权益`
3. `特定范围 YYC 权益`

例如：

- 通用充值：增加 `YYC` 余额
- 月套餐：增加每日 `YYC` + 应急 `YYC`
- 模型包：增加限定模型范围可用的 `YYC`

销售层不关心上游是美元还是人民币，也不关心上游是余额还是周包。

### 3.2 计价层

计价层回答的问题是：

```text
本次请求向用户收多少 YYC？
```

计价层输入：

- 模型官方价格
- 请求实际 usage
- 图片/音频/视频特殊计量规则
- 渠道与分组倍率
- 上游采购成本底线
- 汇率
- 风险缓冲与目标利润率

计价层输出：

- 本次应收法币金额快照
- 本次应收 `YYC`
- 本次计价来源与公式快照

### 3.3 采购层

采购层回答的问题是：

```text
Router 这次真实消耗了多少上游成本？
```

采购层不再只看“渠道余额是多少”，而是维护一组可消耗的采购资源桶。

## 4. 采购资源桶模型

建议把所有上游资源统一成 `采购资源桶`。

一个资源桶至少需要表达：

- `channel_id`
- `resource_type`
- `scope_type`
- `scope_value`
- `purchase_currency`
- `purchase_amount`
- `capacity_unit`
- `capacity_total`
- `capacity_remaining`
- `valid_from`
- `expire_at`
- `reset_cycle`
- `priority`
- `enabled`

### 4.1 `resource_type`

建议至少支持：

- `balance`
- `daily_quota`
- `weekly_quota`
- `monthly_quota`
- `package_quota`

### 4.2 `scope_type`

建议至少支持：

- `global`
- `provider`
- `model`
- `endpoint`
- `model_endpoint`

### 4.3 `capacity_unit`

建议至少支持：

- `token`
- `request`
- `image`
- `second`
- `minute`
- `cny_equivalent`
- `usd_equivalent`

### 4.4 资源桶示例

例 1：余额型渠道

```text
channel = openai-01
resource_type = balance
scope_type = global
purchase_currency = USD
purchase_amount = 100
capacity_unit = usd_equivalent
capacity_total = 100
capacity_remaining = 73.42
```

例 2：月额度渠道

```text
channel = qwen-01
resource_type = monthly_quota
scope_type = model
scope_value = qwen3.7-plus
purchase_currency = CNY
purchase_amount = 299
capacity_unit = token
capacity_total = 100000000
capacity_remaining = 64000000
expire_at = 2026-06-30 23:59:59
```

例 3：图片专属包

```text
channel = image-01
resource_type = package_quota
scope_type = model_endpoint
scope_value = gpt-image-2#/v1/images/generations
purchase_currency = CNY
purchase_amount = 500
capacity_unit = image
capacity_total = 2000
capacity_remaining = 1400
```

## 5. 采购成本计算

采购成本不再只按“模型单价”推算，而是按命中的资源桶计算。

### 5.1 余额型资源桶

如果渠道是余额模式：

```text
本次采购成本 = 本次消耗面值 × 采购汇率/折算系数
```

例如充值 100 USD，实际采购成本是 720 CNY，则：

```text
1 USD 面值的采购成本 = 7.2 CNY
```

后续每次按消耗的面值分摊成本。

### 5.2 固定套餐资源桶

如果渠道是固定套餐：

```text
基础单位成本 = 套餐采购价 / 套餐总容量
本次采购成本 = 本次消耗量 × 基础单位成本
```

### 5.3 周期额度资源桶

如果渠道是日/周/月额度，不能只按理论满用计算，需要考虑利用率：

```text
基础单位成本 = 周期采购价 / 周期总容量
有效单位成本 = 基础单位成本 / 预计利用率
本次采购成本 = 本次消耗量 × 有效单位成本
```

这里的 `预计利用率` 必须来自实际统计，而不是写死 100%。

### 5.4 资源桶选择策略

建议区分两类：

1. 会过期的资源桶
   优先按最早到期使用
2. 不会过期或长期余额桶
   优先按最低有效成本使用

这能同时兼顾：

- 减少过期浪费
- 优化整体毛利

## 6. 当前结算真值现状

在进入采购成本建模前，必须先明确：

```text
Router 当前记录的 usage，并不总是“上游返回的真实 usage”
```

当前代码里，至少存在四类结算真值：

1. `returned_usage_final`
   最终结算直接依赖上游返回的 usage。
2. `hybrid_usage_final`
   最终结算以返回 usage 为主，但会用本地已知字段补齐缺口。
3. `local_estimate_final`
   最终结算不依赖上游 usage，而是依赖 Router 本地估算。
4. `unit_based_final`
   最终结算本来就不是 token 真值问题，而是按图片数、请求数、字符数、秒数、视频数等单位计费。

这里的“真值”指的是：

```text
最终扣用户 YYC 时，Router 依据的那组用量数据到底来自哪里
```

这件事不先分清楚，后面会出现两个问题：

- 一部分模型看起来是“按 token 计费”，但其实 token 只是估算值
- 一部分模型根本不应该继续套用 token 成本模型

## 7. 第一版结算真值矩阵

第一版不按供应商名字展开，而按 Router 当前的端点协议族展开。

原因很简单：

```text
当前决定结算真值来源的，主要是具体控制器和端点族
不是 provider 名字本身
```

### 7.1 文本端点族

覆盖范围：

- `/v1/responses`
- `/v1/messages`
- `/v1/chat/completions`
- `/v1/completions`
- `/v1/embeddings`
- `/v1/moderations`

当前口径：

| 端点族 | 预扣来源 | 最终结算来源 | 结算真值分类 | 代码依据 |
| --- | --- | --- | --- | --- |
| 普通文本生成 | 本地 `tokenestimate` 估算 prompt；completion 预扣按预算值 | 上游 `usage` 优先；若缺失则部分适配器回退本地推导 | `hybrid_usage_final` | `internal/relay/controller/text.go`、`internal/relay/controller/helper.go`、`internal/relay/adaptor/openai/adaptor.go`、`internal/relay/adaptor/openai/helper.go` |
| `responses` 文本且无图片工具 | 同上 | 同上 | `hybrid_usage_final` | 同上 |
| `responses` 文本且带 image generation tool | 文本部分预扣仍按本地估算；图片工具费后补 | 文本 usage 来自上游；图片工具费用由 Router 额外补算 | `hybrid_usage_final` | `internal/relay/controller/responses_image_tool_billing.go` |
| `embeddings` | 本地按请求体估算 | 最终仍走文本 billing 主路径，依赖返回 usage 或本地补齐 | `hybrid_usage_final` | `internal/relay/controller/helper.go` |

解释：

- 预扣阶段，文本类请求统一先走本地 `tokenestimate.Estimate(...)`
- 最终结算阶段，代码语义是“尽量用上游 usage，但允许不完整 usage 被本地推导补齐”
- 因此文本主链路不能定义成纯 `returned_usage_final`

### 7.2 图片端点族

覆盖范围：

- `/v1/images/generations`
- `/v1/images/edits`

当前口径：

| 场景 | 最终结算来源 | 结算真值分类 | 代码依据 |
| --- | --- | --- | --- |
| `gpt-image-2` 生成 | Router 本地估算 prompt text + 输出金额，再换算 output quantity | `local_estimate_final` | `internal/relay/controller/image.go` |
| `gpt-image-2` 编辑 | Router 本地估算 prompt text + 输入图片 token + 输出金额 | `local_estimate_final` | `internal/relay/controller/image.go` |
| 旧版 OpenAI 图片 token 表模型 | Router 本地估算 prompt token + 图片 output token | `local_estimate_final` | `internal/relay/controller/image.go` |
| 按张计费图片模型 | 按图片张数或请求数直接计费 | `unit_based_final` | `internal/relay/billing/pricing.go`、`internal/relay/controller/image.go` |
| Qwen 图片返回实际输出张数 | 先按估算预扣，再按上游返回输出张数改写最终结算 | `returned_usage_final` | `internal/relay/controller/image.go` |

解释：

- 图片链路当前已经显式写入了 `local_estimate_final`、`provider_usage_final`、`estimate_only`
- 其中 `estimate_only` 的本质不是“真值待定”，而是“这个模型本来按张数/调用数收费，不需要 token 真值”

### 7.3 Audio 端点族

覆盖范围：

- `/v1/audio/speech`
- `/v1/audio/transcriptions`
- `/v1/audio/translations`

当前口径：

| 场景 | 最终结算来源 | 结算真值分类 | 代码依据 |
| --- | --- | --- | --- |
| TTS / speech | 按输入字符数直接计费 | `unit_based_final` | `internal/relay/controller/audio.go`、`internal/relay/billing/pricing.go` |
| transcription / translation | Router 从返回文本重新计算 token 后结算 | `local_estimate_final` | `internal/relay/controller/audio.go` |

解释：

- `speech` 路径不是 usage 问题，而是字符数单位计费
- 转写 / 翻译路径当前没有统一消费上游 usage，而是从返回文本重新 `CountTokenText(...)`

### 7.4 Video 端点族

覆盖范围：

- `/v1/videos`

当前口径：

| 场景 | 最终结算来源 | 结算真值分类 | 代码依据 |
| --- | --- | --- | --- |
| 视频生成 | 按视频数、秒数或其他 price unit 直接计费 | `unit_based_final` | `internal/relay/controller/video.go`、`internal/relay/billing/pricing.go` |
| 视频状态查询 | 不计费创建成本，只做状态读取 | 不属于计费结算路径 | `internal/relay/controller/video.go` |

解释：

- 视频当前不是 token usage 结算
- 重点是 price unit 是否为 `per_video`、`per_second` 等单位

## 8. 当前矩阵的关键结论

从现状看，Router 当前并不存在单一的“真实 token 结算口径”。

结论可以直接写成：

1. 文本主链路以 usage 为主，但不是纯 usage 真值。
2. 图片链路大量依赖本地估算或单位计费。
3. 音频链路一部分按字符数计费，一部分按本地文本 token 估算计费。
4. 视频链路本质是单位计费，不是 token 计费。

因此，后续计费架构必须把“结算真值来源”作为正式字段建模，至少保留：

- `settlement_truth_mode`
- `usage_source`
- `estimate_source`
- `price_unit`

其中：

- `settlement_truth_mode` 用于表达这次结算属于哪一类真值
- `usage_source` 用于表达用量来自上游返回还是本地计算
- `estimate_source` 用于表达估算口径来自哪套本地规则
- `price_unit` 用于区分 token 计费与单位计费

## 9. 这对采购成本层意味着什么

采购成本层不能假设“所有请求都能拿到真实 token usage”，必须分四类处理：

### 9.1 `returned_usage_final`

适合直接建立：

```text
采购成本 = returned usage × 上游采购单价
```

### 9.2 `hybrid_usage_final`

适合建立：

```text
采购成本 = 最终结算 usage × 对应采购单价
并明确标注这不是纯上游 usage 真值
```

这类请求可以用于日常核算，但在财务分析上应低于纯返回 usage 的可信度。

### 9.3 `local_estimate_final`

适合建立：

```text
采购成本 = 本地估算 usage × 官方价格锚点
或
采购成本 = 本地估算 usage × 渠道采购折算单价
```

这类请求不能假装成“真实上游 token 成本”，而应明确标注为估算成本。

### 9.4 `unit_based_final`

适合建立：

```text
采购成本 = 实际消耗单位 × 单位采购成本
```

例如：

- 每张图采购成本
- 每次调用采购成本
- 每千字符采购成本
- 每秒视频采购成本

这类请求后续不应被强行折叠回 token 口径。

## 9.5 当前采购批次消耗匹配规则

当前请求落账后，会尝试把本次请求归因到渠道采购批次。

匹配顺序：

1. 优先按本次计费金额的币种等价值匹配。
   例如 `billing_currency=USD`、`billing_amount=0.25` 时，优先消耗 `capacity_unit=usd_equivalent` 的采购批次。
2. 如果没有匹配到币种等价值批次，再按用量单位匹配。
   例如文本 token 请求回退到 `capacity_unit=token`，图片请求回退到 `image`，按次请求回退到 `request`。

这样做的原因：

- 上游余额、信用额度、充值额度通常是金额型资源，不是 token 资源。
- 文本模型虽然按 token 计价，但采购层更常见的可消耗资源是 `USD/CNY 等价值额度`。
- 如果只按 token 匹配，会导致已经配置了余额型采购记录的渠道仍然无法归因采购成本。

采购批次范围当前支持：

- `global`
- `model`

同一请求优先匹配 `model` 范围批次，再匹配 `global` 范围批次。

请求前计价也复用这套匹配顺序，但只做只读成本估算：

- 不扣减 `capacity_remaining`
- 不写入 `request_procurement_consumptions`
- 不改变采购批次状态
- 估算结果只用于 `cost_floor_base_amount`

当前请求前计价已接入文本、图片、音频、视频主链路。后续新链路也应复用同一规则：先算官方锚点，再用只读采购成本估算做底线校验。

## 10. 最终定价公式

建议把最终用户售价拆成两个值：

1. `official_anchor_cost`
   按官方价格算出的参考成本
2. `procurement_cost`
   按真实采购资源桶算出的成本

再计算两个价格：

### 10.1 官方锚点价格

```text
official_anchor_base_amount = 官方金额换算成 CNY
anchor_sell_base_amount = official_anchor_base_amount × official_markup
```

### 10.2 采购底线价格

```text
cost_floor_base_amount = procurement_cost_base_amount × (1 + risk_buffer) / (1 - target_margin)
```

其中：

- `risk_buffer` 用于覆盖汇率波动、过期损耗、对账误差、通道波动
- `target_margin` 是目标利润率

### 10.3 最终销售价

```text
selected_sell_amount = max(anchor_sell_base_amount, cost_floor_base_amount)
charge_amount = ceil(selected_sell_amount × charge_rate)
```

这是当前实现的计价基线。

含义：

- 不让售价脱离官方体系太远
- 不让售价跌破真实采购成本底线

## 11. 动态利润率

如果上游采购成本高度波动，不能只配一个固定利润率。

建议定义：

```text
effective_margin
= base_margin
+ scarcity_adjustment
+ expiry_adjustment
+ fx_adjustment
+ volatility_adjustment
+ margin_gap_adjustment
```

### 11.1 各项含义

- `base_margin`
  基础利润率，例如 20%
- `scarcity_adjustment`
  该模型当前可用上游很少时，提高利润保护
- `expiry_adjustment`
  快过期资源希望加速消耗时可下调，否则可上调
- `fx_adjustment`
  汇率波动时追加保护
- `volatility_adjustment`
  最近采购成本波动过大时追加保护
- `margin_gap_adjustment`
  近 7 天实际毛利率低于目标时自动上调

### 11.2 限制

`effective_margin` 必须有上下界，例如：

```text
10% <= effective_margin <= 60%
```

避免售价抖动过大。

### 11.3 更新粒度

不建议每次请求实时改价。

建议按：

- `模型 × 销售分组`
- `模型 × 渠道`

在固定时间窗内生成稳定价，例如每小时或每日更新一次。

## 12. 请求级账本关系

一次请求至少要同时记录四类值：

1. `官方价格快照`
2. `采购成本快照`
3. `最终销售价快照`
4. `用户扣减结果`

建议后续在请求日志或结算日志里补充：

- `official_anchor_amount`
- `procurement_cost_amount`
- `pricing_floor_amount`
- `effective_margin`
- `selected_procurement_bucket`
- `selected_procurement_bucket_type`

这样后续才能真正分析：

- 哪些模型赚钱
- 哪些渠道在亏损
- 哪些资源桶被浪费
- 哪些套餐定价有问题

## 13. 与当前 Router 的衔接

当前 Router 已经有几个可复用基础：

1. `YYC` 主账本语义
2. `BillingSnapshot`
3. `price_unit + currency`
4. 请求期预扣与结算
5. 套餐和余额两种用户侧扣减模式
6. 渠道账务快照与资源项能力

后续改造不应推翻这些基础，而应新增采购层与成本层。

### 13.1 当前仍可沿用的部分

- 用户侧仍然使用 `YYC`
- 请求结算仍然产出 `billing_amount` 与 `billing_charge_amount`
- 套餐/余额的用户侧扣减逻辑可以继续保留

### 13.2 需要新增的部分

- 渠道采购资源桶
- 请求级采购成本分摊
- 动态利润率与调价任务
- 真实毛利报表

## 14. 明确不做的事

为保持语义稳定，以下行为不建议采用：

1. 不把用户余额直接映射成上游美元余额
2. 不把上游套餐直接暴露成用户产品语义
3. 不用单一“官方价格”代表真实利润
4. 不在缺少官方价格时伪造“官方价”
5. 不让上游瞬时成本直接驱动用户侧每次请求抖动改价

建议按四步推进。

### 第一步：先定口径


## 18. 采购层对象设计

这一节只定义采购层最小对象，不直接决定数据库表名。

先把语义分开：

1. `渠道账务快照`
   记录“这个渠道当前有什么额度、余额、周期权益”。
2. `采购成本批次`
   记录“这部分采购资源是按什么成本买来的”。
3. `资源桶消耗记录`
   记录“某次请求实际消耗了哪个资源桶、消耗了多少”。
4. `单次请求成本快照`
   记录“这次请求最终归因出来的采购成本是多少”。

这四类对象不能混成一个表。

原因：

- 账务快照是渠道侧观测结果
- 采购成本批次是财务口径对象
- 资源桶消耗是分摊过程对象
- 请求成本快照是请求结算结果对象

### 18.1 渠道账务快照

当前代码里已经有一套接近这个语义的对象：

- `channel_billing_profiles`
- `channel_billing_snapshots`
- `channel_billing_snapshot_items`

这套对象适合继续承担：

```text
渠道现在有什么账务资源
```

它不适合直接承担：

```text
这次请求的采购成本是多少
```

因此后续要明确：

- `channel_billing_snapshot_items` 是渠道账务来源数据
- 它可以生成采购资源桶
- 但它本身不是请求级成本归因结果

### 18.2 采购成本批次

采购成本批次回答的问题是：

```text
这一批可用资源，是按什么成本拿到的
```

建议一个采购成本批次至少表达：

- `batch_id`
- `channel_id`
- `resource_type`
- `scope_type`
- `scope_value`
- `capacity_unit`
- `capacity_total`
- `capacity_effective`
- `purchase_currency`
- `purchase_amount`
- `purchase_fx_rate`
- `purchase_cost_amount`
- `cost_per_unit`
- `valid_from`
- `expire_at`
- `reset_cycle`
- `source_type`
- `source_ref`
- `enabled`

字段含义：

- `capacity_total`
  原始账务容量
- `capacity_effective`
  可用于成本分摊的有效容量
- `purchase_cost_amount`
  这批资源的真实采购成本，统一折算到 CNY
- `cost_per_unit`
  单位采购成本
- `source_type`
  这批成本来自哪里，例如人工录入、渠道账务页面、采购单、对账导入
- `source_ref`
  对应外部单号、快照 ID 或备注标识

这里要强调一条规则：

```text
采购成本批次必须表达真实采购成本
不能用官方价格锚点去伪装采购成本
```

### 18.3 为什么需要“批次”

因为同一个渠道会反复采购，而且采购成本会变化。

例如：

```text
2026-06-01 充值 100 USD，采购成本 720 CNY
2026-06-10 再充值 100 USD，采购成本 745 CNY
```

这两次采购不能合并成一个静态“渠道单价”。

否则后面无法回答：

- 哪一批资源更便宜
- 哪一批资源已接近过期
- 最近毛利下滑是不是因为新采购成本上升

因此采购层必须保留批次概念。

### 18.4 资源桶消耗记录

资源桶消耗记录回答的问题是：

```text
这次请求实际从哪些采购批次里扣走了多少容量
```

建议至少表达：

- `consumption_id`
- `request_log_id`
- `channel_id`
- `batch_id`
- `resource_type`
- `scope_type`
- `scope_value`
- `capacity_unit`
- `consumed_quantity`
- `unit_cost_amount`
- `consumed_cost_amount`
- `consumed_at`

这类记录的作用不是给用户看，而是给系统做两件事：

1. 追踪资源桶剩余量
2. 把一次请求的采购成本拆明白

一个请求可以对应多条消耗记录。

例如：

```text
文本请求先消耗月包 token
不足部分再消耗余额桶
```

这时单次请求的采购成本来自多个批次，不能只留一个成本数字。

### 18.5 单次请求成本快照

单次请求成本快照回答的问题是：

```text
这次请求最终被归因出的采购成本、销售价格和毛利是多少
```

建议至少表达：

- `request_log_id`
- `channel_id`
- `provider`
- `model`
- `endpoint`
- `settlement_truth_mode`
- `price_unit`
- `official_anchor_amount`
- `official_anchor_currency`
- `official_anchor_base_amount`
- `procurement_cost_base_amount`
- `procurement_cost_confidence`
- `sell_base_amount`
- `charge_amount`
- `gross_profit_base_amount`
- `gross_margin`
- `pricing_rule_version`
- `cost_rule_version`
- `snapshot_created_at`

字段含义：

- `official_anchor_base_amount`
  官方价格锚点折算后的金额
- `procurement_cost_base_amount`
  本次请求归因出来的真实或估算采购成本
- `procurement_cost_confidence`
  成本可信度，例如 returned usage、hybrid、local estimate、unit based
- `sell_base_amount`
  用户侧最终应收法币等价金额
- `charge_amount`
  用户侧最终扣减的 YYC
- `gross_profit_base_amount`
  `sell_base_amount - procurement_cost_base_amount`

这一层建议和现有消费日志强绑定。

原因：

```text
请求成本快照本质上就是消费日志的成本侧快照
```

它可以是：

- 直接扩展现有消费日志字段
- 或者独立一张 `request_cost_snapshots` 表

但语义上必须是一请求一快照。

## 19. 三类对象之间的关系

最小关系可以写成：

```text
渠道账务快照
-> 生成或更新采购成本批次
-> 请求发生时选择一个或多个批次进行消耗
-> 汇总成本生成单次请求成本快照
```

也就是说：

1. 渠道账务快照解决“资源现状”
2. 采购成本批次解决“资源成本”
3. 资源桶消耗记录解决“资源如何被扣”
4. 请求成本快照解决“请求最终成本和毛利”

### 19.1 不要把快照直接当成本批次

这是后续最容易出错的地方。

例如 `channel_billing_snapshot_items` 里可能有：

- 日额度
- 周额度
- 总额度
- 到期时间

这些是账务事实，但不等于完整成本事实。

因为还缺：

- 采购价
- 采购币种
- 折算汇率
- 有效容量
- 成本分摊口径

所以：

```text
渠道账务快照是采购成本批次的输入
不是采购成本批次本身
```

## 20. 哪些字段应先落消费日志

在正式加完整采购层表之前，最值得先落地到消费日志的字段是：

- `settlement_truth_mode`
- `official_anchor_amount`
- `official_anchor_currency`
- `official_anchor_base_amount`
- `procurement_cost_base_amount`
- `procurement_cost_confidence`
- `sell_base_amount`
- `gross_profit_base_amount`
- `gross_margin`

原因：

1. 这些字段可以先把报表跑起来
2. 不需要等完整采购资源桶上线后才能加
3. 后续即使成本模型升级，也可以通过 `pricing_rule_version`、`cost_rule_version` 区分口径

## 21. 哪些对象适合单独落表

按当前复杂度，后续最适合单独落表的是：

1. `采购成本批次`
2. `资源桶消耗记录`

原因：

- 这两类对象天然是一对多关系
- 需要持续变更、查询和对账
- 不适合硬塞进消费日志宽表

相反，`单次请求成本快照` 可以先走两种路线之一：

1. 先扩展消费日志
2. 等成本模型稳定后再拆独立表

这两条路线都可以，但前提是语义先固定。

## 22. 账务项到采购成本批次的映射

现有 `channel_billing_snapshot_items` 已经能表达渠道侧账务项。

关键字段包括：

- `resource_type`
- `quota_type`
- `quota_label`
- `amount`
- `limit_amount`
- `used_amount`
- `remaining_amount`
- `currency`
- `reset_at`
- `expires_at`
- `source_ref`

这些字段可以作为生成采购成本批次的输入，但不能单独完成可分摊成本批次生成。

因为成本批次还必须有：

- 采购金额
- 采购币种
- 采购汇率
- 有效容量
- 单位成本
- 成本分摊规则

人工新增采购记录必须表达一笔采购事实：

- `purchase_at`
  实际采购、充值或购买套餐的时间。
- `purchase_currency`
  实付币种。
- `purchase_amount`
  实付金额。
- `purchase_fx_rate`
  实付币种折算到 CNY 的汇率。
- `purchase_cost_amount`
  折算后的采购成本。
- `items`
  这笔采购买到的权益项。

采购记录保存后，可消耗权益项会生成采购批次。套餐有效期项不生成采购批次。

如果一笔采购记录生成多个采购批次，当前按各批次有效容量占比分摊 `purchase_cost_amount`。

自动账务刷新得到的是渠道当前权益状态，只用于展示、告警和管理员判断，不直接生成采购批次。采购批次必须来自明确的采购记录，否则周期额度会因为多次刷新被重复计入成本池。

采购记录可以修改或删除，但必须满足一个硬约束：

- 该采购记录生成的采购批次尚未被任何请求消耗。

如果已经产生消耗记录，采购记录不能再修改或删除。原因是消耗记录保存的是当时的单位成本和消耗成本快照，修改采购记录会反向改写历史请求成本。

### 22.0 权益类型语义

渠道账务项当前固定为四类：

| 类型 | 含义 | 是否直接表达可消耗容量 |
| --- | --- | --- |
| `balance` | 上游账户现金余额，例如充值后剩余金额 | 是 |
| `credit` | 上游赠送、授信或活动额度，来源不同于现金余额 | 是 |
| `quota` | 调用额度，可以是日、周、月、总额度或自定义额度 | 是 |
| `plan` | 套餐有效期，只表达套餐名称和截止时间 | 否 |

`credit` 和 `quota` 的区别：

- `credit` 表示上游给到的一类授信或赠送额度，通常按金额或等价额度消耗。
- `quota` 表示明确的调用容量池，例如每日额度、每周额度、每月额度或总额度。

`plan` 不能单独生成采购成本批次。如果一个套餐同时提供日额度、周额度或月额度，需要同时录入对应的 `quota` 权益项。

周期额度和总额度必须区分：

- 人工采购记录里的 `daily`、`weekly`、`monthly` 表示套餐周期额度规则，例如“30 天内每天 1000 额度”，不需要录入 30 条日额度。
- 自动账务快照里的 `daily`、`weekly`、`monthly` 如果带有 `reset_at`，表示当前周期额度实例，到 `reset_at` 后本周期未用完的额度失效，不累计到下个周期。
- `total`、`balance`、`credit` 表示累计可消耗额度，用一点少一点，直到用完或到 `expires_at` 过期。
- 套餐常见形态是“套餐有效期 + 周期额度”，例如 30 天套餐每天 1000 额度；这不是 30000 总额度，而是每天独立的 1000 当前周期额度。

### 22.1 映射总规则

从账务项生成采购成本批次时，必须同时满足：

1. 账务项处于可用状态
2. `remaining_amount > 0`
3. 能识别容量单位
4. 能确定采购成本
5. 能确定有效容量

如果只满足前 3 项，可以生成 `cost_unconfigured` 批次，供管理员后续补充采购成本。

如果 4、5 也满足，才能生成可参与成本分摊的 `active` 批次。

一句话：

```text
有额度不等于有可分摊成本
有采购成本和分摊口径，才是可参与请求成本分摊的批次
```

### 22.2 余额项映射

适用条件：

- `resource_type = balance`
- 或 `resource_type = credit` 且 `quota_type = total`
- 账务项表达的是可直接消耗的余额或总额度

映射建议：

| 账务项字段 | 采购成本批次字段 |
| --- | --- |
| `channel_id` | `channel_id` |
| `resource_type` | `balance` 或 `credit` |
| `quota_type` | `source_ref` 或辅助分类 |
| `remaining_amount` | `capacity_remaining` |
| `limit_amount` 或 `amount` | `capacity_total` |
| `currency` | `capacity_unit` 的币种语义 |
| `expires_at` | `expire_at` |
| `source_ref` | `source_ref` |

余额项的成本计算必须依赖采购记录。

例如：

```text
账务项显示剩余 73.42 USD
采购记录显示充值 100 USD 实付 720 CNY
则 cost_per_unit = 720 / 100 = 7.2 CNY / USD
```

如果只有账务余额，没有充值金额或采购成本，不能生成采购成本批次。

### 22.3 日额度映射

适用条件：

- `resource_type = quota`
- `quota_type = daily`
- 或 `reset_at` 表达日级重置

映射建议：

| 账务项字段 | 采购成本批次字段 |
| --- | --- |
| `limit_amount` | `capacity_total` |
| `remaining_amount` | `capacity_remaining` |
| `reset_at` | `expire_at` 或当前周期结束时间 |
| `quota_label` | `scope_value` 辅助说明 |
| `source_ref` | `source_ref` |

日额度的采购成本不能直接用当天额度推断。

日额度表达的是当前自然日或上游计费日的额度实例：

- 当天没用完的部分到 `reset_at` 失效。
- 当天用完后，在 `reset_at` 前不能继续使用该额度。
- 第二天额度刷新后，应形成新的当前周期额度实例，而不是在原批次上累加。

人工采购记录录入套餐日额度时，录入的是规则：

```text
quota_type = daily
limit_amount = 每日额度
expires_at = 套餐截止时间
reset_at = 0
```

系统按 `purchase_at` 到 `expires_at` 之间的周期数估算采购批次有效容量，不要求管理员手工录入每天额度。

必须知道这份日额度来自哪种采购形式：

- 月包里的每日额度
- 周包里的每日额度
- 单独购买的日包
- 渠道赠送额度

推荐映射规则：

```text
采购批次成本 = 对应套餐采购成本 × 当前日额度占套餐有效总容量的比例
```

如果无法确定套餐采购成本或有效总容量，日额度只能作为账务快照，不进入采购成本批次。

### 22.4 周额度映射

适用条件：

- `resource_type = quota`
- `quota_type = weekly`
- 或 `reset_at` 表达周级重置

周额度和日额度类似，但有效容量按周计算。

推荐映射规则：

```text
采购批次成本 = 对应套餐采购成本 × 当前周额度占套餐有效总容量的比例
```

如果周额度和日额度来自同一个套餐，不能重复把完整套餐成本分别分配给日额度和周额度。

必须先确定资源关系：

1. 周额度和日额度互相独立
2. 周额度包含日额度
3. 日额度只是周额度的展示维度

未确认关系时，不生成采购成本批次。

### 22.5 月额度映射

适用条件：

- `resource_type = quota`
- `quota_type = monthly`
- 或账务项明确表达月级周期

推荐映射规则：

```text
采购批次成本 = 月包采购成本
capacity_effective = 预计可用容量
cost_per_unit = 采购批次成本 / capacity_effective
```

这里的 `capacity_effective` 不一定等于官方标称总额度。

原因：

- 有些渠道按日重置，月内未用完会清零
- 有些渠道并发限制导致理论额度无法吃满
- 有些模型只覆盖部分额度池

因此月额度必须保留：

- `capacity_total`
- `capacity_effective`

两个字段。

### 22.6 总额度映射

适用条件：

- `quota_type = total`
- 或 `resource_type = credit`
- 或账务项表达不随日/周/月重置的累计额度

总额度最接近余额型资源。

总额度和充值余额不同于周期额度：

- 不随日、周、月自动重置。
- 每次请求消耗后扣减剩余量。
- 有效期内未用完可以继续保留；没有有效期则直到用完为止。

推荐映射规则：

```text
capacity_total = limit_amount 或 amount
capacity_remaining = remaining_amount
cost_per_unit = purchase_cost_amount / capacity_effective
```

如果总额度会过期，按 `expires_at` 做资源桶到期控制。

如果总额度不会过期，资源桶选择时可以按最低有效成本优先。

### 22.7 套餐有效期项映射

当前账务项里可能出现：

- `resource_type = plan`
- `limit_amount = 1`
- `remaining_amount = 1`
- `expires_at = 套餐过期时间`

这种项只表达套餐有效期，不表达可消耗容量。

因此：

```text
套餐有效期项不能单独生成采购成本批次
```

它只能作为其他额度项的有效期补充信息。

例如：

- 日额度没有 `expires_at`
- 但同一快照有套餐有效期项
- 可以把套餐有效期作为日额度批次的最大 `expire_at`

### 22.8 缺少采购价时的处理

缺少采购价时，不允许生成可参与请求成本分摊的批次。

可以有三种处理状态：

1. `billing_snapshot_only`
   只保留渠道账务快照，不进入成本批次。
2. `cost_unconfigured`
   已生成采购批次，但管理员需要补采购成本；该批次不参与请求成本分摊。
3. `cost_estimated`
   明确使用估算采购成本，只能用于临时报表，不能标记为真实成本。

推荐规则：

```text
生产毛利报表只能使用真实采购成本批次
估算采购成本必须单独标记，不能混入真实毛利
```

第一阶段实现规则：

- 保存 `channel_billing_snapshot_items` 后，为可识别的余额、总额度、周期额度生成 `cost_unconfigured` 批次
- `plan` 只表达套餐有效期，不生成采购批次
- 已过期、已耗尽、无剩余额度的账务项不生成采购批次
- 生成的 `cost_unconfigured` 批次 `cost_source = none`
- 请求成本分摊只读取 `cost_status = active` 且 `cost_source` 为 `actual`、`estimated` 或 `zero_cost` 的批次

### 22.9 估算采购成本的边界

估算采购成本只适合临时分析，不能作为正式利润核算。

允许估算的场景：

- 刚接入渠道，采购价还未录入
- 上游只提供模糊套餐描述
- 历史数据补算

估算时必须记录：

- `cost_source = estimated`
- `estimate_method`
- `estimate_reason`
- `estimated_by`
- `estimated_at`

估算成本不能用于：

- 自动调价
- 用户售价下限
- 正式财务毛利

### 22.10 映射后的状态

账务项映射后，采购成本批次建议至少有这些状态：

- `active`
- `exhausted`
- `expired`
- `cost_unconfigured`
- `disabled`

含义：

- `active`
  可参与请求成本分摊。
- `exhausted`
  容量已用完。
- `expired`
  已过期。
- `cost_unconfigured`
  有账务资源，但缺采购成本，不参与真实成本分摊。
- `disabled`
  管理员停用。

### 22.11 映射关系总结

| 账务项类型 | 可否直接生成采购成本批次 | 关键前提 |
| --- | --- | --- |
| 余额 | 可以 | 必须有采购金额和采购汇率 |
| 总额度 | 可以 | 必须有采购成本和有效容量 |
| 日额度 | 视情况 | 必须知道所属套餐和成本分摊口径 |
| 周额度 | 视情况 | 必须确认和日额度是否独立 |
| 月额度 | 可以 | 必须有月包采购成本和有效容量 |
| 套餐有效期 | 不可以 | 只能补充其他额度项的有效期 |
| 赠送额度 | 视情况 | 必须明确成本为 0 还是促销成本 |

## 23. 数据模型落地取舍

基于当前代码，建议采用：

```text
采购成本批次：独立表
资源桶消耗记录：独立表
单次请求成本快照：第一阶段扩展 event_logs
```

原因：

- `event_logs` 已经有大量 `billing_*` 请求结算快照字段
- 请求成本快照和消费日志天然是一请求一记录
- 采购成本批次和资源消耗是一对多关系，不适合塞进 `event_logs`

### 23.1 采购成本批次表

建议语义名称：

```text
channel_procurement_batches
```

它表达的是：

```text
某个渠道的一批采购资源，以及这批资源的真实成本
```

建议字段：

| 字段 | 含义 |
| --- | --- |
| `id` | 批次 ID |
| `channel_id` | 渠道 ID |
| `resource_type` | 余额、额度、套餐等资源类型 |
| `quota_type` | daily、weekly、monthly、total、custom |
| `scope_type` | global、model |
| `scope_value` | 范围值 |
| `capacity_unit` | token、request、image、second、minute、usd_equivalent、cny_equivalent |
| `capacity_total` | 原始容量 |
| `capacity_effective` | 用于成本分摊的有效容量 |
| `capacity_remaining` | 当前剩余容量 |
| `purchase_currency` | 采购币种 |
| `purchase_amount` | 采购金额 |
| `purchase_fx_rate` | 采购币种到 CNY 的汇率 |
| `purchase_cost_amount` | 折算后的采购成本 |
| `cost_per_unit_amount` | 单位成本 |
| `cost_source` | actual、estimated、zero_cost |
| `cost_status` | active、cost_unconfigured、exhausted、expired、disabled |
| `valid_from` | 生效时间 |
| `expire_at` | 过期时间 |
| `reset_cycle` | none、daily、weekly、monthly |
| `source_snapshot_id` | 来源账务快照 ID |
| `source_snapshot_item_id` | 来源账务项 ID |
| `source_ref` | 外部来源标识 |
| `metadata` | 扩展信息 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

当前请求成本归因只支持两类采购范围：

- `global`
  适用于当前渠道的所有模型。
- `model`
  适用于当前渠道的指定模型，`scope_value` 必须是模型名称。

当同一请求同时命中模型范围和全局范围时，优先消耗模型范围批次，再消耗全局批次。

关键约束：

- `cost_source = actual` 才能进入正式毛利报表
- `cost_status = active` 才能参与请求成本分摊
- `capacity_effective <= capacity_total`
- `cost_per_unit_amount = purchase_cost_amount / capacity_effective`
- `capacity_effective = 0` 时不能计算单位成本

### 23.2 资源桶消耗记录表

建议语义名称：

```text
request_procurement_consumptions
```

它表达的是：

```text
某次请求从哪个采购批次里消耗了多少容量和成本
```

建议字段：

| 字段 | 含义 |
| --- | --- |
| `id` | 消耗记录 ID |
| `request_log_id` | 对应 `event_logs.id` |
| `channel_id` | 渠道 ID |
| `procurement_batch_id` | 采购批次 ID |
| `resource_type` | 资源类型快照 |
| `quota_type` | 额度类型快照 |
| `scope_type` | 范围类型快照 |
| `scope_value` | 范围值快照 |
| `capacity_unit` | 容量单位 |
| `consumed_quantity` | 消耗数量 |
| `unit_cost_amount` | 消耗时单位成本快照 |
| `consumed_cost_amount` | 本条消耗成本 |
| `settlement_truth_mode` | 请求结算真值分类 |
| `cost_source` | actual、estimated、zero_cost |
| `created_at` | 创建时间 |

关键约束：

- 一个 `request_log_id` 可以对应多条消耗记录
- 消耗记录保存的是当时成本快照，后续批次成本变化不能反向改历史请求
- 正式毛利只统计 `cost_source = actual` 的消耗，或单独标识估算毛利

### 23.3 请求成本快照先扩展 `event_logs`

现有 `event_logs` 已经包含：

- `billing_price_unit`
- `billing_currency`
- `billing_pricing_source`
- `billing_usage_source`
- `billing_estimate_source`
- `billing_settlement_mode`
- `billing_input_quantity`
- `billing_output_quantity`
- `billing_amount`
- `billing_charge_amount`

因此第一阶段建议继续扩展 `event_logs`，补充成本侧字段。

建议新增字段：

| 字段 | 含义 |
| --- | --- |
| `billing_settlement_truth_mode` | 四类结算真值之一 |
| `billing_official_anchor_amount` | 官方价格锚点原币种金额 |
| `billing_official_anchor_currency` | 官方价格锚点币种 |
| `billing_official_anchor_base_amount` | 官方价格锚点 CNY |
| `billing_procurement_cost_base_amount` | 本次采购成本 CNY |
| `billing_procurement_cost_source` | actual、estimated、none |
| `billing_procurement_cost_confidence` | returned_usage、hybrid_usage、local_estimate、unit_based |
| `billing_sell_base_amount` | 用户侧售价 CNY 等价 |
| `billing_gross_profit_base_amount` | 毛利 |
| `billing_gross_margin` | 毛利率 |
| `billing_pricing_rule_version` | 计价规则版本 |
| `billing_cost_rule_version` | 成本规则版本 |

不建议第一阶段单独建 `request_cost_snapshots`。

原因：

1. 当前消费日志已经是请求级结算事实来源
2. 第一阶段目标是把账看清，不是立刻重构日志查询
3. 等成本模型稳定后，再拆独立表也不迟

### 23.4 为什么不是只扩 `event_logs`

只扩 `event_logs` 不能解决采购层问题。

原因：

- 一个请求可能消耗多个采购批次
- 一个采购批次会被多个请求消耗
- 批次剩余量、过期、成本来源需要独立查询
- 渠道资源成本需要和请求日志解耦

所以：

```text
请求成本结果可以先放 event_logs
采购成本来源必须独立建模
```

### 23.5 第一阶段最小落地范围

第一阶段建议只做最小闭环：

1. 新增 `channel_procurement_batches`
2. 新增 `request_procurement_consumptions`
3. 扩展 `event_logs` 成本侧字段
4. 默认仍按官方价格锚点扣费
5. 只有配置了采购成本，且采购底线价格高于官方锚点价格时，才提高本次扣费
6. 同步记录成本归因和毛利观测

这样风险最低。

第一阶段完成后，系统应该能回答：

- 某次请求使用了哪些采购批次
- 某次请求采购成本是多少
- 某次请求售价是多少
- 某次请求毛利是多少
- 某个渠道最近是否亏损

## 24. 第一阶段报表规则

第一阶段报表直接读取 `event_logs` 的请求成本快照字段。

当前已支持按渠道或模型聚合：

- 请求数
- 已配置采购成本请求数
- 未配置采购成本请求数
- 用户侧售价 CNY
- 已配置成本请求对应售价 CNY
- 未配置成本请求对应售价 CNY
- 采购成本 CNY
- 毛利 CNY
- 毛利率

报表接口：

```text
GET /api/v1/admin/billing/procurement-report
```

查询参数：

| 参数 | 含义 |
| --- | --- |
| `start_at` | 开始时间戳 |
| `end_at` | 结束时间戳 |
| `group_by` | `channel` 或 `model` |
| `cost_scope` | `all` 或 `unconfigured` |

`cost_scope = unconfigured` 表示只读取采购成本来源为空或 `none` 的请求，用于定位需要补采购成本的渠道或模型。

### 24.1 毛利率口径

毛利率只按已配置采购成本的请求计算。

```text
毛利率 = 已配置成本请求毛利 / 已配置成本请求售价
```

不能把 `cost_source = none` 的请求放进毛利率分母。

原因：

- 未配置采购成本不代表零成本
- 把未知成本请求计入毛利率会虚高利润
- 未配置成本请求需要单独暴露，推动管理员补成本

### 24.2 成本来源口径

正式报表中成本来源必须区分：

- `actual`
- `estimated`
- `zero_cost`
- `none`

其中：

- `actual` 可作为正式毛利依据
- `estimated` 只能作为临时报表依据
- `zero_cost` 表示明确零采购成本
- `none` 表示采购成本未配置，不能参与毛利计算

## 25. 后续待办

后续继续推进顺序：

1. 前端增加采购成本报表页面
2. 报表支持按渠道、模型、供应商、分组筛选
3. 管理员可以从报表中的未配置成本请求跳转到对应采购批次
4. 采购批次支持更细的范围配置，例如模型、端点、模型端点
5. 基于采购成本底线生成调价建议，但不自动改用户售价

## 26. 第二阶段：计价决策骨架

第二阶段先建立计价层结构，但默认不改变线上扣费结果。

新增的计价决策对象只使用金额对象表达：

```text
{
  amount: 123.45,
  currency: "CNY"
}
```

字段命名不再把币种写进字段名，例如使用：

- `official_anchor`
- `official_sell`
- `cost_floor`
- `selected_sell`
- `selected_charge`

不使用：

- `final_yyc`
- `selected_sell_amount`
- `selected_sell_yyc`
- `selected_sell_cny`

原因：

1. 金额字段必须自带单位，避免字段名和实际单位脱节。
2. 计价层以后可能同时处理 `YYC`、`CNY`、`USD` 或其他单位。
3. 决策对象应该表达业务含义，不应该把币种固化到字段名。

当前默认策略：

```text
official_markup = 1
target_margin = 0
risk_buffer = 0
```

在默认策略下：

```text
selected_charge = 当前已有扣费结果
```

也就是说，计价决策骨架只补充结构和日志快照，不改变当前用户实际扣费。

下一步再把采购层提供的采购成本传入计价决策：

```text
selected_sell = max(official_sell, cost_floor)
```

其中：

```text
cost_floor = procurement_cost × (1 + risk_buffer) / (1 - target_margin)
```
