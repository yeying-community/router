# Router API（统一文档）

> Swagger/OpenAPI  
> 本文档对应的 OpenAPI 文件为 `docs/swagger/openapi.json`，由 swag 注释生成：`scripts/gen-openapi.sh`。

## 认证方式（JWT）

### 请求头

```
Authorization: Bearer <JWT>
```

### JWT 来源（推荐）

- **钱包登录（public/auth 或 public/common/auth）** 会返回 JWT
- `/api/v1/public/profile` 支持 **钱包 JWT 或 UCAN**

> 说明
>
> - 普通用户 JWT 可访问 public 用户侧接口。
> - 管理员 JWT 可访问大多数 admin 接口。
> - `admin/user/*` 这组“用户管理”接口默认对管理员开放。
> - 但如果要“处理管理员账户本身”，例如删除其他管理员、修改其他管理员角色，则当前登录用户的钱包地址还必须命中 `bootstrap.root_wallet_address` 配置。
> - `/api/v1/public/user/login` 是密码登录（Session/Cookie），不属于 JWT 主路径；如只用 JWT 可忽略。

## 响应格式说明

- 业务接口（public/admin）一般返回：

```json
{
  "success": true,
  "message": "",
  "data": {}
}
```

- OpenAI 兼容接口（`/api/v1/public` 下的模型调用）返回 OpenAI 风格响应。
- 少数历史兼容接口可能保留原始响应形状；如存在，会在各自条目单独说明。

## 额度与金额语义

- 平台内部统一账本单位为 `夜莺币（YYC）`。
- 现有接口中所有名为 `quota`、`*_quota`、`used_quota`、`remain_quota`、`daily_quota_limit` 的字段，业务语义都统一表示 `YYC`。
- `quota` 只是历史兼容字段名，不再作为对外产品术语理解。
- `DisplayInCurrencyEnabled` 只影响前端展示和默认录入方式，不改变这些字段的接口语义。
- 如果后续新增显式金额字段，会使用 `amount` 命名，而不会复用 `quota` 字段。
- 统一基线见 [夜莺币（YYC）账本与金额语义基线](./额度与金额语义基线.md)。

#### GET /api/v1/public/user/dashboard

用户侧用量统计。

**Query 参数（可选）**

- `start_timestamp`：起始时间（Unix 秒）
- `end_timestamp`：结束时间（Unix 秒）
- `granularity`：`hour | day | week | month | year`
- `models`：逗号分隔的模型列表（仅统计所选模型）
- `include_meta=1`：返回 `meta.providers`、`meta.granularity`、`meta.start`、`meta.end`

**默认行为**

- 不传参数：默认统计近 7 天，粒度为 `day`

**返回字段**

- 数组项沿用兼容字段：
  - `Day`
  - `ModelName`
  - `RequestCount`
  - `Quota`
  - `PromptTokens`
  - `CompletionTokens`
- 同时显式返回：
  - `yyc_amount`

#### GET /api/v1/public/user/spend/overview

用户花费总览（消费/充值汇总）。

**Query 参数（可选）**

- `period`：`today | last_7_days | last_30_days | this_month | last_month | this_year | last_year | last_12_months | all_time`，默认 `last_30_days`
- `models`：逗号分隔的模型列表（仅统计所选模型）

**返回字段**

- `today_cost` / `today_revenue`：今日消费/充值（兼容字段名为 quota，业务语义为 `YYC`）
- `today_requests` / `today_tokens`：今日请求数 / Token
- `yesterday_cost` / `yesterday_revenue`：昨日消费/充值（`YYC`）
- `period_cost` / `period_revenue`：周期消费/充值（`YYC`）
- `period_requests` / `period_tokens`：周期请求数 / Token
- `period_days`：周期天数（用于计算日均）
- `period_start` / `period_end`：周期开始/结束（Unix 秒）
- `yesterday_start` / `yesterday_end`：昨日开始/结束（Unix 秒）
- 同时显式返回：
  - `today_yyc_cost` / `today_yyc_revenue`
  - `yesterday_yyc_cost` / `yesterday_yyc_revenue`
  - `period_yyc_cost` / `period_yyc_revenue`

**周期说明（服务端）**

- `today`：今天自然日（00:00:00 ~ 23:59:59）
- `last_7_days`：近 7 天（含今天）
- `last_30_days`：近 30 天（含今天）
- `this_month`：本月 1 日至当前时刻
- `last_month`：上个自然月
- `this_year`：本年 1 月 1 日至当前时刻
- `last_year`：去年自然年
- `last_12_months`：近 12 个月（滚动窗口）
- `all_time`：从该用户最早日志时间到当前

**兼容说明**

- `period=last_week` 仍兼容，服务端会自动映射为 `last_7_days`。
- 未识别的 `period` 值会回退为 `last_30_days`。

**前端推荐展示规则（花费总览）**

- 推荐下拉顺序：`today`、`last_7_days`、`last_30_days`、`this_month`、`last_month`、`this_year`、`all_time`
- 推荐趋势粒度映射：`today -> hour`
- 推荐趋势粒度映射：`last_7_days | last_30_days | this_month | last_month -> day`
- 推荐趋势粒度映射：`this_year | all_time -> month`

## Admin（运营/管理）

> 需 Admin JWT  
> 其中 `admin/user/*` 额外要求：
> - 所有管理员都可访问用户管理页面与基础用户管理接口
> - 当前登录用户的钱包地址命中 `bootstrap.root_wallet_address` 时，才可删除其他管理员或修改其他管理员角色
> - `bootstrap.root_wallet_address` 支持多个地址，使用英文逗号分隔
> - 若该配置为空，则普通管理员仍可管理普通用户，但不能处理管理员账户

#### GET /api/v1/admin/dashboard

管理员总览聚合接口，后端一次性返回 KPI、趋势、渠道健康和近期任务，避免前端多接口拼装。

**Query 参数（可选）**

- `period`：`today | last_7_days | last_30_days | this_month | last_month | this_year | last_year | last_12_months | all_time`，默认 `last_7_days`

**返回字段（data）**

- `period` / `granularity` / `start_timestamp` / `end_timestamp`
- `summary`：
  - `consume_quota` / `topup_quota` / `net_quota`
  - `consume_yyc` / `topup_yyc` / `net_yyc`
  - `request_count` / `active_user_count`
  - `channel_total` / `channel_enabled` / `channel_disabled`
  - `group_total` / `provider_total`
  - `task_active_total` / `task_failed_total`
- `trend`：按 `granularity` 聚合的时间序列，包含 `bucket`、`consume_quota`、`topup_quota`、`request_count`、`active_user_count`
  - 同时显式返回：`consume_yyc`、`topup_yyc`
- `top_channels`：渠道健康 TOP 列表，包含基础信息与健康指标：
  - 基础：`id` / `name` / `status` / `capabilities` / `balance` / `used_quota`
  - 同时显式返回：`yyc_used`
  - 健康：`selected_model_count` / `tested_model_count` / `tested_endpoint_count` / `supported_count` / `unsupported_count`
  - 比率：`pass_rate` / `coverage_rate`
  - 时效：`avg_latency_ms` / `last_tested_at` / `has_test_data`
  - 结论：`health_score` / `health_level`
- `recent_tasks`：近期系统任务列表（`type`、`status`、`channel_name`、`model`、`created_at` 等）
- `generated_at`：服务端生成时间（Unix 秒）

#### /api/v1/admin/channel/create

用于新建渠道流程第一步，创建一条 `status=creating` 的渠道记录，后续步骤都基于同一个 `channel_id` 继续编辑。

请求体示例：

```json
{
  "protocol": "openai",
  "name": "openai-main",
  "key": "sk-***",
  "base_url": "https://api.openai.com",
  "config": {}
}
```

响应体示例：

```json
{
  "success": true,
  "message": "",
  "data": {
    "id": "cad8bd65524a4d0c8f345a8c86ea9685"
  }
}
```

#### /api/v1/admin/channel/:id/models

从 `channel_models` 读取渠道模型列表。支持分页参数 `p`、`page_size` 和搜索参数 `keyword`。这里返回的是后端已保存状态，不直接请求上游。

#### /api/v1/admin/channel/:id/refresh

用于创建/编辑渠道时，根据当前已保存的渠道配置刷新渠道运行时信息。

当前支持：

- `refresh_type=models`：请求该渠道的 `/models` 接口，并同步到 `channel_models`
- `refresh_type=balance`：刷新渠道余额

该接口当前只返回任务提交是否成功，具体结果通过任务接口和渠道相关接口查看。

同步策略：

- 上游当前仍存在的模型：更新为 `inactive=false`
- 上游新增模型：插入 `channel_models`，默认 `selected=false`
- 上游已移除模型：不删除历史记录，保留在 `channel_models`，并更新为 `inactive=true, selected=false`
- 不会清空 `channel_tests` 历史记录

#### /api/v1/admin/channel/:id/tests

用于新建渠道流程第三步，在模型选择之后对渠道做真实模型测试。

请求体示例：

```json
{
  "target_models": ["gpt-4o-mini", "dall-e-3", "tts-1"]
}
```

响应体中的 `data.results` 为模型测试结果列表，典型字段包括：

- `endpoint`：本次测试的接口路径
- `model`：用于测试的模型
- `status`：`supported | unsupported | skipped`
- `latency_ms`：测试耗时
- `message`：结果说明或错误信息

## 附录：扩展与自动化

Router 支持基于 API 的系统扩展和自动化集成（例如支付、渠道启停策略、外部运维平台）。

### 鉴权补充

Router 支持 Cookie 与 Token/JWT。若使用 Token/JWT，请通过 `Authorization` 请求头传递。

### 充值链接附加参数

当用户点击充值按钮时，前端会先调用：

`POST /api/v1/public/user/topup/orders`

由 Router 后端生成内部订单和跳转地址。返回的 `redirect_url` 会附带用户与交易上下文参数，例如：

`https://example.com?username=root&user_id=1&transaction_id=4b3eed8055d5443fbd44fb18c648c837&order_id=474c9b7b534047ce9205220990f53087`

外部系统可解析这些参数完成业务联动。
