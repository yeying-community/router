# 模型定价

本文档定义 Router 下一阶段的定价分层模型，用于替代当前依赖系统内置 `ModelRatio / CompletionRatio / GroupRatio` 的方案。

注意：
- 本文档描述的是目标方案，不代表当前代码已经全部实现。
- 当前运行时计费已按数据库定价执行。
- 代码内置倍率表仅保留为初始化/迁移历史数据来源，不再参与运行时定价兜底。

## 1. 设计目标

新的定价模型只保留三层职责：

1. `供应商`：负责模型默认定价，是定价真源
2. `渠道`：负责接入能力，并允许按模型做渠道级绝对价格覆盖
3. `分组`：负责业务倍率，用于控制对外销售策略

一句话：

`最终计费 = 基础价格（供应商价或渠道覆盖价） × 分组业务倍率`

## 2. 核心原则

### 2.1 不再依赖系统内置默认倍率表

- 代码中的内置倍率表只作为初始化与历史兼容数据来源
- 新架构下，不再把系统内置倍率表当成长期可靠的定价真源

### 2.2 供应商层存价格，不存“相对未知基准的倍率”

- 供应商层表达的是“这个模型官方或标准默认价格是多少”
- 因此优先存价格字段，而不是仅存倍率

### 2.3 渠道层采用绝对覆盖

- 同一模型在不同渠道可能拿货成本不同
- 因此渠道层允许按模型配置绝对价格
- 若渠道未配置价格，则回退到供应商默认价格

当前阶段不采用“乘法修正”：

- 不做 `供应商价格 × 渠道倍率`
- 后续如确有需要，可在渠道层追加“乘法修正”能力

### 2.4 分组层保留业务倍率

- 分组层不负责定义模型成本
- 分组层只负责业务侧价格策略，例如：
  - `free`
  - `pro`
  - `enterprise`
  - `internal`

## 3. 价格来源优先级

对某次请求中的某个模型，价格来源优先级应为：

1. 渠道模型价格覆盖
2. 供应商默认价格
3. 若仍缺失，则视为“无有效价格配置”，拒绝计费或按显式兜底规则处理

不再引入“系统内置倍率表”作为运行时默认来源。

## 4. 分层职责

## 4.1 供应商

相关表：
- `providers`
- `provider_models`

职责：
- 管理供应商目录
- 管理每个模型的默认价格
- 区分模型类型：文本 / 图片 / 音频

建议字段语义：
- `provider`
- `model`
- `type`
- `input_price`
- `output_price`
- `price_unit`
- `currency`

说明：
- 文本模型通常同时有输入价和输出价
- 图片、音频模型可根据 `type + price_unit` 解释价格
- `provider_models` 是默认定价真源

## 4.2 渠道

相关表：
- `channels`
- `channel_models`

职责：
- 管理 Base URL / Key / 协议类型 / 可用能力
- 管理该渠道实际支持的模型
- 允许对某个模型做渠道级价格绝对覆盖

渠道层不负责：
- 维护供应商默认价格
- 承担业务分层定价

### 渠道级价格语义

对于 `channel_models` 中的某个模型：

- 若配置了渠道价格，则该价格直接作为基础价格
- 若未配置渠道价格，则回退到对应供应商模型默认价格

因此这里是“绝对覆盖”，不是“倍率叠乘”。

## 4.3 分组

相关表：
- `groups`

职责：
- 管理业务分层
- 管理业务倍率

建议语义：
- 每个 group 持有一个 `billing_ratio`
- 默认值通常为 `1`

示例：
- `free = 1.0`
- `pro = 1.2`
- `enterprise = 1.5`
- `internal = 0`

## 5. 统一计费公式

## 5.1 文本模型

若渠道对该模型配置了价格覆盖：

- `base_input_price = channel_input_price`
- `base_output_price = channel_output_price`

否则：

- `base_input_price = provider_input_price`
- `base_output_price = provider_output_price`

最终价格：

```text
final_input_price  = base_input_price  × group_billing_ratio
final_output_price = base_output_price × group_billing_ratio
```

最终消耗：

```text
yyc_cost = prompt_tokens × final_input_price
         + completion_tokens × final_output_price
```

其中 `yyc_cost` 表示最终写入平台内部主账本的 `YYC` 消耗。

## 5.2 图片 / 音频模型

图片、音频不再强行套“模型倍率 / 补全倍率”概念，而是直接根据价格单位计费：

- 例如：`per_image`
- 例如：`per_minute`
- 例如：`per_request`

统一规则仍然是：

```text
最终价格 = 基础价格（供应商价或渠道覆盖价） × group_billing_ratio
```

## 6. 数据模型建议

## 6.1 `provider_models`

保留并强化为默认价格真源：

- `provider`
- `model`
- `type`
- `input_price`
- `output_price`
- `price_unit`
- `currency`
- `source`
- `updated_at`

## 6.2 `channel_models`

当前 `channel_models` 仍保存：

- `model_ratio`
- `completion_ratio`

目标方案建议改为保存价格覆盖，而不是保存倍率：

- `channel_id`
- `model`
- `upstream_model`
- `selected`
- `input_price`
- `output_price`
- `price_unit`
- `currency`
- `price_source`
- `sort_order`
- `updated_at`

其中：
- 渠道未配置价格时，`input_price / output_price` 可为空或为 0
- `price_source` 可取值：
  - `provider_default`
  - `channel_override`

## 6.3 `groups`

建议在 `groups` 表中增加业务倍率字段：

- `billing_ratio`

这样分组倍率不再依赖独立的 `GroupRatio` JSON 配置，而是直接进入分组目录数据。

## 7. 请求处理顺序

建议新的请求计费顺序为：

1. 根据用户拿到 `group`
2. 根据 `group + model` 选中渠道
3. 对请求模型做渠道模型映射，得到上游模型名
4. 读取渠道模型价格覆盖
5. 若无覆盖，则读取供应商默认价格
6. 读取 group 的 `billing_ratio`
7. 计算最终价格并记账

## 8. 管理端含义

## 9. 复杂计费待扩展

复杂计费字段语法、条件键和当前运行时支持范围，见：

- [模型计费](./模型计费.md)

当前系统已经支持：

- `provider_models` 作为基础价格真源
- `provider_model_price_components` 作为复杂计费扩展
- 文本、图片、音频、视频四类模型在运行时按各自规则计费

其中 `provider_models` 仍保留这几个核心基础字段：

- `input_price`
- `output_price`
- `price_unit`
- `currency`

这些基础字段足够表达大部分“文本输入价 + 文本输出价”的模型，但仍不足以单独覆盖各家官网最新的多模态与复杂计费规则，因此需要 `provider_model_price_components` 补充表达。

### 9.1 当前已遇到的复杂计费类型

1. 同一模型按不同模态分别计费

- OpenAI `gpt-realtime`、`gpt-audio` 一类模型，通常同时存在文本 token、音频 token 两套价格。
- Google `gemini-live` 一类模型，也存在语音/文本混合计费。
- 当前单一 `input_price / output_price` 无法同时表达“文本输入价”和“音频输入价”。

2. 同一模型按不同质量/分辨率/尺寸计费

- OpenAI `gpt-image-1`
- Google `imagen`
- Zhipu `cogview`
- StepFun `step-1x`
- MiniMax `image-01`

这类模型往往按尺寸、质量、张数分别计费，不能简单压成一个稳定的 `per_image` 价格。

3. 视频模型按秒或任务阶段计费

- OpenAI `sora`
- Google `veo`
- Zhipu `cogvideox`
- Tencent `Hunyuan-Video`
- MiniMax `video-01`

这类模型常见计费方式是：

- `per_second`
- 按清晰度或模式区分
- 提交任务和生成结果分开计费

当前已经支持基础视频计费与 `video_generation` 价格组件匹配，但仍不能完整表达所有视频分档规则。

4. 文本模型存在阶梯价或上下文区间价

- Qwen `qwen-max` / `qwen-plus` / `qwen-turbo`
- 一些国内模型会按上下文窗口、命中缓存、Batch 调用与否区分价格

当前 schema 不能表达“同一模型多段价格表”。

5. 单一模型同时具备多种能力

- Anthropic Claude
- Google Gemini
- xAI Grok
- Qwen Omni

这些模型在官网层面往往同时支持文本、图像理解、音频理解，但当前 `provider_models.type` 只有一个值，不能表达多能力集合。

### 9.2 当前落库策略

针对默认初始化数据，当前采用保守策略：

- 能被当前 schema 与价格组件结构准确表达的，直接写入官方价格
- 模型官方存在但价格结构复杂的，先把模型名和主类型落库
- 对于仍无法准确压缩进当前结构的价格，不在默认数据里伪造“单一价格”
- 剩余复杂规则统一沉淀在本文档与《模型计费》中，后续集中扩展

### 9.3 建议的下一阶段扩展方向

1. 扩展价格维度

- 支持多条 price component，而不是单一输入/输出价
- 例如：
  - `text_input`
  - `text_output`
  - `audio_input`
  - `audio_output`
  - `image_generation`
  - `video_output_second`

2. 扩展价格单位

- 在现有 `per_1k_tokens / per_1k_chars / per_image / per_video` 基础上，补充：
  - `per_second`
  - `per_request`
  - `per_task`
  - `per_resolution_tier`

3. 扩展能力表达

- 让 `provider_models` 从单一 `type` 升级为能力集合，例如：
  - `text`
  - `image_input`
  - `image_output`
  - `audio_input`
  - `audio_output`
  - `video_output`

4. 保留官方来源链接

- `providers.official_url` 记录供应商官方文档入口
- 后续如有需要，可继续为模型或价格组件增加官方来源字段，方便追溯

## 8.1 供应商页

应该负责：
- 维护供应商
- 维护模型默认价格
- 维护模型类型与价格单位

## 8.2 渠道页

应该负责：
- 连接信息
- 支持模型
- 上游模型映射
- 渠道价格覆盖

不应该把渠道页设计成“全局默认定价配置页”。

## 8.3 分组页

应该负责：
- 分组基本信息
- 分组绑定渠道
- 分组业务倍率

## 9. 实施建议

建议按以下顺序落地：

1. 继续补齐无法准确表达的多模态复杂价格
2. 扩展更多复杂条件键与单位
3. 让能力集合表达替代单一 `type`
4. 继续补齐更多官方价格来源与模型目录

## 10. 最终结论

统一后的职责应固定为：

1. `供应商` 负责默认定价
2. `渠道` 负责接入能力，并可做渠道级绝对价格覆盖
3. `分组` 负责业务倍率

这是后续数据库、管理界面与计费逻辑重构的统一基线。
