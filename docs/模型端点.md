# 模型端点

本文档沉淀 Router 当前 `main` 分支里，模型端点相关的真实实现与约束。

目标是把四件事情说清楚：

1. 端点能力在系统里由哪些层决定。
2. 渠道协议、模型类型如何影响默认端点。
3. 模型测试结果如何写回、如何影响运行时与 `/v1/models`。
4. 下游端点与上游端点如何协商，哪些是桥接，哪些是直连。

## 1. 当前核心结论

1. 端点能力的运行时真值是 `channel_model_endpoints`（按 `channel+model+endpoint`）。
2. 模型测试是“端点级”测试，测试结果写入 `channel_tests`，并同步更新 `channel_model_endpoints.enabled`。
3. `channel_models.endpoint` 仍存在，但它是模型配置里的主端点字段，不再作为测试结果落库目标。
4. `/v1/models` 的 `supported_endpoints` 只暴露“已启用且最近测试通过”的端点，不再从 `channel_models` 推断兜底。
5. 文本端点按请求端点严格协商：`chat/completions` 不再桥接到 `messages` 或 `responses`。

## 2. 端点分层关系

| 层级 | 主要载体 | 当前职责 | 是否端点真值来源 |
| --- | --- | --- | --- |
| 供应商层 | `provider_models` | 提供模型与供应商归属（`owned_by`）映射 | 否 |
| 渠道层 | `channel_models` | 模型启用、上游模型名、模型类型、主端点与端点列表配置 | 部分（配置基线） |
| 渠道层 | `channel_model_endpoints` | `channel+model+endpoint` 的显式启用状态 | 是 |
| 渠道层 | `channel_tests` | 每轮模型测试结果（含端点、流式与否、支持状态） | 是（经同步） |
| 分组层 | `group_model_channels` | 决定某分组可用哪些渠道/模型 | 间接 |
| 用户接口层 | `/v1/models` | 对客户端暴露 `supported_endpoints` | 读真值，不生产真值 |

说明：

- 供应商层当前不直接管端点可用性，只管模型归属元信息。
- 分组层当前不单独维护“分组级端点白名单”，只通过可用渠道集合间接影响端点。

## 3. 默认端点与归一化

### 3.1 按模型类型默认端点

| 模型类型 | 默认端点 |
| --- | --- |
| text | `/v1/responses` |
| image | `/v1/images/generations` |
| audio | `/v1/audio/speech` |
| video | `/v1/videos` |

对应实现：`DefaultChannelModelEndpoint(modelType)`。

### 3.2 协议对 text 默认端点的覆盖

| 协议 | text 默认端点 |
| --- | --- |
| `openai`（及大多数非 anthropic 协议） | `/v1/responses` |
| `anthropic` | `/v1/messages` |

对应实现：`DefaultChannelModelEndpointWithProtocol(modelType, channelProtocol)`。

### 3.3 端点归一化规则

- text 端点只接受并归一为：`/v1/chat/completions`、`/v1/messages`、`/v1/responses`。
- image/audio/video 会归一到各自端点集合。
- 不能识别的端点在测试入口会报错，不再默认“吃掉”。

## 4. 模型测试规则（管理端）

### 4.1 测试入口约束

1. 模型测试必须带明确端点；空端点直接报错。
2. 端点必须与模型类型匹配；类型不匹配直接报错。
3. text 模型测试支持动态勾选 `is_stream`（本次测试维度）。

对应实现：`resolveChannelModelTestEndpoint`、`isChannelModelTestEndpointAllowed`。

### 4.2 测试请求模板（text）

| 测试端点 | 请求体关键字段 |
| --- | --- |
| `/v1/chat/completions` | `messages` |
| `/v1/messages` | `messages` |
| `/v1/responses` | `input`（`input_text` 结构） |

### 4.3 测试结果落库与能力更新

1. 测试结果先写入 `channel_tests`（保留端点、流式、状态、延迟、报错等）。
2. 再将最新测试结果同步到 `channel_model_endpoints.enabled`：
   - `supported=true` 且 `status=supported` -> `enabled=true`
   - 否则 -> `enabled=false`
3. 该流程不回写 `channel_models.endpoint`。

对应实现：`persistChannelModelTests` -> `AppendChannelTestsForModelsWithDB` -> `ApplyChannelModelEndpointSupportFromTestsWithDB`。

## 5. 文本请求端点协商（下游 -> 上游）

### 5.1 已命中选中模型配置时

请求进入 `resolveChannelTextUpstream` 后，按“请求端点 + 模型端点能力”做严格匹配：

| 下游请求端点 | 上游选择顺序 |
| --- | --- |
| `/v1/chat/completions` | 仅 chat |
| `/v1/messages` | 仅 messages |
| `/v1/responses` | 仅 responses |

说明：

- `messages` 仅在上游适配器支持时可选（如 anthropic）。
- 若不满足请求端点，直接报错，不做端点桥接或隐式回退。

### 5.2 未命中选中模型配置时（兼容分支）

- Anthropic/AwsClaude 通道：仅允许 `messages`，请求 `chat/responses` 会报错。
- 非 Anthropic 通道：`chat` 走 chat，`responses` 走 responses，`messages` 报错。

该分支仅用于缺少模型端点配置时的兜底，建议通过完整配置避免进入该路径。

## 6. `/v1/models.supported_endpoints` 生成规则

生成链路：

1. 取当前用户分组可用模型与渠道（`group_model_channels`）。
2. 对每个 `channel+model` 读取显式端点开关（`channel_model_endpoints`）。
3. 读取该 `channel+model+endpoint` 的最近测试支持状态（`channel_tests` 聚合结果）。
4. 仅当“端点已启用 && 最近测试支持”时，写入 `supported_endpoints`。
5. 最终按固定顺序排序后返回。

结果语义：

- `supported_endpoints` 是“当前分组可达渠道集合下，真实可用端点并集”。
- 它不是协议默认值，也不是静态猜测结果。

## 7. 四层关系的边界说明

### 7.1 供应商层

- 负责模型归属（`owned_by`）来源。
- 不负责端点可用性判定。

### 7.2 渠道层

- 决定某模型在该渠道声明/启用了哪些端点。
- 通过测试结果动态修正端点可用状态。

### 7.3 分组层

- 只控制该分组能走哪些渠道模型组合。
- 不单独定义端点策略，端点能力来自渠道层真值。

### 7.4 用户接口层

- `/v1/models` 只读并暴露可用能力，不做推断写回。

## 8. 当前边界与后续建议

1. 当前不再支持 `chat/completions -> responses/messages` 桥接；客户端应按 `/v1/models.supported_endpoints` 选择请求端点。
2. 分组层目前没有独立端点策略表；若后续需要“同一模型按分组限制端点”，应新增分组端点策略层，而不是在控制器里加条件分支。
3. 供应商层端点能力目前未独立建模；若要做“供应商官方能力约束”，应新增供应商端点能力表并与渠道层做交集校验。

## 9. 管理操作建议（当前实现）

1. 先配置渠道协议、模型、主端点与端点列表。
2. 对每个启用模型按目标端点执行测试（可分别测流式/非流式）。
3. 仅在测试通过后对外开放该端点能力。
4. 联调客户端时，以 `/v1/models.supported_endpoints` 为准，不要只看协议默认端点。
